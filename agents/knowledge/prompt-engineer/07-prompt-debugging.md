# 프롬프트 디버깅

> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview

---

## 프롬프트 디버깅 개요

프롬프트 디버깅은 모델의 응답이 의도와 다를 때 원인을 분석하고 수정하는 과정이다. 코드 디버깅처럼 체계적인 접근이 필요하다.

## 의도 불일치 유형

### 1. 형식 불일치

기대한 형식과 다른 형태로 응답하는 경우.

```markdown
# 증상
기대: JSON 출력
실제: 마크다운으로 설명 + 코드블록 안에 JSON

# 원인 분석
- 출력 형식 지시가 불명확
- "JSON으로 출력해줘"만으로는 부족 — 코드블록 포함 여부, raw JSON 여부 미지정

# 수정
Before: "결과를 JSON으로 출력해줘"
After: "결과를 raw JSON으로만 출력해줘. 코드블록, 설명 텍스트 없이 JSON만."
```

### 2. 범위 불일치

요청한 것보다 많거나 적게 응답하는 경우.

```markdown
# 증상
기대: 특정 함수 하나만 수정
실제: 관련 파일 전체를 리팩토링

# 원인 분석
- 범위 제한이 명시되지 않음
- 모델이 "도움이 되려고" 추가 작업 수행

# 수정
Before: "이 코드를 수정해줘"
After: "findUser 함수만 수정해줘. 다른 함수는 건드리지 마."
```

### 3. 깊이 불일치

너무 얕거나 깊게 분석하는 경우.

```markdown
# 증상
기대: 간단한 코드 수정
실제: 장문의 설계 철학 설명 + 3가지 대안 비교

# 원인 분석
- 상세도 수준 미지정
- 역할이 "시니어 아키텍트"로 설정되어 있어서 과도한 분석

# 수정
Before: "이 버그를 고쳐줘"
After: "이 버그를 고쳐줘. 원인 1줄 + 수정 코드만. 대안이나 설명 불필요."
```

### 4. 톤/스타일 불일치

```markdown
# 증상
기대: 간결한 코드 위주 응답
실제: "물론이죠! 좋은 질문입니다!" 등 불필요한 서두

# 수정
"하지 말 것" 섹션에 명시:
- 서두 인사/감탄 금지
- 불필요한 칭찬 금지
```

## 디버깅 프로세스

### Step 1: 증상 정확히 기록

```markdown
## 디버깅 로그
- 프롬프트: (원본)
- 기대 출력: (원했던 것)
- 실제 출력: (나온 것)
- 불일치 유형: 형식 / 범위 / 깊이 / 톤 / 정확성
```

### Step 2: 원인 가설 수립

흔한 원인 목록:

1. **지시 모호성**: 여러 해석이 가능한 표현
2. **지시 충돌**: 두 지시가 모순
3. **지시 누락**: 당연하다고 생각한 것을 명시하지 않음
4. **컨텍스트 부족**: 판단에 필요한 정보 미제공
5. **지시 과부하**: 너무 많은 지시로 일부 무시
6. **위치 효과**: 중요 지시가 중간에 묻힘

### Step 3: 가설 검증

한 번에 하나의 변경만 적용해서 원인을 특정한다.

```markdown
# 실험 1: 형식 지시 강화
변경: 출력 형식에 예시 추가
결과: 형식 준수율 향상 → 원인은 형식 지시 부족

# 실험 2: 범위 제한 추가
변경: "이 함수만" 명시
결과: 범위 내에서만 작업 → 원인은 범위 미지정
```

### Step 4: 수정 적용 및 검증

수정 후 동일 입력으로 재테스트한다. 최소 3회 반복해서 일관성을 확인한다.

## 지시 충돌 해소

### 충돌 탐지

```markdown
# 충돌 예시 1: 상세도
"항상 상세하게 설명해" ↔ "간결하게 답해"

# 충돌 예시 2: 범위
"관련 파일도 함께 수정해" ↔ "요청한 파일만 수정해"

# 충돌 예시 3: 형식
"코드블록으로 감싸" ↔ "raw JSON만 출력"
```

### 충돌 해소 전략

#### 조건부 분기

```markdown
# Before (충돌)
항상 상세하게 설명해.
간결하게 답해.

# After (조건부 분기)
- 개념/설계 질문 → 상세하게 (배경 + 분석 + 결론)
- 구현/수정 요청 → 간결하게 (코드 위주, 설명 최소화)
```

#### 우선순위 명시

```markdown
# 규칙 충돌 시 우선순위
1. 보안 관련 규칙 (최우선)
2. 출력 형식 규칙
3. 코드 스타일 규칙
4. 톤/스타일 규칙 (최하위)
```

#### 스코프 한정

```markdown
# Before (범위 충돌)
관련 코드도 함께 수정해.
요청한 것만 해.

# After (스코프 한정)
- 동일 파일 내 관련 코드: 함께 수정
- 다른 파일의 관련 코드: 수정 필요성만 언급, 직접 수정하지 마
```

## 우선순위 조정

### 위치 기반 우선순위

```markdown
# 최고 우선순위 (프롬프트 시작)
역할, 핵심 제약

# 높은 우선순위 (프롬프트 중간 앞쪽)
작업별 규칙, 출력 형식

# 보통 우선순위 (프롬프트 중간)
부가 규칙, 스타일 가이드

# recency 활용 (프롬프트 끝)
핵심 규칙 리마인더
```

### 강조 표현 기반 우선순위

```markdown
IMPORTANT: 이 규칙은 반드시 지켜야 합니다.  ← 최고
**중요**: 이 규칙을 따르세요.                ← 높음
이 규칙을 따르세요.                         ← 보통
가능하면 이 규칙을 따르세요.                  ← 낮음
```

## 흔한 디버깅 시나리오

### 시나리오 1: 모델이 지시를 무시

```markdown
# 원인 후보
1. 지시가 너무 많아서 일부 무시 → 지시 수 줄이기
2. 지시가 프롬프트 중간에 묻힘 → 앞이나 끝으로 이동
3. 지시가 모호 → 구체적으로 재작성
4. 다른 지시와 충돌 → 충돌 해소
```

### 시나리오 2: 할루시네이션 발생

```markdown
# 원인 후보
1. 존재하지 않는 API/함수를 사용 → "모르면 모른다고 해" 지시 추가
2. 잘못된 정보 생성 → 참조 자료 제공, fact-check 요청
3. 컨텍스트 부족 → 필요한 정보를 프롬프트에 포함
```

### 시나리오 3: 응답이 불안정 (매번 다름)

```markdown
# 원인 후보
1. 지시가 모호 → 해석의 여지 제거
2. temperature 높음 → 0에 가깝게 설정
3. 예시 부족 → few-shot 예시 추가
4. 출력 형식 미지정 → 명시적 형식 제공
```

## 디버깅 도구

### 프롬프트 변경 이력 관리

```markdown
## 버전 로그
v1.0: 초기 프롬프트
v1.1: 출력 형식 예시 추가 → 형식 준수율 60% → 90%
v1.2: "하지 말 것" 섹션 추가 → 불필요한 서두 제거
v1.3: 범위 제한 명시 → 과도한 수정 방지
```

### A/B 테스트

```markdown
## 테스트 케이스 3개로 비교
프롬프트 A: 지시만 (zero-shot)
프롬프트 B: 지시 + 예시 2개 (few-shot)
프롬프트 C: 지시 + 예시 2개 + 반례 1개

결과:
A: 형식 준수 60%, 정확도 70%
B: 형식 준수 85%, 정확도 80%
C: 형식 준수 95%, 정확도 85% ← 채택
```

## 프롬프트 디버깅 체크리스트

- [ ] 불일치 유형을 정확히 파악했는가 (형식/범위/깊이/톤/정확성)
- [ ] 한 번에 하나의 변경만 적용했는가
- [ ] 지시 간 충돌이 없는지 확인했는가
- [ ] 중요 지시의 위치가 적절한가 (앞 또는 끝)
- [ ] 수정 후 최소 3회 재테스트했는가
- [ ] 변경 이력을 기록했는가
