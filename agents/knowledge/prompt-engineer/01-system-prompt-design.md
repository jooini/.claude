> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/system-prompts, https://platform.openai.com/docs/guides/text-generation#system-messages

## 시스템 프롬프트의 역할

시스템 프롬프트는 모델의 전체 행동을 규정하는 최상위 지시문이다. 사용자 메시지보다 먼저 처리되며, 모든 후속 응답의 기본 동작을 결정한다.

### 핵심 기능

- **역할 정의**: 모델이 어떤 존재로서 응답할지 결정
- **행동 제약**: 하지 말아야 할 것, 반드시 해야 할 것
- **출력 형식**: 응답의 구조와 포맷 지정
- **톤/스타일**: 말투, 상세도, 전문성 수준

## 역할 정의 원칙

### 구체적 역할 > 추상적 역할

모호한 역할 지정은 모델의 응답 품질을 떨어뜨린다. 구체적인 전문가 페르소나를 부여해야 한다.

```markdown
# 나쁜 예
당신은 도움이 되는 어시스턴트입니다.

# 좋은 예
당신은 10년 경력의 NestJS 백엔드 개발자입니다.
TypeORM과 MariaDB를 주로 사용하며, 클린 아키텍처 원칙을 따릅니다.
코드 리뷰 시 성능과 보안을 최우선으로 검토합니다.
```

### 역할에 포함할 요소

1. **전문 분야**: 구체적인 기술 스택, 도메인
2. **경험 수준**: 시니어, 주니어 등 (응답 깊이에 영향)
3. **판단 기준**: 의사결정 시 우선순위
4. **커뮤니케이션 스타일**: 간결 vs 상세, 격식 vs 비격식

## 제약 조건 설계

### 명시적 금지 (Negative Constraints)

모델이 하지 말아야 할 행동을 명확히 정의한다. "~하지 마"가 "~해"보다 효과적인 경우가 많다.

```markdown
## 하지 말 것
- "물론이죠!", "좋은 질문이에요!" 같은 서두 금지
- 내가 안 물어본 대안 줄줄이 나열하지 마
- 코드 일부만 주고 나머지는 `// ...동일` 처리 금지
- 모르면 모른다고 해. 그럴듯하게 지어내지 마
```

### 명시적 허용 (Positive Constraints)

반드시 해야 할 행동도 명시한다.

```markdown
## 반드시 할 것
- 구현 요청이면 군더더기 없이 바로 코드로
- 확신 없는 부분은 "확실하지 않은데" 라고 먼저 말해줘
- 수정된 파일은 전체 다 써줘
```

## 출력 형식 구조화

### 형식 지정의 3요소

1. **구조 (Structure)**: 섹션, 헤더, 리스트
2. **포맷 (Format)**: JSON, 마크다운, 코드블록
3. **길이 (Length)**: 최소/최대 제한

```markdown
## 응답 형식
- 코드 변경 시: 파일 경로를 먼저 표시하고, 전체 파일 내용을 코드블록으로
- 설명이 필요한 경우: 코드 위에 1~2문장으로 요약
- 대안이 있으면: 코드 아래에 짧게 1~2개만
```

### 조건부 형식

입력 유형에 따라 출력 형식을 분기할 수 있다.

```markdown
## 응답 규칙
- 버그 리포트 → 원인 분석 + 수정 코드
- 새 기능 요청 → 설계 요약 + 구현 코드
- 코드 리뷰 요청 → 문제점 목록 + 개선 코드
```

## 시스템 프롬프트 구조 패턴

### 레이어 패턴

큰 시스템 프롬프트는 레이어로 분리하면 관리가 쉽다.

```markdown
# Layer 1 — 사용자 프로필 (모든 도구 공통)
## Stack
- 백엔드: NestJS, TypeORM, MariaDB
## 코드 스타일
- 주석은 인라인으로

# Layer 2 — 도구별 설정 (Claude Code 전용)
## 에이전트 시스템
...
## 파이프라인 시스템
...
```

### 섹션 우선순위

모델은 시스템 프롬프트의 앞부분에 더 높은 가중치를 둔다. 중요한 지시를 먼저 배치한다.

1. 역할 정의 (가장 중요)
2. 핵심 제약 조건
3. 출력 형식
4. 부가 규칙

## 흔한 실수와 해결

### 1. 지시 충돌

```markdown
# 문제: 두 지시가 모순
"항상 상세하게 설명해" + "간결하게 답해"

# 해결: 조건으로 분기
- 개념 설명 요청 → 상세하게
- 구현 요청 → 간결하게, 바로 코드로
```

### 2. 과도한 지시

시스템 프롬프트가 너무 길면 모델이 일부 지시를 무시할 수 있다. 핵심만 남기고 나머지는 사용자 메시지에서 보충한다.

### 3. 모호한 표현

```markdown
# 나쁜 예
적절하게 답변해주세요.

# 좋은 예
- 코드는 TypeScript strict 모드 기준으로 작성
- 응답은 3문장 이내로 요약한 뒤 코드 제시
```

## 실전 시스템 프롬프트 템플릿

```markdown
# 역할
당신은 [구체적 전문가]입니다. [핵심 기술/도메인]을 전문으로 합니다.

## 기술 스택
- [스택 나열]

## 응답 규칙
- [핵심 규칙 3~5개]

## 하지 말 것
- [금지 행동 3~5개]

## 출력 형식
- [형식 규칙]
```

## 시스템 프롬프트 검증 체크리스트

- [ ] 역할이 구체적으로 정의되어 있는가
- [ ] 금지 행동이 명시되어 있는가
- [ ] 출력 형식이 명확한가
- [ ] 지시 간 충돌이 없는가
- [ ] 토큰을 낭비하는 불필요한 지시가 없는가
- [ ] 가장 중요한 지시가 프롬프트 앞부분에 있는가
