# 모델별 패턴

> 참조 링크: https://docs.anthropic.com/en/docs/about-claude/models, https://platform.openai.com/docs/models

## 모델별 프롬프팅이 필요한 이유

각 모델은 학습 데이터, 아키텍처, fine-tuning 방식이 다르다. 같은 프롬프트라도 모델에 따라 응답 품질이 크게 달라진다. 모델의 특성을 이해하고 프롬프트를 조정해야 최적의 결과를 얻을 수 있다.

## Claude 모델 특성

### Claude의 강점

```markdown
# Claude 핵심 특성
- 긴 컨텍스트 처리: 최대 200K 토큰 (Opus/Sonnet)
- 지시 따르기: 복잡한 다중 제약 조건을 잘 따름
- 코드 생성: 전체 파일 생성, 타입 안전성 높음
- 안전성: 가드레일 지시를 잘 준수
- 구조화된 출력: JSON, XML 등 형식 준수율 높음
- 도구 사용: function calling 정확도 높음
```

### Claude 프롬프팅 팁

```markdown
# Claude에 효과적인 패턴

## 1. XML 태그 활용
Claude는 XML 태그를 구조 마커로 잘 인식한다.
<context>
프로젝트 정보
</context>

<instructions>
수행할 작업
</instructions>

<constraints>
지켜야 할 제약
</constraints>

## 2. 명시적 사고 유도
"Think step by step" 대신 extended thinking을 활용하거나,
구체적 단계를 나열하는 것이 효과적이다.

## 3. 직접적 지시
Claude는 직접적이고 구체적인 지시를 선호한다.
"가능하면 해주세요" → "반드시 해주세요"
모호한 표현보다 명확한 규칙이 효과적.

## 4. 네거티브 지시 준수율 높음
"하지 마" 류의 지시를 잘 따른다.
금지 목록을 명시하면 대부분 준수.
```

### Claude 모델 티어별 차이

```markdown
# Opus (최상위)
- 복잡한 추론, 다단계 분석
- 긴 코드 생성, 아키텍처 설계
- 미묘한 뉘앙스 이해
- 비용 높음, 속도 느림
→ 용도: 복잡한 개발, 설계, 어려운 디버깅

# Sonnet (중간)
- 일상적 코드 생성, 리뷰
- 적절한 비용 대비 성능
- 지시 따르기 우수
→ 용도: 일반 개발, 코드 리뷰, 테스트, 문서

# Haiku (경량)
- 빠른 응답, 저비용
- 간단한 분류, 추출, 변환
- 복잡한 추론에는 한계
→ 용도: 분류, 데이터 추출, 간단한 Q&A
```

## GPT 모델 특성

### GPT의 강점

```markdown
# GPT 핵심 특성
- 풍부한 학습 데이터: 다양한 도메인 지식
- 코드 생성: 특히 Python, JavaScript에 강함
- 대화 자연스러움: 자연스러운 한국어 생성
- 플러그인/도구 생태계: 다양한 통합 지원
- JSON mode: 구조화된 출력 강제 가능
```

### GPT 프롬프팅 팁

```markdown
# GPT에 효과적인 패턴

## 1. 역할 부여가 효과적
"You are an expert..." 패턴이 응답 품질에 큰 영향.
Claude보다 역할 설정에 민감하게 반응한다.

## 2. 마크다운 구조 선호
XML보다 마크다운 헤더(##)와 리스트(-) 구조를 잘 인식.

## 3. JSON mode 활용
response_format: { type: "json_object" } 사용 시
프롬프트에 "JSON으로 응답해줘"보다 훨씬 안정적.

## 4. 시스템 메시지 준수율
시스템 메시지의 규칙을 잘 따르지만,
긴 대화에서 뒤쪽으로 갈수록 준수율이 하락할 수 있다.
주기적으로 규칙을 user 메시지에서 리마인드하면 효과적.
```

### GPT 모델 티어별 특성

```markdown
# GPT-4o
- 멀티모달 (텍스트 + 이미지)
- 빠른 응답 속도
- 일상적 작업에 적합
→ 용도: 일반 개발, 이미지 분석 포함 작업

# o1 / o3 시리즈
- 추론 특화 (chain-of-thought 내장)
- 수학, 과학, 코딩 문제에 강함
- 단순 작업에는 과잉 사양
→ 용도: 알고리즘 설계, 복잡한 로직, 수학적 추론

# GPT-4o-mini
- 저비용, 빠른 속도
- 간단한 작업에 충분한 성능
→ 용도: 분류, 요약, 간단한 코드 생성
```

## Gemini 모델 특성

### Gemini의 강점

```markdown
# Gemini 핵심 특성
- 초대형 컨텍스트: 최대 1M~2M 토큰
- 멀티모달 네이티브: 텍스트, 이미지, 비디오, 오디오
- Google 생태계 통합: Search, Workspace 등
- 코드 실행: 내장 코드 실행 환경
- 구조화된 데이터: 표, 그래프 처리에 강함
```

### Gemini 프롬프팅 팁

```markdown
# Gemini에 효과적인 패턴

## 1. 긴 문서 처리에 최적
100K 토큰 이상의 문서도 처리 가능.
전체 코드베이스를 컨텍스트에 넣고 분석하는 작업에 적합.

## 2. 구조화된 입력 선호
JSON, 표 형태의 입력을 잘 처리한다.
비구조화된 텍스트보다 구조화된 형식이 효과적.

## 3. 멀티모달 프롬프트
이미지와 텍스트를 함께 입력할 때,
이미지를 먼저 제공하고 질문은 뒤에 배치하면 효과적.

## 4. System Instruction 활용
Gemini의 system instruction은 모든 턴에 적용.
일관된 동작을 위해 핵심 규칙을 system instruction에 배치.
```

## 모델 간 비교

### 프롬프트 이식성

```markdown
# 모델 간 프롬프트 이식 시 조정 필요 사항

## Claude → GPT
- XML 태그 → 마크다운 헤더로 변경
- prefill 제거 (GPT는 assistant prefill 미지원, 비공식적으로 가능)
- extended thinking → 명시적 "Think step by step"
- 네거티브 지시 강화 (GPT가 금지 규칙 무시율 더 높음)

## GPT → Claude
- "You are an expert..." → 더 구체적인 역할/행동 규칙으로 확장
- JSON mode 지시 → Claude는 프롬프트에서 직접 형식 지정
- temperature 의존 → Claude는 프롬프트로 변동성 제어가 효과적

## Claude → Gemini
- 200K 이내 컨텍스트 → Gemini에서는 더 많은 컨텍스트 제공 가능
- 도구 정의 형식 차이 주의
- system instruction 형식 맞춤 필요
```

```markdown
| 에이전트 시스템 | Claude Sonnet | GPT-4o | 도구 사용 |
```

## 모델별 프롬프트 최적화 전략

### 범용 프롬프트 작성법

여러 모델에서 동작하는 프롬프트를 작성하려면.

```markdown
# 범용 호환 패턴
1. 마크다운 구조 사용 (XML, JSON 등 모델 특화 형식 회피)
2. 명시적이고 직접적인 지시 (모호한 표현 회피)
3. 예시 포함 (few-shot은 모든 모델에서 효과적)
4. 구체적 제약 조건 (모든 모델이 따르기 쉬운 명확한 규칙)
5. 출력 형식을 예시로 제시 (문장 설명보다 효과적)
```

### 모델 전환 체크리스트

```markdown
# 모델 전환 시 확인 사항
□ 컨텍스트 크기 한도가 다른가?
□ 시스템 프롬프트 형식이 다른가?
□ 도구/function calling 형식이 다른가?
□ 특수 기능(prefill, JSON mode 등) 사용 여부?
□ 가격/속도 차이를 고려했는가?
□ 기존 테스트 케이스를 새 모델에서 재실행했는가?
```

## 모델 업데이트 대응

```markdown
# 모델 버전 업데이트 시
1. 기존 프롬프트를 새 버전에서 테스트
2. 회귀 테스트 전체 실행
3. 성능 변화 기록
4. 필요 시 프롬프트 조정
5. 모델 버전을 명시적으로 고정 (프로덕션)

# 버전 고정 권장
프로덕션 환경에서는 특정 모델 버전을 고정한다.
"claude-sonnet-4-20250514"처럼 날짜 버전을 사용.
"claude-3-sonnet-latest" 같은 자동 업데이트 버전은
개발/테스트 환경에서만 사용한다.
```
