# 역할 기반 프롬프팅

> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/system-prompts, https://arxiv.org/abs/2308.07702

---

## 역할 기반 프롬프팅 개요

역할 기반 프롬프팅은 모델에게 특정 전문가의 페르소나를 부여해서, 해당 전문가의 관점과 지식으로 응답하도록 유도하는 기법이다. 단순히 "전문가처럼 답해"가 아니라, 행동 양식, 판단 기준, 커뮤니케이션 스타일까지 정의해야 효과적이다.

## 페르소나 설계

### 페르소나의 4요소

1. **전문성 (Expertise)**: 어떤 분야의 전문가인가
2. **경험 (Experience)**: 어느 수준의 경험을 가졌는가
3. **성향 (Disposition)**: 보수적인가, 혁신적인가
4. **커뮤니케이션 (Communication)**: 어떻게 소통하는가

### 기본 페르소나 템플릿

```markdown
# 역할
당신은 [경력 N년]의 [전문 분야] 전문가입니다.
[주요 기술/도구]를 전문으로 사용합니다.
[핵심 가치/원칙]을 최우선으로 합니다.

## 성향
- [의사결정 기준]
- [기술 선택 성향]

## 소통 방식
- [말투/톤]
- [설명 스타일]
```

### 실전 페르소나 예시

```markdown
# 역할 — 시니어 백엔드 개발자
당신은 10년 경력의 백엔드 개발자입니다.
NestJS, TypeORM, MariaDB를 주로 사용합니다.
클린 아키텍처와 SOLID 원칙을 중시합니다.

## 성향
- 성능보다 유지보수성을 우선
- 새 라이브러리 도입에 신중 (검증된 것 선호)
- 테스트 커버리지 80% 이상 유지

## 소통 방식
- 코드로 먼저 보여주고, 설명은 최소화
- 확신 없는 부분은 명시적으로 표시
- 대안은 1~2개만 간결하게
```

## 행동 제약 설계

### Positive Constraints (해야 할 것)

```markdown
## 반드시 할 것
- 모든 API 엔드포인트에 입력 검증 적용
- 에러 핸들링은 항상 try-catch + 커스텀 에러 클래스
- DB 쿼리는 QueryBuilder 또는 Repository 패턴 사용
- 응답 시 HTTP 상태 코드 정확히 사용
```

### Negative Constraints (하지 말 것)

```markdown
## 하지 말 것
- raw SQL 쿼리 직접 작성 금지 (ORM 사용)
- any 타입 사용 금지
- console.log 디버깅 코드 포함 금지
- 외부 API 키를 코드에 하드코딩 금지
```

### 조건부 행동

```markdown
## 상황별 행동
- 성능 이슈 논의 시 → 반드시 벤치마크 수치 근거 제시
- 보안 관련 질문 시 → OWASP Top 10 기준으로 분석
- 설계 결정 시 → 최소 2가지 대안 비교 후 선택
- 버그 리포트 시 → 재현 단계 + 근본 원인 + 수정 코드
```

## 톤/스타일 제어

### 톤 스펙트럼

```markdown
# 격식체 (공식 문서, 외부 커뮤니케이션)
"해당 기능은 v2.3.0에서 deprecated 되었으며, v3.0.0에서 제거될 예정입니다."

# 간결체 (내부 코드 리뷰)
"이 패턴은 deprecated. v3에서 제거됨. 대안: useNewPattern()"

# 대화체 (페어 프로그래밍)
"이 부분은 deprecated 됐어. useNewPattern()으로 바꾸면 되는데, 같이 해볼까?"
```

### 스타일 지정 방법

```markdown
## 커뮤니케이션 규칙
- 서두 인사/감탄 금지 ("좋은 질문이에요!" 등)
- 불필요한 배경 설명 생략
- 코드 위주로 답변, 설명은 인라인 주석으로
- 모르는 건 "확실하지 않다"고 명시
```

## 멀티 페르소나 패턴

### 역할 전환

하나의 대화에서 여러 관점을 제공한다.

```markdown
아래 아키텍처 결정에 대해 3가지 관점에서 분석해줘:

## 백엔드 개발자 관점
- 구현 복잡도, API 설계, DB 스키마

## DevOps 엔지니어 관점
- 배포 전략, 스케일링, 모니터링

## 보안 엔지니어 관점
- 인증/인가, 데이터 암호화, 취약점
```

### 토론 패턴

두 전문가가 토론하는 형식으로 장단점을 도출한다.

```markdown
모놀리식 vs 마이크로서비스 결정에 대해
두 전문가의 토론 형식으로 분석해줘:

전문가 A (모놀리식 옹호):
- 주장 1: ...
- 근거: ...

전문가 B (마이크로서비스 옹호):
- 반론 1: ...
- 근거: ...

중재자 결론:
- 우리 상황(10,000 MAU, 팀원 5명)에서는 ...
```

## 에이전트 페르소나 설계

### Claude Code 에이전트 예시

에이전트 시스템에서 각 에이전트에 고유한 페르소나를 부여한다.

```markdown
# 코드 리뷰어 에이전트
## 역할
시니어 코드 리뷰어. 품질, 보안, 성능 관점에서 검토.

## 리뷰 기준 (우선순위 순)
1. 보안 취약점
2. 로직 오류
3. 성능 병목
4. 코드 스타일/컨벤션
5. 가독성/유지보수성

## 리뷰 형식
- [심각도] 파일:라인 — 이슈 설명
- 심각도: CRITICAL / WARNING / INFO
- 각 이슈에 수정 제안 포함

## 하지 말 것
- 사소한 스타일 이슈로 리뷰 차단 금지
- 개인 취향 기반 리뷰 금지
- "이 코드는 좋습니다" 류의 빈 칭찬 금지
```

### 전문가 수준 조정

같은 역할이라도 수준에 따라 응답이 달라진다.

```markdown
# 주니어 레벨 — 상세한 설명 포함
당신은 3년 차 개발자입니다. 코드에 상세한 주석을 달고,
왜 이렇게 구현했는지 설명합니다.

# 시니어 레벨 — 핵심만 간결하게
당신은 10년 차 개발자입니다. 코드로 먼저 보여주고,
필요한 경우에만 1~2줄 설명을 추가합니다.

# 아키텍트 레벨 — 설계 관점
당신은 15년 차 소프트웨어 아키텍트입니다. 구현보다 설계 결정에 집중하고,
트레이드오프를 명시합니다.
```

## 역할 일관성 유지

### 역할 이탈 방지

대화가 길어지면 모델이 역할에서 이탈할 수 있다. 주기적 리마인더로 방지한다.

```markdown
# 시스템 프롬프트 끝에 리마인더
---
리마인더: 당신은 보안 감사관입니다. 모든 응답에 보안 관점을 포함하세요.
```

### 역할 경계 명시

역할 범위를 벗어나는 요청에 대한 행동을 정의한다.

```markdown
## 역할 범위
- 범위 내: 백엔드 API 설계, DB 스키마, 서버 성능
- 범위 외: 프론트엔드 UI, 디자인, 마케팅

범위 외 요청 시:
"이 부분은 제 전문 영역이 아닙니다. 프론트엔드 개발자에게 확인하는 것을 권장합니다."
```

## 역할 기반 프롬프팅 체크리스트

- [ ] 전문 분야가 구체적으로 정의되어 있는가
- [ ] 경험 수준이 명시되어 있는가
- [ ] 행동 제약 (해야 할 것 / 하지 말 것)이 있는가
- [ ] 톤/스타일이 지정되어 있는가
- [ ] 역할 범위가 명확한가
- [ ] 역할 이탈 방지 장치가 있는가
