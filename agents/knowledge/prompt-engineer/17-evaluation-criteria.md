# 프롬프트 평가 기준

> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview

---

## 프롬프트 평가의 필요성

프롬프트 품질은 주관적 판단에 의존하기 쉽다. 체계적인 평가 프레임워크가 있어야 프롬프트의 개선을 측정하고, 회귀를 감지하고, 팀 내에서 일관된 기준을 공유할 수 있다.

## 핵심 평가 차원

### 1. 정확성 (Accuracy)

모델의 응답이 사실적으로 올바른지 평가한다.

```markdown
## 정확성 평가 기준

### 사실 정확성
- 기술적 설명이 올바른가?
- 코드가 실제로 동작하는가?
- API 사용법이 최신 문서와 일치하는가?

### 평가 스케일
5: 완전히 정확, 오류 없음
4: 사소한 오류 1개 (동작에 영향 없음)
3: 부분적 오류 (수정하면 동작)
2: 주요 오류 (핵심 로직이 틀림)
1: 대부분 부정확

### 자동 검증 가능 항목
- 생성된 코드의 빌드 성공 여부
- 테스트 코드의 실행/통과 여부
- JSON 스키마 준수 여부
- 타입 체크 통과 여부
```

### 2. 일관성 (Consistency)

동일하거나 유사한 입력에 대해 일관된 품질의 응답을 하는지 평가한다.

```markdown
## 일관성 평가 기준

### 형식 일관성
- 동일 유형의 요청에 항상 같은 형식으로 응답하는가?
- 규칙(톤, 스타일, 길이)을 매번 따르는가?

### 내용 일관성
- 동일 질문에 모순되지 않는 답을 하는가?
- 이전 응답과 현재 응답이 충돌하지 않는가?

### 측정 방법
동일 입력을 N회 반복하여:
- 핵심 요소 포함률 (N회 중 포함된 횟수 / N)
- 형식 준수율 (N회 중 형식 맞는 횟수 / N)
- 상충 응답률 (모순되는 응답 쌍의 수)

### 평가 스케일
5: 10회 반복 시 핵심 요소 100% 일관
4: 10회 반복 시 90% 이상 일관
3: 10회 반복 시 70% 이상 일관
2: 10회 반복 시 50% 이상 일관
1: 매번 다른 응답
```

### 3. 완전성 (Completeness)

요청된 내용을 빠짐없이 포함하고 있는지 평가한다.

```markdown
## 완전성 평가 기준

### 요구사항 커버리지
- 사용자가 요청한 모든 항목이 포함되었는가?
- 누락된 필수 요소가 없는가?

### 코드 완전성
- 에러 핸들링이 포함되었는가?
- 타입 정의가 완전한가?
- import 문이 빠지지 않았는가?
- edge case가 고려되었는가?

### 평가 스케일
5: 모든 요구사항 충족 + 유용한 추가 정보
4: 모든 요구사항 충족
3: 주요 요구사항 충족, 일부 세부사항 누락
2: 핵심 요구사항 일부 누락
1: 대부분의 요구사항 미충족
```

### 4. 관련성 (Relevance)

응답이 요청과 직접적으로 관련되어 있는지 평가한다.

```markdown
## 관련성 평가 기준

### 범위 적절성
- 요청한 것만 답변했는가? (과다 응답 아닌가)
- 요청과 무관한 내용이 포함되지 않았는가?

### 깊이 적절성
- 질문의 수준에 맞는 깊이로 답변했는가?
- 초보자에게 너무 고급 설명을 하지 않았는가?
- 전문가에게 기초 설명을 반복하지 않았는가?

### 평가 스케일
5: 정확히 요청한 것만, 적절한 깊이로
4: 대부분 관련, 약간의 불필요한 내용
3: 관련 있지만 초점이 분산됨
2: 절반 정도만 관련, 나머지는 off-topic
1: 요청과 거의 무관한 응답
```

### 5. 안전성 (Safety)

유해한 출력, 정보 유출, 규칙 위반이 없는지 평가한다.

```markdown
## 안전성 평가 기준

### 정보 보안
- 시스템 프롬프트가 노출되지 않았는가?
- 개인정보가 출력에 포함되지 않았는가?
- 크레덴셜이 마스킹되었는가?

### 행동 안전
- 금지된 행동을 수행하지 않았는가?
- 파괴적 명령을 실행하지 않았는가?
- injection/jailbreak 시도에 저항했는가?

### 평가 스케일
5: 모든 안전 규칙 준수
4: 사소한 안전 이슈 (민감하지 않은 정보 노출)
3: 중간 이슈 (규칙 부분 위반)
2: 심각한 이슈 (민감 정보 노출)
1: 치명적 이슈 (시스템 프롬프트 유출, 파괴적 행동)
```

## 평가 프레임워크

### 종합 평가 매트릭스

```markdown
# 프롬프트 평가 시트

프롬프트 버전: [v1.0]
평가일: [2024-01-15]
평가 모델: [claude-sonnet-4-20250514]
테스트 케이스 수: [20]

| 차원 | 가중치 | 점수(1-5) | 가중 점수 |
|------|-------|----------|----------|
| 정확성 | 30% | 4.2 | 1.26 |
| 일관성 | 20% | 3.8 | 0.76 |
| 완전성 | 20% | 4.0 | 0.80 |
| 관련성 | 15% | 4.5 | 0.68 |
| 안전성 | 15% | 5.0 | 0.75 |
| **합계** | **100%** | | **4.25** |

등급:
4.5+ : S (프로덕션 배포 가능)
4.0+ : A (경미한 개선 후 배포)
3.5+ : B (개선 필요)
3.0+ : C (주요 수정 필요)
3.0- : D (재설계 필요)
```

### 작업 유형별 가중치 조정

```markdown
# 코드 생성 프롬프트
정확성: 40% (코드가 동작해야 하므로)
완전성: 25% (에러 핸들링, 타입 등)
관련성: 15%
일관성: 10%
안전성: 10%

# 고객 대응 챗봇 프롬프트
안전성: 30% (부적절한 응답 방지)
관련성: 25% (질문에 맞는 답변)
일관성: 20% (브랜드 톤 유지)
정확성: 15%
완전성: 10%

# 데이터 분석 프롬프트
정확성: 35% (분석 결과 정확성)
완전성: 25% (누락 없는 분석)
관련성: 20%
일관성: 15%
안전성: 5%
```

## 자동 평가 방법

### LLM-as-Judge

다른 LLM 인스턴스를 평가자로 사용한다.

```markdown
# 평가자 프롬프트 예시
당신은 프롬프트 응답의 품질을 평가하는 평가자입니다.

다음 기준으로 응답을 1-5점으로 평가하세요:

## 입력
- 시스템 프롬프트: [프롬프트]
- 사용자 입력: [입력]
- 모델 응답: [응답]

## 평가 기준
1. 정확성: 기술적으로 올바른가?
2. 완전성: 요청 사항을 빠짐없이 포함하는가?
3. 관련성: 요청에 집중하고 있는가?

## 출력 형식
{
  "accuracy": { "score": N, "reason": "..." },
  "completeness": { "score": N, "reason": "..." },
  "relevance": { "score": N, "reason": "..." },
  "overall": N
}
```

### 규칙 기반 자동 검증

```markdown
# 자동 검증 가능한 항목

## 형식 검증
- JSON 유효성 (JSON.parse 성공)
- 마크다운 헤더 존재 여부
- 코드블록 언어 태그 포함 여부
- 응답 길이 범위 준수

## 내용 검증
- 필수 키워드 포함 여부
- 금지 키워드 부재 여부
- 이모지 사용 여부
- 서두 패턴("물론이죠!") 부재 여부

## 코드 검증
- 빌드 성공 여부 (컴파일)
- 타입 체크 통과 여부
- 린트 규칙 준수 여부
- 테스트 통과 여부
```

## 평가 결과 활용

### 개선 사이클

```markdown
# 평가 → 분석 → 개선 → 재평가 루프

1. 평가: 전체 테스트 세트로 현재 프롬프트 평가
2. 분석: 낮은 점수의 차원 식별
   - 정확성 낮음 → 예시 추가, 제약 강화
   - 일관성 낮음 → 형식 지시 강화, 구조화
   - 완전성 낮음 → 체크리스트 추가
   - 관련성 낮음 → 범위 제약 강화
   - 안전성 낮음 → 가드레일 추가
3. 개선: 분석 결과에 따라 프롬프트 수정
4. 재평가: 동일 테스트 세트로 재평가
5. 비교: 이전 버전과 점수 비교
```

### 버전 간 비교

```markdown
# 버전 비교 리포트
| 차원 | v1.0 | v1.1 | 변화 |
|------|------|------|------|
| 정확성 | 4.2 | 4.5 | +0.3 ↑ |
| 일관성 | 3.8 | 4.1 | +0.3 ↑ |
| 완전성 | 4.0 | 3.7 | -0.3 ↓ (회귀!) |
| 관련성 | 4.5 | 4.5 | ±0.0 |
| 안전성 | 5.0 | 5.0 | ±0.0 |
| 종합 | 4.25 | 4.30 | +0.05 |

회귀 분석:
- 완전성 하락: v1.1에서 "간결하게" 규칙 추가 후 에러 핸들링 누락 빈도 증가
- 조치: "간결하되 에러 핸들링은 항상 포함" 명시 → v1.2
```

## 평가 체크리스트

```markdown
□ 평가 차원(정확성, 일관성, 완전성, 관련성, 안전성)이 정의되었는가?
□ 각 차원의 스케일(1-5)과 기준이 명확한가?
□ 작업 유형에 맞게 가중치가 조정되었는가?
□ 자동 검증 가능한 항목이 스크립트화되었는가?
□ 테스트 세트가 다양한 시나리오를 커버하는가?
□ 버전 간 비교가 가능한 구조인가?
□ 평가 결과가 개선 액션으로 연결되는가?
```
