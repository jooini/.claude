# Chain-of-Thought

> 참조 링크: https://arxiv.org/abs/2201.11903, https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/chain-of-thought

---

## Chain-of-Thought (CoT) 개요

Chain-of-Thought는 모델에게 최종 답변 전에 단계별 추론 과정을 거치도록 유도하는 기법이다. 복잡한 문제에서 정확도를 크게 향상시킨다.

### CoT가 효과적인 작업

- 수학적/논리적 추론
- 멀티스텝 문제 해결
- 코드 디버깅 (원인 추적)
- 아키텍처 설계 결정
- 비교/평가 작업

### CoT가 불필요한 작업

- 단순 사실 질문 ("TypeScript의 최신 버전은?")
- 코드 형식 변환 (JSON → TypeScript interface)
- 번역, 요약 등 단순 변환

## Standard CoT (Few-shot CoT)

예시에 추론 과정을 포함해서 모델이 같은 패턴으로 추론하도록 유도한다.

```markdown
## 예시

Q: 이 API 엔드포인트의 보안 취약점을 분석해줘.
\```typescript
@Post('/transfer')
async transfer(@Body() body: { from: string; to: string; amount: number }) {
  await this.accountService.transfer(body.from, body.to, body.amount);
  return { success: true };
}
\```

A:
단계 1 — 인증/인가 확인:
- 인증 가드가 없음 → 누구나 호출 가능
- 계좌 소유자 확인 없음 → 타인 계좌에서 이체 가능

단계 2 — 입력 검증 확인:
- DTO 클래스 미사용 → class-validator 적용 불가
- amount에 음수 값 검증 없음 → 역방향 이체 가능
- from/to 동일 값 검증 없음

단계 3 — 비즈니스 로직 확인:
- 잔액 부족 처리 여부 불명 (서비스 레이어 확인 필요)
- 트랜잭션 격리 수준 확인 필요 (동시 이체 시 race condition)

단계 4 — 응답 확인:
- 에러 시 상세 정보 누출 가능성 (글로벌 에러 핸들러 확인 필요)

결론: 심각도 순으로 [인증 미비] > [입력 검증 부재] > [race condition 가능성]
```

## Zero-shot CoT

예시 없이 "단계별로 생각해봐" 류의 트리거 문구만으로 추론을 유도한다.

### 기본 트리거 문구

```markdown
# 영어
Let's think step by step.
Think through this carefully, step by step.

# 한국어
단계별로 분석해줘.
하나씩 살펴보자.
```

### 효과적인 Zero-shot CoT 패턴

```markdown
이 쿼리의 성능 문제를 분석해줘.
먼저 실행 계획을 추론하고, 각 단계에서 병목이 될 수 있는 부분을 식별한 뒤,
최적화 방안을 제시해줘.

\```sql
SELECT u.*, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2024-01-01'
GROUP BY u.id
HAVING order_count > 5
ORDER BY order_count DESC;
\```
```

### 도메인 특화 트리거

```markdown
# 코드 디버깅
"이 버그의 근본 원인을 추적해줘. 데이터 흐름을 따라가며 어디서 문제가 발생하는지 단계별로 분석해."

# 아키텍처 설계
"각 선택지의 장단점을 하나씩 분석하고, 우리 상황에 가장 적합한 것을 선택해줘."

# 코드 리뷰
"이 PR의 변경사항을 파일별로 검토해. 각 파일에서 잠재적 문제를 식별하고, 심각도를 매겨줘."
```

## 구조화된 CoT

추론 과정의 형식을 명시적으로 지정한다.

### 분석 프레임워크 제공

```markdown
다음 프레임워크로 이 설계 결정을 분석해줘:

1. **현재 상태**: 지금 어떤 상황인가
2. **선택지 나열**: 가능한 접근법 2~3가지
3. **각 선택지 평가**: 
   - 구현 복잡도 (상/중/하)
   - 성능 영향
   - 유지보수성
4. **결론**: 추천 선택지 + 근거
```

### 추론 태그 활용

Claude에서는 XML 태그로 추론 과정을 구조화할 수 있다.

```markdown
아래 문제를 분석해줘.

<thinking> 태그 안에 분석 과정을 작성하고,
<answer> 태그 안에 최종 답변을 작성해줘.

\```
<thinking>
1. 먼저 입력 데이터의 구조를 파악한다...
2. 다음으로 각 필드의 유효성을 검증한다...
3. 엣지 케이스를 고려한다...
</thinking>

<answer>
최종 구현 코드와 설명
</answer>
\```
```

## CoT 변형 기법

### Self-Consistency

같은 문제에 대해 여러 추론 경로를 생성하고, 다수결로 답을 결정한다.

```markdown
이 버그의 원인을 3가지 다른 관점에서 분석해줘:

관점 1 — 데이터 흐름 추적:
(입력 → 처리 → 출력 과정에서 어디가 문제인지)

관점 2 — 상태 변화 추적:
(각 단계에서 상태가 어떻게 변하는지)

관점 3 — 경계 조건 분석:
(null, 빈 배열, 최대값 등에서 어떻게 동작하는지)

세 분석의 공통 원인이 있으면 그것이 근본 원인일 가능성이 높다.
```

### Least-to-Most Prompting

복잡한 문제를 하위 문제로 분해한 뒤, 쉬운 것부터 해결한다.

```markdown
이 기능을 구현하기 위해 필요한 하위 작업을 나열하고,
가장 단순한 것부터 하나씩 구현해줘.

요구사항: 사용자 활동 기반 추천 시스템

하위 작업 (쉬운 것 → 어려운 것):
1. 사용자 활동 로그 스키마 정의
2. 활동 기록 API 구현
3. 활동 데이터 집계 쿼리
4. 유사도 계산 알고리즘
5. 추천 결과 API
```

### Tree of Thought (ToT)

여러 추론 분기를 탐색하고, 유망한 경로만 계속 진행한다.

```markdown
이 시스템 설계에 대해 3가지 접근법을 각각 2단계까지 전개해줘.
그 중 가장 유망한 1개를 선택해서 끝까지 완성해줘.

접근법 A: 모놀리식
  → 단계 1: ...
  → 단계 2: ...
  → 평가: 적합도 ★★☆

접근법 B: 마이크로서비스
  → 단계 1: ...
  → 단계 2: ...
  → 평가: 적합도 ★★★ ← 선택

접근법 C: 서버리스
  → 단계 1: ...
  → 단계 2: ...
  → 평가: 적합도 ★☆☆
```

## CoT 주의사항

### 과도한 CoT 회피

단순한 작업에 CoT를 강제하면 오히려 응답 품질이 떨어지고 토큰을 낭비한다.

```markdown
# 나쁜 예: 단순 작업에 CoT 강제
"hello world를 출력하는 Python 코드를 작성해줘. 단계별로 생각해서."

# 좋은 예: 복잡한 작업에 CoT 적용
"이 분산 시스템의 데이터 일관성 문제를 분석해줘. 단계별로 추론해서."
```

### 추론 길이 제어

추론이 너무 길어지면 핵심을 놓칠 수 있다.

```markdown
# 추론 길이 제한
각 단계는 2~3문장으로 요약하고, 전체 분석은 5단계 이내로 완료해줘.
```

## CoT 적용 체크리스트

- [ ] 작업이 멀티스텝 추론을 필요로 하는가
- [ ] Few-shot CoT vs Zero-shot CoT 중 적절한 것을 선택했는가
- [ ] 추론 프레임워크/구조를 제공했는가
- [ ] 추론 길이가 적절하게 제한되어 있는가
- [ ] 최종 답변과 추론 과정이 구분되어 있는가
