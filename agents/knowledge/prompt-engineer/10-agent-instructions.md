## 에이전트 프롬프트의 특수성

에이전트 프롬프트는 단순 Q&A 프롬프트와 근본적으로 다르다. 에이전트는 도구를 호출하고, 다단계 작업을 수행하며, 자율적으로 판단해야 한다. 프롬프트가 에이전트의 두뇌 설계도 역할을 한다.

## 단일 에이전트 지시문 구조

### 필수 구성 요소

```markdown
# [에이전트 이름]

## 역할 정의
당신은 [구체적 전문 분야]의 전문가입니다.
[핵심 기술 스택]을 사용합니다.

## 행동 규칙
- [핵심 행동 원칙 1]
- [핵심 행동 원칙 2]

## 도구 사용 규칙
- [도구 A]: [언제, 어떻게 사용하는지]
- [도구 B]: [사용 조건과 방법]

## 출력 형식
[응답의 구조와 포맷]

## 하지 말 것
- [금지 행동 목록]
```

### 실제 에이전트 지시문 예시

```markdown
# 백엔드 개발자

## 역할
NestJS + TypeORM + MariaDB 스택의 시니어 백엔드 개발자.
클린 아키텍처와 DDD 원칙을 따른다.

## 행동 규칙
- 구현 요청이면 바로 코드로 시작한다
- 코드는 전체를 출력한다. 부분 생략 금지
- 주석은 인라인으로만 넣는다

## 도구 사용
- Read: 기존 코드 확인 시 사용. 수정 전에 반드시 현재 상태 확인
- Edit: 코드 수정 시 사용. old_string은 고유해야 한다
- Grep: 코드 검색 시 사용. 파일명 모를 때 패턴으로 검색
- Bash: 빌드/테스트 실행 시 사용

## 하지 말 것
- 사용자가 요청하지 않은 리팩토링
- "좋은 질문이에요!" 류의 서두
- 확실하지 않은 정보를 확정적으로 서술
```

## 멀티 에이전트 프롬프트 설계

### 역할 분리 원칙

각 에이전트의 책임 범위를 명확히 구분해야 한다. 겹치는 영역이 있으면 충돌이 발생한다.

```markdown
# 역할 분리 매트릭스
| 작업         | 백엔드  | 프론트 | 코드리뷰어 | 테스터 |
|-------------|---------|--------|-----------|--------|
| API 설계     | ●       |        |           |        |
| UI 구현      |         | ●      |           |        |
| 코드 품질    |         |        | ●         |        |
| 테스트 작성  | △       | △      |           | ●      |
| 보안 검토    |         |        | ●         |        |

● = 주담당, △ = 보조 (주담당 에이전트 지시 하에)
```

### 에이전트 간 인터페이스

에이전트끼리 직접 소통하지 않고 오케스트레이터(메인 세션)를 통해 중계한다.

```markdown
# 오케스트레이션 규칙
1. 에이전트는 다른 에이전트를 직접 호출하지 않는다
2. 작업 결과는 오케스트레이터에게 반환한다
3. 오케스트레이터가 필요 시 다른 에이전트에 전달한다
4. 에이전트 간 의존성은 오케스트레이터가 순서를 관리한다

# 금지 패턴
에이전트 A: "이 작업은 에이전트 B에게 넘기겠습니다" ← 금지
에이전트 A: "이 부분은 프론트엔드 전문가의 검토가 필요합니다" ← 허용 (제안만)
```

### 파이프라인 패턴

순차적 에이전트 실행 흐름 설계.

```markdown
# 개발 파이프라인
1. 개발 에이전트 → 코드 작성
2. 테스트 에이전트 → 린트/빌드/테스트 검증
3. 리뷰 에이전트 → 코드 리뷰
4. (이슈 발견 시) 개발 에이전트 → 수정
5. 테스트 에이전트 → 재검증

# 파이프라인 지시문 (오케스트레이터 레벨)
개발 에이전트가 코드를 완성하면:
1. 변경된 파일 목록을 수집한다
2. 테스트 에이전트에 해당 파일 목록과 함께 검증을 요청한다
3. 테스트 통과 시 리뷰 에이전트에 전달한다
4. 리뷰 이슈가 있으면 개발 에이전트에 수정을 요청한다
```

## 도구 사용 지시

### 도구 선택 기준 설계

에이전트가 여러 도구를 사용할 수 있을 때, 선택 기준을 명시해야 한다.

```markdown
## 검색 도구 선택 기준
- 파일 위치를 아는 경우 → Read 직접 사용
- 파일명 패턴으로 찾을 때 → Glob
- 코드 내용으로 찾을 때 → Grep
- 의미 기반 검색 → RAG (mcp__local-rag__query_documents)

## 코드 수정 도구 선택 기준
- 기존 파일의 일부 수정 → Edit (old_string → new_string)
- 새 파일 생성 → Write
- 파일 전체 재작성 → Read 후 Write (기존 내용 확인 필수)

## 실행 도구 사용 기준
- 빌드 확인 → Bash: npm run build
- 테스트 실행 → Bash: npm test
- 파일 시스템 조회 → Bash: ls, pwd
```

### 도구 호출 순서 패턴

```markdown
## 코드 수정 작업 시 도구 호출 순서
1. Grep/Glob으로 관련 파일 검색
2. Read로 현재 코드 확인
3. Edit으로 수정 적용
4. Bash로 빌드/테스트 검증
5. 실패 시 2~4 반복

## 금지 순서
- Read 없이 Edit 하지 않는다 (현재 코드 모르면 old_string 틀림)
- 빌드 확인 없이 완료 선언하지 않는다
```

### 도구 실패 처리 지시

```markdown
## 도구 실패 시 행동
- Edit 실패 (old_string 불일치): Read로 현재 내용 재확인 후 재시도
- Bash 실패 (빌드 에러): 에러 메시지 분석 후 코드 수정
- Grep 결과 없음: 다른 키워드/패턴으로 재검색, 또는 Glob으로 파일 구조 확인
- 3회 연속 실패: 현재 상황을 사용자에게 보고하고 지시 요청
```

## 위임 패턴

### 작업 분해 후 위임

```markdown
# 오케스트레이터 지시문
사용자의 기능 요청을 받으면:
1. 작업을 BE/FE로 분해한다
2. API 스펙을 먼저 정의한다
3. BE 에이전트에 API 구현을 위임한다
4. BE 완료 후 FE 에이전트에 UI 구현을 위임한다
5. 양쪽 완료 후 통합 테스트를 테스트 에이전트에 위임한다
```

### 조건부 위임

```markdown
# 조건부 위임 규칙
코드리뷰 에이전트가 다음을 발견하면 해당 전문 에이전트에 위임:
- 성능 이슈 → 백엔드/프론트 에이전트에 최적화 요청
- 보안 이슈 → 보안 관점으로 백엔드 에이전트에 수정 요청
- UI/UX 이슈 → 디자이너 에이전트에 리뷰 요청
- 테스트 부족 → 테스트 에이전트에 테스트 작성 요청
```

### 에스컬레이션 패턴

```markdown
# 에스컬레이션 규칙
에이전트가 다음 상황에 처하면 오케스트레이터에 보고한다:
1. 자신의 전문 범위를 벗어나는 작업
2. 모호하여 판단할 수 없는 요구사항
3. 기존 코드와 충돌하는 수정이 필요한 경우
4. 3회 이상 동일 작업 실패

보고 형식:
- 현재 상태: [수행 중이던 작업]
- 문제: [발생한 이슈]
- 시도한 것: [이미 시도한 해결책]
- 필요한 것: [다음 단계에 필요한 정보/권한]
```

## 에이전트 자율성 수준 설계

### 수준별 패턴

```markdown
# Level 1 — 수동 (사용자 확인 필수)
매 단계마다 사용자에게 확인을 요청한다.
코드 수정 전에 수정 계획을 제시하고 승인을 받는다.

# Level 2 — 반자동 (핵심 판단만 확인)
파일 읽기, 검색은 자율적으로 수행한다.
코드 수정, 파일 생성은 수행 후 결과를 보고한다.
아키텍처 변경은 사전 확인을 요청한다.

# Level 3 — 자동 (결과만 보고)
작업 전체를 자율적으로 수행하고 결과만 보고한다.
실패 시에만 사용자에게 알린다.
```

### 자율성 지시문 예시

```markdown
## 자율성 규칙 (Level 2)
다음은 자율적으로 수행한다:
- 파일 읽기/검색
- 에러 메시지 분석
- 기존 패턴을 따르는 코드 수정

다음은 수행 전 보고한다:
- 새 파일 생성
- 기존 API 시그니처 변경
- 의존성(패키지) 추가/삭제

다음은 사용자 승인 후 수행한다:
- 데이터베이스 스키마 변경
- 환경 변수 추가
- 기존 테스트 삭제/대폭 수정
```

## 에이전트 지시문 안티패턴

### 1. 과도한 자율성

```markdown
# 나쁜 예
"알아서 최선의 판단을 내려라"
→ 예측 불가능한 행동, 디버깅 어려움

# 좋은 예
"다음 판단 기준을 따른다: [구체적 기준 나열]"
```

### 2. 역할 경계 모호

```markdown
# 나쁜 예
"필요하면 프론트엔드 코드도 수정해라"
→ 백엔드 에이전트가 FE 코드를 임의로 수정

# 좋은 예
"프론트엔드 수정이 필요하면 변경 사항을 명시하고 오케스트레이터에 보고한다"
```

### 3. 도구 지시 누락

```markdown
# 나쁜 예
에이전트 지시에 도구 사용 규칙 없음
→ 불필요한 도구 호출, 비효율적 검색

# 좋은 예
각 도구별 사용 조건, 순서, 실패 처리를 명시
```
