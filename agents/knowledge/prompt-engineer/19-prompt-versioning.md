# 프롬프트 버전 관리

---

## 프롬프트 버전 관리의 필요성

프롬프트는 코드와 마찬가지로 시간이 지나면서 변경된다. 변경 이력 추적, 성능 비교, 롤백이 가능해야 프로덕션 환경에서 안전하게 프롬프트를 운영할 수 있다.

## 버전 번호 체계

### Semantic Versioning 적용

```markdown
# 프롬프트 버전: MAJOR.MINOR.PATCH

MAJOR: 역할, 핵심 규칙이 바뀌는 대규모 변경
  예: 역할을 백엔드→풀스택으로 변경, 출력 형식 전면 변경

MINOR: 새 규칙 추가, 기존 규칙 수정
  예: 새 제약 조건 추가, 예시 추가, 도구 규칙 변경

PATCH: 표현 수정, 오탈자, 미세 조정
  예: 문구 다듬기, 오탈자 수정, 순서 변경

# 예시
v1.0.0 — 최초 프로덕션 배포
v1.1.0 — 에러 핸들링 규칙 추가
v1.1.1 — 규칙 문구 명확화
v2.0.0 — 에이전트 구조 전면 재설계
```

### 버전 메타데이터

```markdown
# 각 버전에 기록할 메타데이터
- 버전 번호
- 변경 일자
- 변경 사유
- 변경 내용 요약
- 대상 모델 (claude-sonnet-4-20250514 등)
- 평가 점수 (테스트 결과)
- 담당자
```

## 변경 이력 관리

### Changelog 형식

```markdown
# CHANGELOG.md

## v2.1.0 (2024-03-15)
### Added
- 코드 리뷰 시 보안 체크리스트 추가
- TypeScript strict mode 규칙 추가

### Changed
- "간결하게" 규칙을 "3줄 이내" 로 구체화
- 에러 핸들링 규칙 강화

### Fixed
- 긴 코드에서 // ...동일 처리가 발생하던 문제 수정

### Removed
- 사용하지 않는 Python 관련 규칙 제거

---

## v2.0.0 (2024-02-01)
### Breaking Changes
- 에이전트 시스템 전면 재설계
- 레이어 구조 2단계 → 3단계로 변경
```

### Git 기반 버전 관리

```markdown
# 프롬프트를 Git으로 관리하는 방법

## 디렉토리 구조
prompts/
├── system/
│   ├── backend-developer.md
│   ├── frontend-developer.md
│   └── code-reviewer.md
├── templates/
│   └── review-template.md
├── CHANGELOG.md
└── README.md

## 커밋 컨벤션
feat(prompt): 백엔드 에이전트에 보안 체크 규칙 추가
fix(prompt): 코드 리뷰 에이전트의 형식 불일치 수정
refactor(prompt): 공통 규칙을 글로벌 레이어로 이동
test(prompt): 새 테스트 케이스 5개 추가

## 브랜치 전략
main — 현재 프로덕션 프롬프트
develop — 개발/테스트 중인 프롬프트
prompt/feature-name — 특정 기능 변경
```

### 변경 사유 기록

모든 프롬프트 변경에는 사유를 기록한다.

```markdown
# 변경 사유 템플릿
## 변경 요청
- 요청자: [이름/팀]
- 일자: [날짜]
- 배경: [왜 변경이 필요한지]

## 현재 문제
- [현재 프롬프트의 어떤 동작이 문제인지]
- [재현 방법 또는 예시]

## 제안 변경
- [변경할 내용]
- [기대 효과]

## 영향 범위
- [이 변경으로 영향받는 기능/시나리오]
- [회귀 테스트 필요 범위]
```

## A/B 테스트

### A/B 테스트 설계

```markdown
# 프롬프트 A/B 테스트 프로세스

## 1. 가설 수립
"코드 리뷰 에이전트에 구체적 예시를 추가하면
보안 이슈 감지율이 향상될 것이다"

## 2. 변수 정의
- 독립 변수: 프롬프트 버전 (A: 기존, B: 예시 추가)
- 종속 변수: 보안 이슈 감지율, 전체 리뷰 품질
- 통제 변수: 동일 모델, 동일 테스트 케이스, 동일 temperature

## 3. 테스트 세트 준비
- 보안 이슈가 있는 코드 20개
- 보안 이슈가 없는 코드 10개
- 총 30개 케이스

## 4. 실행
- A 프롬프트로 30개 케이스 실행
- B 프롬프트로 동일 30개 케이스 실행
- 각 케이스를 3회 반복 (일관성 확인)

## 5. 결과 비교
| 지표 | A (기존) | B (예시 추가) |
|------|---------|-------------|
| 보안 이슈 감지율 | 75% | 90% |
| 오탐율 | 5% | 8% |
| 평균 리뷰 품질 | 4.2 | 4.4 |
| 평균 토큰 사용 | 1200 | 1500 |
```

### 다변량 테스트

여러 변수를 동시에 테스트할 때.

```markdown
# 다변량 테스트 설계
## 변수 조합
| 변형 | 예시 수 | 지시 강도 | 톤 |
|------|---------|---------|------|
| A | 0개 | 기본 | 전문적 |
| B | 1개 | 기본 | 전문적 |
| C | 1개 | 강화 | 전문적 |
| D | 1개 | 강화 | 직접적 |

각 변형을 동일 테스트 세트로 실행 후 최적 조합 선택
```

### A/B 테스트 주의사항

```markdown
# 주의사항
1. 한 번에 하나의 변수만 변경 (순수 A/B 테스트 시)
2. 충분한 샘플 크기 (최소 20개 케이스)
3. 반복 실행으로 변동성 확인 (최소 3회)
4. 모델 버전 고정 (테스트 중 모델 업데이트 방지)
5. 비용도 함께 비교 (성능이 비슷하면 저비용 선택)
```

## 롤백 전략

### 롤백이 필요한 상황

```markdown
# 즉시 롤백
- 안전 관련 회귀 (시스템 프롬프트 유출, 금지 행동 수행)
- 핵심 기능 실패 (코드 생성 불가, 도구 호출 실패)
- 심각한 성능 저하 (정확도 20% 이상 하락)

# 모니터링 후 롤백 판단
- 부분적 품질 저하 (특정 시나리오에서만 문제)
- 비용 증가 (토큰 사용량 30% 이상 증가)
- 사용자 피드백 부정적
```

### 롤백 프로세스

```markdown
# 롤백 절차

## 1. 문제 감지
- 자동 평가 시스템에서 점수 하락 감지
- 사용자 피드백 또는 수동 확인

## 2. 영향 평가
- 어떤 시나리오가 영향받는지 식별
- 심각도 판단 (Critical / Major / Minor)

## 3. 롤백 실행
- Git에서 이전 버전 checkout
- 프로덕션 프롬프트를 이전 버전으로 교체
- 롤백 사실을 팀에 공유

## 4. 원인 분석
- 어떤 변경이 문제를 일으켰는지 분석
- 테스트에서 놓친 시나리오 식별

## 5. 재수정
- 원인을 수정한 새 버전 작성
- 누락된 테스트 케이스 추가
- 전체 회귀 테스트 후 재배포
```

### 카나리 배포

전체 배포 전에 일부 트래픽에만 새 프롬프트를 적용한다.

```markdown
# 카나리 배포 전략

1단계: 새 프롬프트를 내부 테스트 환경에 배포 (1일)
2단계: 전체 트래픽의 5%에 적용 (2~3일)
3단계: 문제 없으면 25%로 확대 (2~3일)
4단계: 50% → 100% 순차 확대

모니터링 지표:
- 에러율 변화
- 평균 응답 품질 점수
- 사용자 피드백 (부정적 반응률)
- 토큰 사용량 변화
```

## 프롬프트 라이프사이클

```markdown
# 프롬프트 라이프사이클 관리

Draft → Review → Test → Stage → Production → Monitor → Retire

Draft: 초안 작성
Review: 팀 리뷰 (PR 형식)
Test: 테스트 세트 실행, 평가 통과
Stage: 스테이징 환경 배포, 카나리 테스트
Production: 전체 배포
Monitor: 지속적 모니터링, 피드백 수집
Retire: 더 이상 사용하지 않는 버전 아카이브

# 각 단계의 게이트 조건
Draft → Review: 최소 기능 동작 확인
Review → Test: 리뷰 승인
Test → Stage: 평가 점수 A 이상, 회귀 없음
Stage → Production: 카나리 2일 이상 문제 없음
Production → Retire: 새 버전 안정화 후 1주
```

## 버전 관리 도구 및 자동화

```markdown
# 자동화 가능한 영역
1. Git hook으로 프롬프트 변경 시 자동 린트
2. CI/CD 파이프라인에 프롬프트 테스트 포함
3. 배포 시 자동 회귀 테스트 실행
4. 프로덕션 프롬프트 성능 대시보드
5. 점수 하락 시 자동 알림

# 최소 자동화 권장 사항
- Git 버전 관리 (필수)
- Changelog 유지 (필수)
- 배포 전 핵심 테스트 실행 (필수)
- 모니터링 + 알림 (권장)
- A/B 테스트 프레임워크 (선택)
```
