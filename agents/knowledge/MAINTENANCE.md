# Knowledge 유지보수 가이드

이 문서는 `agents/knowledge/` 시스템의 품질 기준과 유지보수 규칙을 정의한다.

---

## 1. 파일 구조 표준

모든 knowledge 파일은 다음 구조를 따른다:

```markdown
# 제목

> 원본: https://ironact.gitbook.io/...
> ✏️ 원본 대비 내용이 보강되었습니다    ← 원본 외 내용 추가 시

---

## 개요

(1-2문장. 이 문서가 다루는 범위와 핵심 메시지. RAG 검색의 첫 chunk가 되므로 핵심 키워드 포함 필수)

## 첫 번째 섹션
...
```

### 필수 요소

| 요소 | 설명 |
|------|------|
| H1 제목 | 파일의 주제를 명확히 나타내는 영문 제목 |
| 원본 URL | gitbook 원본 링크 (출처 추적용) |
| 보강 메모 | 원본 외 내용이 추가된 경우 `> ✏️ 원본 대비 내용이 보강되었습니다` 표기 |
| `## 개요` | 1-2문장 요약. RAG retrieval 정밀도에 직접 영향 |
| `---` 구분선 | 메타(제목+URL) 와 본문 사이 |

### 언어별 backend 파일 H1 규칙

공통 파일과 구분하기 위해 H1에 언어/프레임워크를 명시한다:

```
# API Design — Python/FastAPI
# API Design — Kotlin/Spring Boot
# API Design — PHP/CodeIgniter
```

## 2. 품질 기준

### 파일 크기

| 기준 | 값 | 비고 |
|------|-----|------|
| 최소 | 150줄 | 이 미만이면 내용이 얕을 가능성 높음 |
| 권장 | 200-500줄 | RAG chunk 효율 최적 범위 |
| 최대 | 700줄 | 이 초과 시 파일 분할 검토 |

### 문체

- **해라체** 통일 (~한다, ~이다, ~다)
- 높임말(합니다, 됩니다, 습니다) 사용 금지
- 코드 블록 내 주석은 한국어/영어 혼용 가능

### 코드 예시

- Good/Bad 패턴: `// ❌` / `// ✅` (또는 `-- ❌` / `-- ✅` for SQL)
- 프레임워크/라이브러리 구체적으로 명시
- 추상적 원칙보다 실행 가능한 코드 우선

### 내용 구성

- H2(`##`)로 대주제, H3(`###`)으로 소주제
- 비교 테이블 적극 활용 (trade-off 표현에 효과적)
- 판단 기준/의사결정 가이드 포함 권장
- 체크리스트는 파일 하단에 배치

---

## 3. 네이밍 규칙

### 파일명

```
{번호}-{주제}.md
```

- 번호: 01부터 연속 (갭 없이)
- 주제: 케밥 케이스 (영문 소문자, 하이픈 구분)
- 예: `01-api-design.md`, `13-performance.md`

### 에이전트별 번호 관리

- 번호 갭이 생기면 마지막 파일을 앞으로 재배정
- 새 파일 추가 시 마지막 번호 +1

---

## 4. RAG 최적화

### retrieval 품질에 영향을 주는 요소

1. **`## 개요` 섹션**: 파일의 의미를 대표하는 첫 chunk. 핵심 키워드 필수
2. **H1 제목**: 검색 쿼리와 직접 매칭되는 진입점
3. **코드 주석**: 한국어 쿼리 ↔ 영어 기술 용어 브릿지 역할
4. **파일 크기 균일성**: 극단적 편차(66줄 vs 960줄)는 chunk 품질 불균형 유발

### 크로스-에이전트 주제 중복

accessibility, testing 등 여러 에이전트가 공유하는 주제는 **각 에이전트의 관점 차이**를 반영하므로 허용한다. 단, 동일 에이전트의 공통/언어별 파일 간 내용 중복은 최소화한다.

## 5. 변경 이력 추적

### 원본 대비 보강 표기

원본(gitbook) 외에 내용을 추가/보강한 경우 반드시 표기한다:

```markdown
> 원본: https://ironact.gitbook.io/...
> ✏️ 원본 대비 내용이 보강되었습니다
```

이를 통해:
- 원본 업데이트 시 보강 부분이 덮어씌워지지 않도록 보호
- 어떤 파일이 커스텀 내용을 포함하는지 빠르게 파악 가능
