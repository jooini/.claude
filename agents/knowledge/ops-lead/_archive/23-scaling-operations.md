## 1. 운영 확장 시 주요 과제

```
성장하면 복잡성이 증가:
  클라이언트 3개 → 10개 → 20개
    → 소통 채널 폭발
    → 품질 일관성 유지 어려움
    → 지식 공유 어려움
    → 온보딩 반복
```

---

## 2. 확장을 위한 표준화

```
표준화 우선순위:
  1. 가장 많이 반복하는 것부터
  2. 오류가 자주 발생하는 것
  3. 팀원이 "이건 어떻게 해요?" 자주 묻는 것

표준화 방법:
  SOP 문서화 → 체크리스트 → 템플릿 → 자동화

예시:
  클라이언트 온보딩 과정
  현재: PM마다 다름, 매번 3~5일 소요
  표준화: 체크리스트 + 템플릿 → 1~2일로 단축
```

---

## 3. 프리랜서 네트워크 구축

```
확장을 위한 탄력적 인력 모델:

핵심 팀 (정규직):
  - 전략, 품질 관리, 클라이언트 관계
  - 항상 고정 필요한 역할

탄력 인력 (프리랜서):
  - 콘텐츠 작성, 디자인, 번역
  - 물량에 따라 조절

프리랜서 풀 관리:
  - 각 분야 2~3명 예비 유지
  - 품질 등급별 분류 (A/B/C)
  - 정기 작업으로 관계 유지
  - 온보딩 문서 표준화
```

---

## 4. 위임 구조

```
확장 시 PM의 위임:

레벨 1 (직접 실행):
  전략 수립, C레벨 클라이언트 소통

레벨 2 (승인):
  납품물 최종 승인, 이슈 대응

레벨 3 (모니터링):
  일상 업무 진행 상황

레벨 4 (위임):
  팀원이 자율 처리, 결과만 보고

핵심: PM이 레벨 1~2에 집중할 수 있도록
      레벨 3~4는 팀원/자동화로
```

---

## 5. 확장 단계별 체크리스트

```
팀 2~3명 → 5~6명 전환 시:
  [ ] 역할 전문화 (콘텐츠 전담 vs 운영 전담)
  [ ] 팀 내 커뮤니케이션 구조 정립
  [ ] 지식 베이스 구축 시작
  [ ] 핵심 SOP 최소 5개 문서화

5~6명 → 10명+ 전환 시:
  [ ] 팀장/시니어 레이어 추가
  [ ] 프로젝트 관리 도구 고도화
  [ ] 자동화 투자 (리포트, 알림 등)
  [ ] 품질 관리 체계화 (QC 담당자 지정)
```

---

## 6. 안티패턴

- **모든 것을 PM이 직접**: 확장에 한계 → 위임 구조 필수
- **표준화 없는 확장**: 10개 클라이언트가 10개 다른 방식
- **품질 희생 성장**: 빠른 확장 → 품질 저하 → 이탈 → 악순환
- **채용만으로 확장**: 프로세스 없이 사람만 추가 → 혼선
- **문서화 미루기**: 바빠서 나중에 → 영원히 못 함
