---

## 1. 문서화가 중요한 이유

- **지식 보존**: 특정 사람만 아는 것 → 팀 자산으로
- **온보딩**: 신규 팀원이 빠르게 파악
- **일관성**: 누가 해도 같은 방식으로
- **감사/검증**: 의사결정 근거 추적 가능

---

## 2. 문서 유형별 기준

### SOPs (Standard Operating Procedures)

```markdown
# SOP: 클라이언트 월간 리포트 발행

**목적**: 매월 5일 클라이언트에게 성과 리포트 발행
**담당**: PM
**주기**: 매월

## 단계별 절차

### 1단계: 데이터 수집 (D-3, 매월 2일)
1. Google Analytics 접속 → 대상 클라이언트 뷰 선택
2. 기간: 전월 1일 ~ 말일 설정
3. 주요 지표 복사: 세션, 페이지뷰, 이탈률, 전환
4. Google Sheets 리포트 탬플릿에 붙여넣기 [링크]

### 2단계: 리포트 작성 (D-2, 매월 3일)
1. 탬플릿 열기 [링크]
2. 데이터 입력 및 차트 자동 업데이트 확인
3. 인사이트 섹션 작성 (전월 대비, 목표 대비)
4. 다음 달 제안 작성

### 3단계: 검토 및 발송 (D-1~D-day)
1. 팀 리드 검토 요청 (슬랙 @[이름])
2. 피드백 반영
3. 이메일 발송 (탬플릿: [링크])
4. 발송 완료 후 트래킹 시트 업데이트 [링크]

## 문제 발생 시
- 데이터 접근 불가: IT팀 티켓 등록 [링크]
- 납기 어려울 시: 에스컬레이션 가이드 참고 [링크]
```

## 3. 문서 명명 규칙

```
파일명 형식:
  [날짜]_[클라이언트]_[문서유형]_[버전]

예시:
  2024-03_A사_월간리포트_v1.pdf
  2024-03-15_B사_미팅노트.md
  2024-04_C사_콘텐츠캘린더_v2.xlsx

버전 관리:
  v1: 최초 작성
  v2: 클라이언트 피드백 반영
  v_final: 최종본
  (v_final 이후 수정 시 → v_final_20240315)
```

## 4. 지식 베이스 구조 (Notion 기준)

```
📁 회사 위키
  📁 온보딩
    📄 신규 팀원 가이드
    📄 도구 목록 및 접근 권한
    📄 용어집
  📁 프로세스
    📄 클라이언트 온보딩 SOP
    📄 콘텐츠 워크플로우
    📄 리포트 발행 SOP
  📁 클라이언트
    📁 A사
      📄 계약 정보 (접근 제한)
      📄 브랜드 가이드라인
      📄 히스토리 & 미팅 노트
  📁 템플릿
    📄 미팅 아젠다
    📄 콘텐츠 브리프
    📄 월간 리포트
```

## 5. 문서 유지 관리

```
문서 생명 주기:
  작성 → 검토 → 승인 → 활용 → 업데이트 → (폐기)

정기 검토:
  분기마다 주요 SOP 최신성 확인
  담당자 변경 시 즉시 업데이트
  프로세스 변경 시 동시에 문서 업데이트

문서 오너십:
  각 문서에 오너 지정 (최종 수정 책임자)
  오너 부재 시 이관 절차 명확히
```

---

## 6. 안티패턴

- **문서화 없이 구두 전달만**: 퇴사 시 지식 유실
- **문서 작성 후 방치**: 오래된 SOP가 더 위험함
- **너무 상세한 문서**: 유지보수 불가 → 핵심만
- **접근성 없는 문서**: 어디에 있는지 모르면 없는 것
- **문서 오너십 없음**: 누구도 업데이트 안 함
