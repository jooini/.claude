# SLA Management

> 원본: https://ironact.gitbook.io/ironact-docs/CrEhRPJQJpia3xh9iqbi/knowledge-ops-lead/sla-management

---

## 1. SLA란

Service Level Agreement. 서비스 품질 기준을 수치로 명문화한 계약.
"기대치"를 "측정 가능한 약속"으로 만드는 것.

---

## 2. SLA 항목 예시

```markdown
## 납품 SLA

| 작업 유형 | 기본 납품 기한 | 수정 기한 |
|----------|--------------|---------|
| 블로그 포스트 (1000자) | 영업일 3일 | 피드백 후 1일 |
| SNS 콘텐츠 (월 20개) | 매월 25일 | 피드백 후 1일 |
| 성과 리포트 | 익월 5일 | - |
| 광고 소재 | 영업일 5일 | 피드백 후 2일 |

## 커뮤니케이션 SLA

| 채널 | 응답 기준 |
|------|---------|
| 슬랙 긴급 태그 | 2시간 이내 |
| 슬랙 일반 | 4시간 이내 (업무 시간) |
| 이메일 | 24시간 이내 |

## 수정 요청 SLA

- 1차 수정: 무제한 (계약 범위 내)
- 3차 이상 수정: 추가 비용 협의
- 범위 외 수정 요청: 별도 견적 후 진행
```

## 3. SLA 모니터링

```markdown
## 주간 SLA 현황 (예시)

기간: 3/4 ~ 3/8

| 클라이언트 | 납품 건수 | 납기 준수 | 평균 납품 시간 | 비고 |
|-----------|---------|---------|------------|------|
| A사 | 5건 | 5/5 (100%) | D+2.3일 | - |
| B사 | 3건 | 2/3 (67%) | D+4.1일 | 1건 지연 (사유: 자료 늦게 수령) |
| C사 | 8건 | 8/8 (100%) | D+1.8일 | - |

이슈:
- B사 1건 지연: 클라이언트 측 브리프 전달 지연(3/6→3/7)으로 납품 1일 지연.
  클라이언트 확인 완료, 이슈 없음.
```

## 4. SLA 위반 처리

```
SLA 위반 발생 시 프로세스:

1. 즉시 클라이언트에게 알림 (위반 확정 즉시)
   "안녕하세요, [작업명]이 약속된 [날짜]까지 납품이 어렵게 됐습니다.
   [사유] 때문이며, [새 일정]까지 완료하겠습니다."

2. 내부 원인 분석
   - 클라이언트 측 요인 (자료 지연, 피드백 지연)
   - 내부 요인 (인력, 작업 복잡도 예측 실패)

3. 재발 방지 조치
   - 프로세스 개선
   - 버퍼 시간 추가
   - 조기 경보 시스템 강화

4. SLA 조항 패널티 적용 (있는 경우)
   - 계약서 확인
   - 협의 후 적용
```

## 5. SLA 설정 원칙

```
현실적으로 설정:
  - 실제 달성 가능한 수치 + 10% 버퍼
  - "95% 준수" 목표 → 실제 98% 준수 가능하면 95%로 설정

측정 가능하게:
  - "빠르게" → "영업일 3일 이내"
  - "충분한 수정" → "2차 수정까지 무료"

예외 사항 명시:
  - 클라이언트 측 지연으로 인한 납기 조정
  - 공휴일/연휴 제외
  - 불가항력 (천재지변 등)
```

---

## 6. 안티패턴

- **SLA 없이 서비스**: 기대치 불일치 → 분쟁
- **달성 불가능한 SLA**: 항상 위반 → 신뢰 손상
- **SLA 모니터링 안 함**: 위반 인지가 늦음
- **위반 숨기기**: 클라이언트가 나중에 알면 더 큰 신뢰 손상
- **패널티 없는 SLA**: 강제력 없는 약속 → 형식적 문서로 전락
