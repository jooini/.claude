## Outcome-Based Roadmap: Now / Next / Later

전통적인 "날짜 + 기능" 로드맵이 아닌 **결과(Outcome) 중심** 로드맵.

```
┌─────────────────────────────────────────────────────────┐
│                    Product Roadmap                       │
├──────────────┬──────────────────┬───────────────────────┤
│     NOW      │      NEXT        │       LATER           │
│   (이번 분기) │   (다음 분기)     │    (향후 6-12개월)    │
│              │                  │                       │
│  Outcome 1   │  Outcome 3       │  Outcome 5            │
│  높은 확신    │  보통 확신        │  낮은 확신             │
│  구체적 범위  │  대략적 범위      │  방향만               │
├──────────────┼──────────────────┼───────────────────────┤
│  확정 80%     │  확정 50%        │  확정 20%             │
└──────────────┴──────────────────┴───────────────────────┘
```

**장점:**
- 날짜 약속 없음: "Q2에 이 기능"이 아닌 "다음에 이 결과를 목표로"
- 유연성: 학습에 따라 조정 가능
- Outcome 중심: "무엇을 만들지"가 아닌 "어떤 변화를 만들지"
- 불확실성 수용: 미래로 갈수록 덜 구체적인 것이 정직

### Outcome 작성법

- ❌ "대시보드 리뉴얼" (기능)
- ✅ "팀 리더가 5분 이내에 프로젝트 상태를 파악할 수 있다" (결과)

- ❌ "모바일 앱 출시" (기능)
- ✅ "이동 중에도 핵심 작업을 처리할 수 있다" (결과)

## Theme-Based vs Feature-Based 로드맵

**Feature-Based (피해야 할 방식)**
```
Q1: 다크 모드, SSO, 대시보드 리뉴얼
Q2: 모바일 앱, API v2, 팀 관리
```
문제점: 날짜에 기능 약속 → 못 지키면 신뢰 하락. "왜?"가 빠져 있음. 변경이 어려움.

**Theme-Based (추천하는 방식)**
```
Q1 Theme: "첫 경험 혁신" — 신규 사용자가 10분 내에 핵심 가치를 경험
Q2 Theme: "팀 협업 강화" — 팀원 간 작업 가시성과 소통 개선
Q3 Theme: "자동화로 시간 절약" — 반복 작업을 80% 줄이기
```
장점: 전략적 방향 명확, 구체적 기능은 Discovery로 결정, 유연하게 조정 가능.

## 로드맵 커뮤니케이션

### 이해관계자별 맞춤

| 대상 | 관심사 | 로드맵 형식 | 업데이트 주기 |
|------|-------|-----------|------------|
| **경영진/보드** | 비즈니스 임팩트, 전략 정합성 | High-level themes + 핵심 지표 | 분기 |
| **Engineering** | 기술 방향, 의존성, 리소스 | 상세 backlog + 기술 요구사항 | 격주/스프린트 |
| **Sales/CS** | 고객 약속, 경쟁력 | 주요 기능 타임라인 + 고객 가치 | 월간 |
| **전체 팀** | 방향 감각, 동기 부여 | Vision 연결 + 진행 상황 | 월간 all-hands |

### 경영진 로드맵 프레젠테이션 구조

```
1. 전략 리마인드 (1분) — 비전과 이번 분기 전략 방향
2. 핵심 결과 (3분) — 지난 분기 성과: 지표 변화, 학습, 주요 출시
3. 로드맵 (5분)
   → Now: 현재 집중하는 것과 기대 임팩트
   → Next: 다음 분기 방향
   → Later: 장기 탐색 영역
4. 리스크 & 의사결정 필요사항 (2분)
5. Q&A (5분)
```

## 분기별 계획 수립 프로세스

```
Week -4: 데이터 수집 — 지난 분기 성과, 고객 피드백, 시장 동향
Week -3: 전략 정렬 — 경영진과 방향 확인, OKR 초안, Cross-functional input
Week -2: 우선순위 결정 — Opportunity scoring, 리소스 계획, 트레이드오프
Week -1: 확정 & 커뮤니케이션 — 최종 확정, 이해관계자 커뮤니케이션
Week 1-12: 실행 & 모니터링 — 주간 Sprint, 월간 체크, 분기 말 회고
```

**분기 계획 회의 어젠다 (약 2시간):**
1. 지난 분기 회고 (30분) — 목표 vs 실제, 학습, 개선점
2. 전략 정렬 (30분) — 회사 목표 연결, 핵심 가정 점검
3. 로드맵 수립 (60분) — Opportunity 리뷰, 우선순위, 리소스
4. 리스크 & 의존성 (15분) — 기술, 팀 간, 외부 의존성
5. 확정 & 다음 스텝 (15분)

## 로드맵 업데이트 원칙

- **정기**: 월 1회 진행 상황 반영
- **비정기**: 큰 학습, 전략 변화, 시장 급변 시
- 변경 사항 + 이유를 명확히 전달 ("왜 바뀌었는가?"에 항상 답)
- 영향받는 이해관계자에게 사전 고지

**로드맵 ≠ 약속:** 로드맵은 현재 시점의 최선의 계획이지, 변하지 않는 약속이 아니다. 학습에 따라 변하는 것이 정상. 단, 너무 자주 바뀌면 신뢰 문제 → 변경 이유의 일관성이 중요.
