# 리뷰 커뮤니케이션

> 참조 링크: https://google.github.io/eng-practices/review/reviewer/comments.html, https://conventionalcomments.org/

---

## 개요

코드 리뷰의 효과는 코멘트의 품질과 톤에 크게 좌우된다. 건설적인 피드백, 일관된 심각도 분류, 적절한 코멘트 톤은 팀 문화와 코드 품질을 동시에 높인다.

## 1. 건설적 피드백

### 나쁜 코멘트 vs 좋은 코멘트

```typescript
// 리뷰 대상 코드
async getUser(id: string): Promise<User> {
  const user = await this.userRepo.findOne({ where: { id } });
  if (!user) return null;
  return user;
}
```

```
❌ 나쁜 코멘트:
"이게 뭐예요? null 반환하면 안 되죠."
"이런 식으로 하면 안 됩니다."
"왜 이렇게 짰어요?"

✅ 좋은 코멘트:
"[suggestion] 반환 타입이 `Promise<User>`인데 null을 반환할 수 있어요.
`Promise<User | null>`로 변경하거나, NotFoundException을 던지는 것은 어떨까요?
호출하는 쪽에서 null 체크를 놓칠 수 있어서 예외를 권장합니다."
```

### 피드백 작성 원칙

```
1. 문제를 지적할 때 "왜"를 설명한다
   ❌ "이 변수명 바꿔주세요."
   ✅ "[nitpick] `data`보다 `userProfile`이 의도를 더 잘 드러내서 이후 유지보수가 편해질 것 같아요."

2. 대안을 함께 제시한다
   ❌ "이 로직은 비효율적이에요."
   ✅ "[suggestion] 현재 O(n²)인데, Map으로 lookup 테이블을 만들면 O(n)으로 개선할 수 있어요."

3. 질문 형태로 작성한다 (확실하지 않을 때)
   ❌ "이건 버그입니다."
   ✅ "[question] 여기서 `status`가 null인 경우도 들어올 수 있을 것 같은데, 의도된 건가요?"

4. 칭찬도 남긴다
   ✅ "[praise] 이 추상화 좋네요! 확장성이 좋아졌어요."
```

### 피드백 체크리스트

- [ ] 코멘트에 "왜 변경이 필요한지" 이유가 포함되어 있는가?
- [ ] 대안 또는 개선 방향을 제시했는가?
- [ ] 개인 취향이 아닌 객관적 기준(성능, 안전성, 가독성)에 근거하는가?
- [ ] 작성자의 의도를 먼저 이해하려 했는가?

## 2. 심각도 분류

### Conventional Comments 기반 라벨

```
[blocker]    — 머지 불가. 반드시 수정 필요 (보안, 데이터 손실, 크래시)
[issue]      — 버그 또는 명확한 문제. 수정 필요
[suggestion] — 개선 제안. 선택적이지만 권장
[question]   — 이해를 위한 질문. 의도 확인
[nitpick]    — 사소한 스타일/포맷. 무시 가능
[praise]     — 좋은 코드에 대한 칭찬
[thought]    — 아이디어 공유. 이번 PR에서 액션 불필요
[chore]      — 오타, import 정리 등 단순 작업
```

### 심각도별 예시

```typescript
// 리뷰 대상 코드
@Post('/login')
async login(@Body() dto: LoginDto): Promise<TokenResponse> {
  const user = await this.userRepo.findOne({ where: { email: dto.email } });
  if (user && user.password === dto.password) {  // 평문 비교!
    const token = jwt.sign({ id: user.id }, 'secret-key'); // 하드코딩 시크릿
    return { accessToken: token };
  }
  throw new UnauthorizedException();
}
```

```
[blocker] 비밀번호를 평문으로 비교하고 있어요. bcrypt 등 해싱 비교를 사용해야 합니다.
레인보우 테이블 공격에 취약하고, DB 유출 시 전체 계정이 노출됩니다.

[blocker] JWT secret이 하드코딩되어 있어요. 환경 변수로 분리해야 합니다.
코드 저장소에 시크릿이 노출됩니다.

[issue] 로그인 실패 시 "이메일이 없는지" vs "비밀번호가 틀린지" 구분 없이
동일한 UnauthorizedException을 던지는 건 좋아요 (타이밍 공격 방지).
하지만 실패 로그는 남겨야 합니다.

[suggestion] 로그인 시도 횟수 제한(rate limiting)을 추가하면 브루트포스 방어가 됩니다.
```

### 심각도 분류 기준

| 심각도 | 기준 | 액션 |
|--------|------|------|
| blocker | 보안, 데이터 손실, 장애 | 반드시 수정 후 재리뷰 |
| issue | 버그, 로직 에러 | 머지 전 수정 |
| suggestion | 개선 가능 | 작성자 판단에 위임 |
| nitpick | 스타일, 네이밍 | 무시 가능 |

## 3. 코멘트 톤

### 피해야 할 톤

```
❌ 명령형/권위적
"이거 고치세요." / "왜 이렇게 짰어요?" / "이건 안 됩니다."

❌ 비꼬는 톤
"아, 이렇게도 짤 수 있군요..." / "재미있는 접근이네요."

❌ 극단적 표현
"이건 최악이에요." / "이 코드는 절대 안 됩니다."

❌ 수동공격적
"물론 이렇게 해도 되긴 하죠..." / "본인이 좋다면야..."
```

### 권장하는 톤

```
✅ 협업적
"~하면 어떨까요?" / "~하는 방법도 있어요."

✅ 겸손한
"제가 잘못 이해한 걸 수 있는데, ~" / "확실하지 않지만 ~일 수 있어요."

✅ 이유 중심
"~때문에 ~가 더 안전할 것 같아요."

✅ 인정하는
"이 부분 깔끔하게 잘 처리하셨네요." / "이 접근 좋아요, 한 가지만 더 보면 ~"
```

### 상황별 코멘트 템플릿

```
[새로운 팀원의 코드]
"[suggestion] 우리 프로젝트에서는 에러 처리를 이렇게 하는 컨벤션이 있어요.
(링크) 참고해주시면 좋겠어요! 다른 부분은 잘 작성하셨습니다."

[반복되는 같은 실수]
"[issue] 이전 리뷰에서도 언급했던 부분인데, ~
팀 위키에 가이드를 정리해두면 좋을 것 같아요. 제가 초안 작성할까요?"

[의견이 다를 때]
"[thought] 저는 ~하는 방식을 선호하는데, 이유는 ~입니다.
다만 현재 방식도 동작에 문제는 없으니, 팀에서 논의해보면 좋겠어요."

[긴급 핫픽스]
"[issue] 급한 건 알지만, ~는 다음 PR에서 반드시 수정해야 해요.
TODO 코멘트와 이슈 생성 부탁드려요."
```

## 4. 리뷰 프로세스 커뮤니케이션

### Approve 기준

```
Approve — 수정 사항 없음 또는 nitpick만 있음
Comment — 질문이나 논의 필요, 블로커는 아님
Request Changes — blocker 또는 issue가 있어 수정 필수
```

### 재리뷰 시

```
✅ 수정된 부분만 재확인하고 빠르게 응답
✅ "확인했어요, LGTM!" 으로 짧게 마무리
❌ 이전에 없던 새로운 nitpick을 추가로 요구 (scope creep)
```

### 코멘트 해결

```
작성자가 코멘트에 응답할 때:
✅ 수정했으면: "수정했습니다." + 커밋 링크
✅ 반박하면: 이유를 설명하고 리뷰어와 논의
✅ 나중에 하겠다면: TODO + 이슈 번호 첨부

리뷰어가 Resolve할 때:
✅ 작성자의 응답을 확인한 후에만 Resolve
❌ 작성자의 응답 없이 일방적으로 Resolve
```

## 리뷰어 종합 체크리스트

| 항목 | 확인 내용 | 심각도 |
|------|----------|--------|
| 라벨 없음 | 코멘트에 심각도 라벨 없음 | - |
| 대안 미제시 | 문제만 지적하고 해결 방향 없음 | - |
| 공격적 톤 | 명령형, 비꼬는 표현 사용 | - |
| 칭찬 부재 | 좋은 코드에 대한 긍정 피드백 없음 | - |
| scope creep | 재리뷰 시 새로운 요구 추가 | - |
