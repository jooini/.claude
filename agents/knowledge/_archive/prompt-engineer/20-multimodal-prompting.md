# 멀티모달 프롬프팅

> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/vision, https://docs.anthropic.com/en/docs/build-with-claude/pdf-support

---

## 멀티모달 프롬프팅의 개념

멀티모달 프롬프팅은 텍스트뿐만 아니라 이미지, PDF, 차트, 스크린샷 등 다양한 형태의 입력을 조합하여 모델에게 지시하는 기법이다. 시각 자료를 활용하면 텍스트만으로 설명하기 어려운 맥락을 효과적으로 전달할 수 있다.

## 이미지 입력 프롬프팅

### 이미지 활용 시나리오

```markdown
# 이미지를 사용하면 효과적인 경우
1. UI/UX 디자인 → 코드 변환
2. 스크린샷 기반 버그 리포트
3. 아키텍처 다이어그램 분석
4. ERD/데이터 모델 해석
5. 에러 스크린샷 디버깅
6. 와이어프레임 → 컴포넌트 설계
7. 차트/그래프 데이터 추출
```

### 이미지 + 텍스트 조합 패턴

```markdown
# 패턴 1: 이미지 설명 요청
[이미지 첨부]
"이 스크린샷에서 발생한 에러의 원인을 분석해줘"

# 패턴 2: 이미지 → 코드 변환
[디자인 이미지 첨부]
"이 디자인을 React + Tailwind CSS 컴포넌트로 구현해줘.
레이아웃, 색상, 타이포그래피를 최대한 재현해."

# 패턴 3: 이미지 비교
[이미지 A 첨부] [이미지 B 첨부]
"두 디자인의 차이점을 분석하고,
A에서 B로 변경하기 위한 CSS 수정 사항을 알려줘."

# 패턴 4: 이미지 내 텍스트 추출
[스크린샷 첨부]
"이 에러 메시지의 텍스트를 추출하고 해결 방법을 제안해줘."
```

### 이미지 프롬프팅 효과적인 방법

```markdown
## 이미지 배치 순서
이미지를 먼저, 질문을 나중에 배치하면 더 효과적이다.

# 좋은 순서
[이미지]
"이 UI에서 접근성 문제를 찾아줘"

# 덜 효과적
"접근성 문제를 찾아줘"
[이미지]
→ 모델이 이미지를 받기 전에 답변 방향을 결정할 수 있음

## 구체적 지시와 함께
이미지만 주면 모델이 무엇에 집중할지 모른다.

# 모호한 지시
[이미지]
"이것에 대해 피드백해줘"

# 구체적 지시
[이미지]
"이 대시보드 UI에서:
1. 색상 대비가 WCAG AA 기준을 충족하는지
2. 모바일 반응형에서 레이아웃이 깨질 부분
3. 데이터 시각화의 가독성
을 검토해줘"
```

### 이미지 해상도와 품질

```markdown
## 이미지 입력 가이드라인
- 최소 해상도: 텍스트가 읽을 수 있는 수준 (100px 이하 텍스트는 인식 어려움)
- 최대 크기: 모델별 제한 확인 (Claude: ~5MB)
- 형식: PNG, JPEG, GIF, WebP 지원
- 텍스트가 포함된 이미지: 고해상도 권장
- 다이어그램/차트: 레이블이 선명해야 함

## 이미지 토큰 비용
이미지는 크기에 비례하여 토큰을 소비한다.
불필요하게 큰 이미지는 crop하여 관련 부분만 제공하면
토큰을 절약할 수 있다.
```

## PDF 입력 프롬프팅

### PDF 활용 시나리오

```markdown
# PDF 입력이 효과적인 경우
1. API 문서 분석 → 클라이언트 코드 생성
2. 기술 스펙 문서 → 구현 코드 작성
3. 계약서/약관 → 핵심 조건 추출
4. 논문 → 알고리즘 구현
5. ERD 다이어그램 (PDF 내 포함) → 스키마 코드 생성
6. 슬라이드 자료 → 핵심 요약
```

### PDF 프롬프팅 패턴

```markdown
# 패턴 1: 특정 섹션 참조
[PDF 첨부]
"이 API 문서의 Authentication 섹션을 기반으로
NestJS의 JWT 인증 Guard를 구현해줘"

# 패턴 2: 정보 추출
[PDF 첨부]
"이 기술 스펙 문서에서 다음을 추출해줘:
1. 필수 API 엔드포인트 목록
2. 각 엔드포인트의 요청/응답 스키마
3. 인증 방식
JSON 형식으로 출력해줘"

# 패턴 3: 문서 기반 구현
[PDF 첨부]
"이 ERD를 기반으로 TypeORM Entity 클래스를 작성해줘.
모든 관계(1:N, N:M)를 데코레이터로 정의해."
```

### PDF 처리 시 주의사항

```markdown
## PDF 처리 가이드라인
- 큰 PDF는 페이지 범위 지정 (pages: "1-5")
- 최대 20페이지/요청 제한
- 스캔된 PDF(이미지 PDF)는 OCR 품질에 따라 인식률 차이
- 표 데이터가 많은 PDF는 표 구조 유지 확인 필요
- 암호화된 PDF는 처리 불가

## 효과적인 지시
페이지가 많은 PDF를 다룰 때:
"이 문서의 3장(p.15-25)에 있는 데이터 모델을 기반으로..."
→ 구체적 범위를 지정하면 정확도 향상
```

## 시각 자료 참조 패턴

### 스크린샷 기반 디버깅

```markdown
# 에러 스크린샷 디버깅 프롬프트
[에러 스크린샷]
"이 에러 스크린샷을 분석해줘:
1. 에러 메시지 전문 추출
2. 스택 트레이스에서 원인 파일/줄 식별
3. 가능한 원인 3가지
4. 가장 유력한 원인의 해결 코드"
```

### 디자인 → 코드 변환

```markdown
# UI 디자인 구현 프롬프트
[Figma 스크린샷 또는 디자인 이미지]
"이 디자인을 구현해줘.
- 프레임워크: React + Next.js
- 스타일: Tailwind CSS
- 반응형: 모바일 우선
- 접근성: aria 속성 포함
- 픽셀 단위가 아닌 상대 단위(rem) 사용

레이아웃, 색상, 간격, 타이포그래피를 최대한 재현해.
이미지 자리는 placeholder로 처리해."
```

### 다이어그램 해석

```markdown
# 아키텍처 다이어그램 분석 프롬프트
[아키텍처 다이어그램 이미지]
"이 시스템 아키텍처 다이어그램을 분석해줘:
1. 각 컴포넌트의 역할
2. 컴포넌트 간 통신 방식 (동기/비동기)
3. 데이터 흐름
4. 잠재적 병목 지점
5. 이 아키텍처를 NestJS + MariaDB로 구현할 때의 모듈 구조 제안"
```

## 멀티모달 프롬프트 조합 전략

### 복합 입력 순서

```markdown
# 여러 형태의 입력을 조합할 때

## 권장 순서
1. 컨텍스트 설정 (텍스트): 배경 설명, 기술 스택
2. 시각 자료 (이미지/PDF): 참조할 디자인, 문서
3. 구체적 지시 (텍스트): 수행할 작업, 제약 조건
4. 출력 형식 (텍스트): 기대하는 응답 형태

## 예시
"현재 프로젝트는 Next.js + Tailwind CSS를 사용한다."  ← 컨텍스트
[디자인 이미지]                                         ← 시각 자료
"이 디자인의 카드 컴포넌트를 구현해줘.                  ← 지시
호버 효과와 다크모드를 포함해.
TypeScript 컴포넌트로 작성해줘."                        ← 출력 형식
```

### 여러 이미지 비교

```markdown
# 다중 이미지 입력 패턴

## before/after 비교
[수정 전 스크린샷]
[수정 후 스크린샷]
"두 스크린샷의 UI 차이점을 목록으로 정리하고,
변경된 부분의 CSS 코드를 작성해줘"

## 여러 화면 구현
[홈페이지 디자인]
[상세 페이지 디자인]
[마이페이지 디자인]
"이 3개 화면에서 공통으로 사용되는 컴포넌트를 식별하고,
재사용 가능한 컴포넌트로 분리해서 구현해줘"
```

## 모델별 멀티모달 지원

```markdown
# 모델별 멀티모달 지원 현황

| 기능 | Claude | GPT-4o | Gemini |
|------|--------|--------|--------|
| 이미지 입력 | O | O | O |
| PDF 입력 | O (네이티브) | O (파싱 필요) | O |
| 비디오 입력 | X | X | O |
| 오디오 입력 | X | O | O |
| 이미지 생성 | X | O (DALL-E) | O (Imagen) |
| 최대 이미지 수 | ~20/요청 | ~20/요청 | ~16/요청 |

# 모델 선택 기준
- 이미지+텍스트 분석: Claude, GPT-4o 모두 우수
- PDF 분석: Claude (네이티브 PDF 지원)
- 비디오 분석: Gemini (유일하게 지원)
- 이미지 생성이 필요: GPT (DALL-E 통합)
```

## 멀티모달 프롬프팅 주의사항

```markdown
## 일반 주의사항
1. 이미지에만 의존하지 말 것 — 핵심 정보는 텍스트로도 제공
2. 이미지 내 텍스트가 작으면 별도로 타이핑하여 제공
3. 복잡한 다이어그램은 부분별로 나눠서 질문
4. 이미지 토큰 비용을 고려하여 필요한 부분만 crop
5. 개인정보가 포함된 스크린샷은 마스킹 후 전달

## 안전 관련
- 민감 정보가 포함된 스크린샷 주의 (이메일, 이름, 주소 등)
- 내부 시스템 스크린샷의 보안 분류 확인
- 저작권이 있는 이미지의 사용 범위 확인
```

## 멀티모달 프롬프팅 체크리스트

```markdown
□ 이미지/PDF의 품질이 모델 인식에 충분한가?
□ 시각 자료와 텍스트 지시의 순서가 적절한가?
□ 구체적으로 무엇을 분석/구현할지 명시했는가?
□ 이미지에 민감 정보가 포함되어 있지 않은가?
□ 불필요하게 큰 이미지를 crop했는가?
□ PDF의 경우 페이지 범위를 지정했는가?
□ 출력 형식을 명시했는가?
□ 모델의 멀티모달 지원 범위를 확인했는가?
```
