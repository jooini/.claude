# 프롬프트 최적화

> 참조 링크: https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview, https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching

---

## 최적화의 두 축

프롬프트 최적화는 두 가지 축에서 이루어진다: **성능(품질)**과 **비용(토큰)**. 대부분의 경우 이 둘은 트레이드오프 관계에 있다.

```markdown
성능 ←────────→ 비용
상세한 지시        짧은 지시
많은 예시          예시 없음
풍부한 컨텍스트     최소 컨텍스트
```

## 토큰 절약 기법

### 1. 지시 압축

동일한 의미를 더 적은 토큰으로 전달한다.

```markdown
# 압축 전 (47 토큰)
당신은 시니어 백엔드 개발자입니다.
NestJS와 TypeORM을 주로 사용하며,
MariaDB를 데이터베이스로 사용합니다.
코드를 작성할 때는 클린 아키텍처 원칙을 따르고,
SOLID 원칙을 준수합니다.

# 압축 후 (28 토큰)
시니어 BE 개발자. 스택: NestJS + TypeORM + MariaDB.
클린 아키텍처, SOLID 준수.
```

### 압축 시 주의점

```markdown
# 너무 압축하면 의미 손실
"BE dev. NestJS." → 역할이 모호, TypeORM/MariaDB 정보 유실

# 적정 압축 기준
- 의미가 변하지 않는 범위에서 압축
- 기술 용어는 약어 사용 가능 (BE, FE, DB, API)
- 조사/접속사 제거 가능
- 구체적 기술명은 생략하지 않음
```

### 2. 중복 제거

```markdown
# 중복 있는 프롬프트
코드를 작성할 때 에러 핸들링을 포함하세요.
모든 API 엔드포인트에 에러 핸들링을 추가하세요.
서비스 레이어에서도 에러 핸들링을 잊지 마세요.

# 통합 후
모든 레이어(Controller, Service)에 에러 핸들링을 포함하세요.
```

### 3. 구조화로 토큰 절약

표나 리스트는 산문보다 토큰 효율이 높다.

```markdown
# 산문 형태 (토큰 많음)
사용자가 코드 리뷰를 요청하면 보안, 성능, 가독성 순서로 검토합니다.
버그 수정을 요청하면 에러 재현, 원인 분석, 수정 순서로 진행합니다.
새 기능을 요청하면 인터페이스 정의, 구현, 테스트 순서로 진행합니다.

# 구조화 형태 (토큰 적음)
작업별 순서:
- 코드 리뷰: 보안 → 성능 → 가독성
- 버그 수정: 재현 → 원인 → 수정
- 새 기능: 인터페이스 → 구현 → 테스트
```

### 4. Few-shot 예시 최적화

```markdown
# 비효율: 긴 예시 3개 (300+ 토큰)
예시 1: [15줄 입력] → [15줄 출력]
예시 2: [15줄 입력] → [15줄 출력]
예시 3: [15줄 입력] → [15줄 출력]

# 효율: 짧은 예시 1개 + 패턴 설명 (80 토큰)
출력 패턴: `[심각도] 파일:줄 — 설명 (수정 제안)`
예시: `[Major] user.service.ts:42 — N+1 쿼리 (findOne 대신 relation 로드)`
```

## 성능 vs 비용 트레이드오프

### 비용 최적화가 우선인 경우

```markdown
# 대량 처리 (배치)
- 수천 건의 텍스트 분류, 데이터 추출
- 개별 품질보다 처리량이 중요
- 최적화: 시스템 프롬프트 최소화, 예시 1개, 짧은 출력 지시

# 실시간 응답
- 채팅봇, 자동 완성
- 응답 속도(TTFT)가 중요
- 최적화: 프롬프트 길이 최소화, prefill 활용
```

### 성능 최적화가 우선인 경우

```markdown
# 고품질 요구
- 코드 생성, 분석 리포트, 의사결정 지원
- 정확성과 완전성이 중요
- 최적화: 상세한 지시, 충분한 예시, 체인 오브 싱킹

# 안전 크리티컬
- 의료/법률/금융 관련 출력
- 오류 비용이 매우 큼
- 최적화: 다중 검증, 제약 조건 강화, 불확실성 표현 의무화
```

## 시스템 프롬프트 캐싱

### Prompt Caching 활용

동일한 시스템 프롬프트를 반복 사용할 때 비용을 크게 절감할 수 있다.

```markdown
# 캐싱 효과적인 구조
시스템 프롬프트 (고정, 캐싱됨)
├── 역할 정의
├── 행동 규칙
├── 출력 형식
└── 제약 조건

사용자 메시지 (동적, 매번 다름)
└── 구체적 작업 요청
```

### 캐싱 최적화 전략

```markdown
# 1. 고정 부분을 앞에 배치
시스템 프롬프트의 앞부분이 동일할수록 캐시 히트율 높음

# 2. 동적 부분은 뒤에 배치
변경되는 내용(날짜, 세션 정보 등)은 프롬프트 후반에

# 3. 캐시 경계 활용
cache_control breakpoint를 적절히 배치
고정 구간 뒤에 breakpoint → 해당 지점까지 캐싱
```

## 출력 토큰 최적화

### 출력 길이 제어

```markdown
# 토큰 낭비 패턴
모델이 불필요한 서두, 요약, 반복을 생성

# 제어 방법
1. max_tokens 파라미터로 하드 리밋
2. 프롬프트에 길이 지시: "3줄 이내로 답변"
3. 불필요한 요소 명시적 금지: "서두/요약/인사 없이 본문만"
```

### 출력 형식 최적화

```markdown
# 비효율 출력
"이 코드에서 발견된 문제점을 설명하겠습니다.
첫 번째로, 42번 줄에서 N+1 쿼리 문제가 있습니다.
이는 루프 안에서 findOne을 호출하기 때문입니다.
해결 방법은 relation 로드를 사용하는 것입니다."

# 효율 출력
[Major] L42: N+1 쿼리 — 루프 내 findOne → relation 로드로 대체
```

## 모델 선택 최적화

### 작업별 모델 선택

```markdown
# 고급 모델 (Opus) 사용
- 복잡한 아키텍처 설계
- 다파일 리팩토링
- 새로운 패턴의 코드 생성

# 중급 모델 (Sonnet) 사용
- 코드 리뷰
- 버그 수정
- 테스트 작성
- 문서 생성

# 경량 모델 (Haiku) 사용
- 텍스트 분류
- 데이터 추출
- 형식 변환
- 간단한 Q&A
```

## 점진적 최적화 프로세스

### 1단계: 베이스라인 측정

```markdown
# 측정 항목
- 입력 토큰 수
- 출력 토큰 수
- 응답 품질 점수 (테스트 케이스 통과율)
- 응답 시간 (TTFT, 전체)
- 비용 ($/요청)
```

### 2단계: 병목 식별

```markdown
# 토큰이 많이 소비되는 부분 찾기
- 시스템 프롬프트: 규칙이 너무 많은가?
- 예시: 예시가 과도한가?
- 컨텍스트: 불필요한 정보가 포함되어 있는가?
- 출력: 모델이 불필요한 내용을 생성하는가?
```

### 3단계: 최적화 적용

```markdown
# 우선순위
1. 불필요한 컨텍스트 제거 (리스크 낮음, 효과 높음)
2. 지시 압축 (리스크 낮음, 효과 중간)
3. 예시 수 조정 (리스크 중간, 효과 중간)
4. 모델 다운그레이드 (리스크 높음, 효과 높음)
```

### 4단계: 회귀 검증

```markdown
# 최적화 후 반드시 확인
- 기존 테스트 케이스 통과율 유지
- 핵심 기능의 응답 품질 저하 없음
- 안전 관련 제약 조건 여전히 동작
```

## 실전 최적화 체크리스트

```markdown
□ 시스템 프롬프트에 중복 규칙이 없는가?
□ 예시가 3개 이상이면 줄일 수 있는가?
□ 산문을 리스트/표로 바꿀 수 있는가?
□ 사용하지 않는 규칙이 포함되어 있지 않은가?
□ 출력 형식이 토큰 효율적인가?
□ 모델 수준이 작업 난이도에 적합한가?
□ 프롬프트 캐싱을 활용하고 있는가?
□ 불필요한 서두/요약 생성을 금지했는가?
```
