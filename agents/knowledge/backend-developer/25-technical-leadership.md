## 1. 기술 리더십이란

직책이 아닌 영향력. 코드 품질, 아키텍처, 팀 성장에 책임.

**시니어 → 스태프 엔지니어 전환:**
- 시니어: 주어진 문제를 잘 해결
- 스태프: 올바른 문제를 정의하고 팀이 해결하도록 이끔

## 2. 기술 의사결정

### ADR (Architecture Decision Record)

중요한 기술 결정을 문서화.

```markdown
# ADR-001: 이벤트 기반 서비스 통신 채택

## 상태
결정됨 (2024-01-15)

## 컨텍스트
현재 서비스 간 동기 HTTP 호출로 결합도 높음.
A 서비스 장애가 B, C까지 전파됨.

## 결정
서비스 간 통신을 BullMQ 이벤트 기반으로 전환.
동기 응답 필요한 경우만 HTTP 유지.

## 결과
- 긍정: 서비스 독립성, 장애 격리
- 부정: 최종 일관성, 디버깅 복잡도 증가
- 위험: 이벤트 유실 → Outbox 패턴으로 대응

## 대안 고려
- gRPC: 타입 안전하지만 학습 비용, 브라우저 지원 복잡
- Kafka: 대용량에 적합하지만 운영 복잡도 높음
```

### RFC (Request for Comments)

큰 변경 전 팀 의견 수렴.

```markdown
# RFC: TypeORM → Drizzle ORM 마이그레이션

## 동기
TypeORM strict 모드 타입 추론 한계, 번들 크기 이슈

## 제안
신규 모듈부터 Drizzle 적용, 기존 모듈 점진적 마이그레이션

## 마이그레이션 계획
1단계: Drizzle 세팅, 신규 모듈 적용 (2주)
2단계: 비핵심 모듈 마이그레이션 (4주)
3단계: 핵심 모듈 마이그레이션 (4주)

## 위험
- 두 ORM 공존 기간 복잡도
- 팀 학습 시간

## 피드백 요청
2024-02-01까지 코멘트
```

## 3. 코드 리뷰 리더십

```
리뷰 문화 정착:
- PR 크기 기준 설정 (400줄 이하 권장)
- 리뷰 응답 시간 기준 (영업일 1일 이내)
- Nitpick과 Blocking 구분
- 칭찬도 남기기

리뷰 코멘트 품질:
❌ "이거 왜 이렇게 했어요?"
✅ "이 부분에서 N+1이 발생할 수 있을 것 같아요.
   relations: ['items']로 한 번에 로딩하면 어떨까요?
   참고: [링크]"

아키텍처 리뷰:
- 설계 단계에서 리뷰 (코드 완성 후보다 비용 낮음)
- 영향 범위 큰 변경은 RFC 선행
```

## 4. 기술 부채 관리

```ts
// 기술 부채 추적
// TODO는 이슈 번호 포함
// TODO(#123): 다음 스프린트에 인덱스 추가
// FIXME(#456): 임시 방편, 제대로 된 해결 필요

// 기술 부채 가시화
// 스프린트마다 부채 상환 시간 확보 (20%)
// 리팩토링을 기능 작업과 분리
```

**기술 부채 분류:**
| 종류 | 설명 | 우선순위 |
|------|------|---------|
| 고의적/신중 | 기한 맞추기 위해 의도적으로 | 계획 필요 |
| 고의적/부주의 | "규칙 따위" | 높음 |
| 비고의적/신중 | 설계 완료 후 더 나은 방법 발견 | 중간 |
| 비고의적/부주의 | 무지 | 낮음 (교육 필요) |

## 5. 팀 성장 지원

```
1:1 미팅 (격주):
  - 업무 진행 상황보다 성장/어려움 집중
  - "요즘 가장 어려운 부분이 뭔가요?"
  - "어떤 기술을 더 깊이 배우고 싶으세요?"
  - Career ladder 기반 구체적 피드백

코드 리뷰를 통한 멘토링:
  - "왜"를 설명하는 코멘트
  - 관련 자료 링크
  - 페어 프로그래밍 세션

기술 공유:
  - 팀 내 기술 공유 세션 (격주)
  - Slack 기술 채널 활성화
  - 외부 컨퍼런스/스터디 지원
```

## 6. 온콜 & 인시던트

```markdown
# 인시던트 대응 절차

1. 탐지 및 초기 대응 (5분)
   - 알림 확인, 심각도 판단
   - 인시던트 채널 개설 (#incident-YYYYMMDD)

2. 격리 및 완화 (15분)
   - 영향 범위 파악
   - 즉시 조치 (롤백, 트래픽 차단 등)

3. 근본 원인 분석
   - 로그, 메트릭, 트레이스 분석
   - 타임라인 재구성

4. 해결 및 복구
   - 영구 수정 또는 안정적인 임시 수정

5. 사후 분석 (Post-mortem)
   - 24~48시간 내 작성
   - 비난 없는 분석 (Blameless)
   - 재발 방지 Action Items
```

## 7. 기술 전략

```
현재 상태 파악 → 목표 상태 정의 → 갭 분석 → 로드맵

예시: DB 마이그레이션
현재: 단일 RDS, 쿼리 느려짐
목표: Read Replica + 캐싱으로 조회 성능 5배 향상
갭: Read Replica 설정, 코드 분리, 캐싱 레이어
로드맵: Q1 Read Replica / Q2 Redis 캐싱 / Q3 쿼리 최적화
```

## 8. 안티패턴

- **영웅 문화**: 한 사람이 모든 것을 앎 → 지식 공유, 문서화
- **아키텍처 결정 독단**: RFC로 팀 의견 수렴
- **기술 부채 무시**: 적당히 쌓이면 개발 속도 급락
- **결과 없는 1:1**: 액션 아이템 없는 미팅 → 구체적 다음 단계
- **완벽주의적 리뷰**: 모든 PR에 20개 코멘트 → 팀 모럴 저하
