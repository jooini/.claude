# PR 표준

> 참조 링크: https://google.github.io/eng-practices/review/developer/, https://docs.github.com/en/pull-requests

---

## 개요

좋은 PR은 리뷰어의 인지 부하를 줄이고, 효율적인 리뷰를 가능하게 한다. PR 크기, 커밋 구조, 설명 작성, 셀프 리뷰 체크리스트를 기준으로 검토한다.

## 1. PR 크기

### 적정 크기 가이드

```
Small (1-50줄)
→ 리뷰 시간: 10분
→ 리뷰 품질: 최고
→ 머지 속도: 당일

Medium (51-200줄)
→ 리뷰 시간: 30분
→ 리뷰 품질: 높음
→ 머지 속도: 1일 이내

Large (201-500줄)
→ 리뷰 시간: 1시간+
→ 리뷰 품질: 보통 (피로도 증가)
→ 머지 속도: 1-2일

XL (500줄+)
→ 리뷰 시간: 예측 불가
→ 리뷰 품질: 낮음 (LGTM 유발)
→ 권장: PR 분할
```

### PR 분할 전략

```
❌ 하나의 PR에 모든 것
PR #1: "사용자 관리 기능 구현"
→ 엔티티 + 마이그레이션 + 서비스 + 컨트롤러 + 테스트 + 프론트 = 1500줄

✅ 기능 단위로 분할
PR #1: "User 엔티티 및 마이그레이션 추가" (50줄)
PR #2: "UserService CRUD 구현" (120줄)
PR #3: "UserController API 엔드포인트 추가" (80줄)
PR #4: "User API 테스트 추가" (150줄)
PR #5: "사용자 관리 UI 구현" (200줄)

분할 기준:
1. 레이어별: DB → 서비스 → API → UI
2. 기능별: 목록 조회 → 생성 → 수정 → 삭제
3. 리팩토링 분리: 기능 변경과 리팩토링을 같은 PR에 섞지 않음
```

### PR 크기 체크리스트

- [ ] 변경이 400줄 이하인가?
- [ ] 하나의 PR에 하나의 목적만 있는가?
- [ ] 리팩토링과 기능 변경이 분리되어 있는가?
- [ ] 테스트 코드를 제외한 순수 변경이 과도하지 않은가?

## 2. 커밋 구조

### Conventional Commits

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### 타입 분류

```
feat:     새 기능
fix:      버그 수정
docs:     문서 변경
style:    코드 포맷 (기능 변경 없음)
refactor: 리팩토링 (기능 변경 없음, 버그 수정 없음)
perf:     성능 개선
test:     테스트 추가/수정
chore:    빌드, 설정, 의존성 변경
ci:       CI 설정 변경
```

### 좋은 커밋 vs 나쁜 커밋

```
❌ 나쁜 커밋 메시지
- "수정"
- "WIP"
- "fix bug"
- "asdf"
- "커밋"
- "Update index.ts"

✅ 좋은 커밋 메시지
- "feat(user): 이메일 인증 기능 추가"
- "fix(order): 동시 주문 시 재고 차감 레이스 컨디션 수정"
- "refactor(auth): JWT 검증 로직을 Guard로 분리"
- "perf(query): 주문 목록 조회 N+1 쿼리 제거"
- "test(payment): 환불 시나리오 통합 테스트 추가"
```

### 커밋 히스토리 정리

```
❌ 지저분한 커밋 히스토리
abc1234 feat: user api
def5678 fix: typo
ghi9012 fix: 다시 수정
jkl3456 wip
mno7890 제대로 수정

✅ 깔끔한 커밋 히스토리
abc1234 feat(user): User 엔티티 및 마이그레이션 추가
def5678 feat(user): UserService CRUD 구현
ghi9012 feat(user): UserController REST API 추가
jkl3456 test(user): UserService 단위 테스트 추가

정리 방법:
- 작업 중: 자유롭게 커밋
- PR 전: git rebase -i로 squash/fixup (WIP, typo fix 등 정리)
- 각 커밋이 독립적으로 빌드/테스트 가능한 상태 유지
```

### 커밋 체크리스트

- [ ] Conventional Commits 형식을 따르는가?
- [ ] 각 커밋이 하나의 논리적 변경을 나타내는가?
- [ ] WIP, fixup 커밋이 squash 되었는가?
- [ ] 커밋 메시지가 "무엇을" "왜" 변경했는지 설명하는가?

## 3. PR 설명 작성

### PR 템플릿

```markdown
## 요약
<!-- 이 PR이 해결하는 문제와 접근 방식 -->

## 변경 사항
<!-- 주요 변경 내용 (bullet point) -->
- 
- 
- 

## 관련 이슈
<!-- Closes #123, Related to #456 -->

## 테스트
<!-- 테스트 방법 / 결과 -->
- [ ] 단위 테스트 추가/수정
- [ ] 통합 테스트 추가/수정
- [ ] 수동 테스트 완료

## 스크린샷
<!-- UI 변경이 있는 경우 -->

## 체크리스트
- [ ] 셀프 리뷰 완료
- [ ] 테스트 통과
- [ ] 문서 업데이트 (필요 시)
- [ ] Breaking change 없음 (또는 명시)
```

### 좋은 PR 설명 예시

```markdown
## 요약
주문 생성 시 동시 요청으로 재고가 마이너스로 떨어지는 문제를 수정합니다.
원인: check-then-act 패턴에서 레이스 컨디션 발생
해결: 원자적 UPDATE 쿼리로 변경

## 변경 사항
- OrderService.createOrder()에서 재고 차감을 원자적 쿼리로 변경
- 재고 부족 시 명확한 에러 코드(INSUFFICIENT_STOCK) 반환
- 동시 주문 시나리오 통합 테스트 추가

## 관련 이슈
Closes #234

## 테스트
- [x] 동시 주문 10건 테스트: 재고 정합성 확인
- [x] 재고 부족 시 에러 응답 확인
- [x] 기존 정상 주문 플로우 회귀 테스트 통과
```

### PR 설명 체크리스트

- [ ] "왜" 이 변경이 필요한지 설명되어 있는가?
- [ ] 주요 설계 결정의 이유가 있는가?
- [ ] 관련 이슈가 링크되어 있는가?
- [ ] Breaking change가 명시되어 있는가?
- [ ] UI 변경 시 스크린샷이 포함되어 있는가?
- [ ] 테스트 방법이 기술되어 있는가?

## 4. 셀프 리뷰 체크리스트

### PR 제출 전 셀프 리뷰

```
코드 품질:
- [ ] 불필요한 console.log / debugger 제거
- [ ] 하드코딩된 값 없음 (환경 변수 또는 상수로)
- [ ] TODO 코멘트에 이슈 번호 연결
- [ ] import 정리 (미사용 import 제거)
- [ ] 타입 any 사용하지 않음

기능:
- [ ] 엣지 케이스 처리 (null, undefined, empty, 0)
- [ ] 에러 처리 적절
- [ ] 입력 유효성 검증

보안:
- [ ] 민감 정보 커밋 안 됨 (.env, 시크릿)
- [ ] SQL 인젝션 방지 (파라미터 바인딩)
- [ ] 인증/인가 체크

성능:
- [ ] N+1 쿼리 없음
- [ ] 불필요한 DB 호출 없음
- [ ] 메모리 누수 가능성 없음

테스트:
- [ ] 새 기능에 테스트 추가
- [ ] 기존 테스트 통과
- [ ] 엣지 케이스 테스트 포함

문서:
- [ ] 코드 변경에 맞게 주석 업데이트
- [ ] API 변경 시 문서 업데이트
- [ ] README 업데이트 (필요 시)
```

### Diff 리뷰 습관

```
PR 제출 전 반드시:
1. git diff --staged 로 변경 내용 전체 읽기
2. GitHub에서 "Files Changed" 탭으로 다시 확인
3. 리뷰어 입장에서 "이 코드를 처음 보는 사람이 이해할 수 있는가?" 자문
4. 불필요한 변경(공백, import 순서 등)이 섞여 있으면 분리
```

## 5. Draft PR 활용

### Draft PR 사용 시점

```
✅ Draft PR이 유용한 경우
- 설계 방향에 대한 초기 피드백 요청
- 구현 진행 중 중간 확인
- 큰 기능의 점진적 리뷰
- CI 파이프라인 사전 확인

사용 방법:
1. 구현 초기에 Draft PR 생성
2. 설명에 "피드백 요청" 포인트 명시
3. 구현 완료 후 "Ready for Review"로 전환
```

### WIP 표시 컨벤션

```
Draft PR 제목: "[WIP] feat(order): 주문 생성 API"
→ Ready 시: "feat(order): 주문 생성 API"

PR 설명에 진행 상황 표시:
- [x] 엔티티 정의
- [x] 서비스 로직
- [ ] 컨트롤러
- [ ] 테스트
- [ ] 문서
```

## 리뷰어 종합 체크리스트

| 항목 | 확인 내용 | 심각도 |
|------|----------|--------|
| 민감 정보 커밋 | .env, 시크릿 등 포함 | P0 |
| PR 설명 부재 | 변경 이유/내용 설명 없음 | P1 |
| 과대 PR | 500줄+ 변경, 분할 필요 | P1 |
| 혼합 목적 | 리팩토링 + 기능 변경 혼합 | P2 |
| 커밋 메시지 | 불명확한 커밋 메시지 | P2 |
| 셀프 리뷰 미실시 | console.log, debugger 잔존 | P2 |
| 테스트 미작성 | 새 기능에 테스트 없음 | P1 |
| Breaking Change | 미명시된 하위 호환 파괴 | P0 |
