# 작업 복잡도 분석

> 목적: 작업을 S/M/L/XL로 분류하고, 그에 맞는 에이전트 수와 검증 깊이를 결정한다.

---

## 1. 복잡도 분류의 의미

복잡도는 코드량만으로 결정하지 않는다.

파일 수, 도메인 수, 영향 범위, 실패 비용, 불확실성, 테스트 난이도를 함께 본다.

**복잡도 분석 결과는 다음 결정을 좌우한다:**
- 사전 분석 깊이
- 필요한 에이전트 조합
- 병렬 실행 여부
- 리뷰 계층 수
- 테스트 범위
- 사용자 에스컬레이션 필요성

---

## 2. 빠른 분류표

| 등급 | 파일 수 | 도메인 수 | 영향 범위 | 권장 처리 |
|------|---------|-----------|-----------|-----------|
| S | 1~2개 | 1개 | 지역적 | 단일 구현 + 기본 검증 |
| M | 3~5개 | 1~2개 | 모듈 내부 | 개발 + 리뷰 + 테스트 |
| L | 6~15개 | 2~3개 | 서비스 흐름 | Phase 전체 + 병렬 검증 |
| XL | 16개 이상 | 3개 이상 | 시스템/조직 영향 | 스펙 분리 + 작업 분할 |

### 숫자는 보조 지표다

파일 1개라도 인증 우회 가능성이 있으면 L이 될 수 있다.

파일 20개라도 단순 문자열 치환이고 자동 테스트가 충분하면 M으로 낮출 수 있다.

---

## 3. S 작업 기준

S는 즉시 구현 가능한 작은 작업이다.

**대표 사례:**
- 버튼 라벨 변경
- 테스트 fixture의 명확한 오류 수정
- 문서 오탈자 수정
- 한 함수의 null 체크 추가
- 한 API 응답 필드의 직렬화 오류 수정

**S 작업 체크리스트:**
- [ ] 변경 파일이 1~2개인가?
- [ ] 외부 API 계약이 바뀌지 않는가?
- [ ] DB 스키마 변경이 없는가?
- [ ] 권한/인증/결제와 무관한가?
- [ ] 실패해도 영향이 제한적인가?
- [ ] 로컬 테스트로 충분히 검증 가능한가?

### S 작업 에이전트 구성

```typescript
Agent("backend-developer", {
  task: "UserDto의 nickname null 처리 수정",
  scope: "single file",
  verification: "targeted unit test",
});
```

---

## 4. M 작업 기준

M은 한 모듈 또는 인접 모듈에 걸친 일반적인 개발 작업이다.

**대표 사례:**
- API 엔드포인트 1개 추가
- 화면 컴포넌트와 상태 관리 변경
- 기존 쿼리 최적화
- 버그 수정 + 회귀 테스트 추가
- 서비스 내부 리팩터링

**M 작업 판단 기준:**
- 변경 파일 3~5개
- 도메인 1~2개
- API 또는 UI 동작에 작은 변화
- 테스트 추가 필요
- 리뷰 없이는 놓칠 수 있는 엣지 케이스 존재

### M 작업 실행 패턴

```python
developer = Agent("backend-developer", task="쿠폰 적용 API 구현")
reviewer = Agent("code-reviewer", task="쿠폰 중복 사용과 금액 계산 리뷰")
tester = Agent("code-tester", task="단위/통합 테스트 실행")

pipeline = [developer, reviewer, tester]
```

---

## 5. L 작업 기준

L은 여러 도메인 또는 사용자 흐름에 영향을 주는 작업이다.

**대표 사례:**
- 로그인 플로우 개편
- 주문 생성부터 결제까지 변경
- 백엔드 API와 프론트 화면 동시 변경
- 캐시 전략 변경
- 데이터 마이그레이션 포함 기능

**L 작업 신호:**
- 변경 파일 6~15개
- Backend와 Frontend가 모두 관련됨
- QA 시나리오가 여러 개 필요함
- 장애 발생 시 사용자 영향이 큼
- 기존 설계와 충돌 가능성이 있음

### L 작업 권장 구성

| 역할 | 에이전트 | 산출물 |
|------|----------|--------|
| 사전 분석 | explorer 또는 Gemini scan | 관련 파일, 위험 목록 |
| 설계 | dev-lead + domain agent | 계약, 단계별 계획 |
| 구현 | backend/frontend/data | 코드 변경 |
| 검증 | code-reviewer + qa | 리뷰, 테스트 |
| 최종 확인 | tester | 실행 결과 |

---

## 6. XL 작업 기준

XL은 한 번에 끝내면 위험한 작업이다.

**대표 사례:**
- 인증 시스템 교체
- 결제 모듈 재설계
- 대규모 프론트 상태 관리 전환
- DB 샤딩 또는 마이그레이션
- 멀티 서비스 API 계약 변경

**XL 작업 특징:**
- 변경 파일 16개 이상
- 도메인 3개 이상
- 배포 순서가 중요함
- 롤백 전략 필요
- 여러 PR 또는 여러 세션으로 나누어야 함

### XL 작업 처리 원칙

- 스펙 문서를 먼저 작성한다.
- 작업을 독립 가능한 단계로 분해한다.
- 각 단계의 완료 기준을 만든다.
- 위험한 변경은 feature flag 또는 dual-run을 검토한다.
- 사용자 판단이 필요한 지점을 명시한다.

---

## 7. 위험 가중치

복잡도는 다음 위험 요소로 상향 조정한다.

| 위험 요소 | 상향 기준 |
|-----------|-----------|
| 인증/권한 | 최소 M, 플로우 변경이면 L |
| 결제/정산 | 최소 L |
| 데이터 삭제 | 최소 L |
| DB 마이그레이션 | 최소 L |
| 배포/인프라 | 최소 M, 운영 영향이면 L/XL |
| 보안 취약점 | 최소 L |
| 장애 대응 | 재현 불가면 L |

---

## 8. 복잡도 산정 예시

```typescript
type Complexity = "S" | "M" | "L" | "XL";

function classifyTask(input: {
  files: number;
  domains: number;
  touchesAuth: boolean;
  touchesPayment: boolean;
  hasMigration: boolean;
}): Complexity {
  if (input.touchesPayment || input.hasMigration) return "L";
  if (input.files >= 16 || input.domains >= 3) return "XL";
  if (input.files >= 6 || input.domains === 2 || input.touchesAuth) return "L";
  if (input.files >= 3) return "M";
  return "S";
}
```

---

## 9. 복잡도별 필수 검증

| 등급 | 필수 검증 |
|------|-----------|
| S | 대상 테스트 또는 수동 확인 |
| M | 단위/통합 테스트 + 리뷰 |
| L | 다층 리뷰 + 회귀 테스트 + 통합 확인 |
| XL | 단계별 검증 + 롤백 계획 + 사용자 판단 |

---

## 10. 복잡도 판정 체크리스트

- [ ] 변경 파일 수를 확인했는가?
- [ ] 도메인 수를 확인했는가?
- [ ] 사용자 플로우 영향을 확인했는가?
- [ ] 데이터와 권한 리스크를 확인했는가?
- [ ] 배포 순서가 필요한지 확인했는가?
- [ ] 테스트 난이도를 확인했는가?
- [ ] 불확실성이 높으면 한 단계 상향했는가?
- [ ] XL이면 한 번에 구현하지 않도록 분해했는가?

---

## 11. 최종 기준

애매하면 복잡도를 낮추지 않는다.

분류가 보수적이면 시간이 조금 더 들지만, 분류가 과소하면 리뷰와 테스트가 부족해진다.
