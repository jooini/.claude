# 에이전트 실패 복구

> 실패를 숨기지 않고 분류한 뒤, 재시도·대안·에스컬레이션으로 수렴한다.

---

## 1. 실패 복구의 목적

멀티에이전트 시스템에서는 실패가 정상적으로 발생한다.

중요한 것은 실패를 빠르게 분류하고, 같은 방식의 반복 실패를 막는 것이다.

**복구 원칙:**
- 실패 원인을 먼저 분류한다.
- 같은 프롬프트를 무작정 반복하지 않는다.
- 재시도에는 변경된 조건이 있어야 한다.
- 2회 실패하면 접근을 바꾼다.
- 3회 루프를 넘으면 사용자 판단을 요청한다.

---

## 2. 실패 유형

| 유형 | 증상 | 대응 |
|------|------|------|
| 컨텍스트 부족 | 엉뚱한 파일 수정 | 핸드오프 보강 |
| 범위 초과 | unrelated refactor | 소유권 재명시 |
| 품질 부족 | 테스트 실패, 타입 오류 | 재수정 |
| 설계 충돌 | BE/FE 계약 불일치 | Phase 1 재진입 |
| 환경 실패 | 명령 실행 불가 | 환경 로그 확인 |
| 요구사항 충돌 | 어떤 정책이 맞는지 불명확 | 사용자 에스컬레이션 |

---

## 3. 실패 감지

dev-lead는 다음 신호를 놓치지 않는다.

- 에이전트가 요구 파일이 아닌 파일을 수정함
- 테스트 실패 원인을 설명하지 못함
- 리뷰 finding이 반복됨
- 계약이 에이전트마다 다름
- "아마", "추정"이 근거 없이 많음
- 결과가 작업 목표와 어긋남

---

## 4. 재시도 기준

재시도는 조건을 바꿀 때만 의미가 있다.

| 1차 실패 | 재시도 시 변경할 것 |
|----------|--------------------|
| 파일을 못 찾음 | 관련 파일 목록 제공 |
| 테스트 실패 | 실패 로그와 기대 동작 제공 |
| 계약 불일치 | 계약을 명시적으로 제공 |
| 범위 초과 | 허용/금지 파일 재명시 |
| 품질 낮음 | 구체적 finding 전달 |

```typescript
Agent("backend-developer", {
  task: "P1 리뷰 finding만 수정",
  findings: ["권한 체크 누락", "중복 요청 테스트 없음"],
  allowedFiles: ["server/order/order.service.ts", "server/order/order.service.spec.ts"],
  forbidden: ["web/**", "infra/**"],
});
```

## 5. 대안 에이전트 투입

같은 에이전트가 두 번 실패하면 다른 관점이 필요하다.

| 실패 영역 | 대안 |
|-----------|------|
| 구현 난항 | codex 대안 구현 |
| 원인 불명 | debug-master |
| 테스트 설계 부족 | qa |
| 설계 불일치 | code-reviewer |
| 성능 미개선 | data-analyst |
| 배포 실패 | ops-lead |

```python
Agent("codex", task="""
현재 접근이 두 번 실패했다.
같은 요구사항을 만족하는 더 작은 변경 경로를 제안하라.
코드 수정 전 설계와 patch 후보만 제시하라.
""")
```

## 6. Phase 재진입

실패 유형에 따라 이전 Phase로 돌아간다.

| 실패 | 돌아갈 Phase |
|------|--------------|
| 관련 파일 누락 | Phase 0 |
| 계약 불명확 | Phase 1 |
| 구현 충돌 | Phase 2 |
| 테스트 실패 | Phase 3 |
| 결과 범위 초과 | Phase 4 |

### 재진입 기록

```markdown
## 재진입 결정
- 실패: Frontend mock은 `undefined`, Backend는 `null`
- 원인: API 계약에 nullable 규칙 누락
- 조치: Phase 1로 돌아가 계약 수정 후 FE 재작업
```

## 7. 사용자 에스컬레이션

다음은 사용자가 결정해야 한다.

- 요구사항이 서로 충돌함
- 데이터 삭제 또는 migration 방식 선택
- 보안 정책 선택
- 운영 배포 중단 가능성
- 세 번째 재수정 후에도 합의 불가
- 테스트 환경 권한이 필요함

### 에스컬레이션 형식

```markdown
결정이 필요합니다.

문제:
- 기존 정책은 중복 쿠폰을 허용하지만 신규 요구사항은 금지합니다.

선택지:
- A: 기존 정책 유지, 신규 기능에서만 제한
- B: 전체 정책 변경, migration 필요

권장:
- A. 영향 범위가 작고 기존 사용자 동작을 깨지 않습니다.
```

## 8. 실패 후 품질 회복

실패 복구 후에는 검증을 강화한다.

| 실패 종류 | 추가 검증 |
|-----------|-----------|
| 권한 누락 | negative auth test |
| 계약 불일치 | contract test |
| 동시성 버그 | duplicate request test |
| UI 상태 오류 | e2e interaction |
| 성능 미개선 | before/after benchmark |

## 11. 최종 기준

실패 복구는 실패를 없애는 능력이 아니라, 실패를 증거로 삼아 더 정확한 경로로 수렴하는 능력이다.
