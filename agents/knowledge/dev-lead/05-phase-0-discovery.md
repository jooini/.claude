# Phase 0 사전 분석

> 구현 전에 코드베이스, 영향 범위, 위험을 확인한다.

---

## 1. Phase 0의 목적

Phase 0는 작업을 시작하기 전 지도 만들기다.

코드를 고치기 전에 다음을 확인한다.

- 관련 파일은 어디인가?
- 실제 흐름은 어떻게 연결되는가?
- 기존 컨벤션은 무엇인가?
- 변경하면 깨질 수 있는 테스트는 무엇인가?
- 비슷한 구현이 이미 있는가?
- 보안/데이터/운영 위험이 있는가?

---

## 2. 언제 Phase 0가 필요한가

| 상황 | Phase 0 필요도 |
|------|----------------|
| S 문구 수정 | 낮음 |
| M 기능 추가 | 중간 |
| L 풀스택 변경 | 높음 |
| XL 구조 변경 | 필수 |
| 장애 원인 불명 | 필수 |
| 인증/결제/데이터 삭제 | 필수 |

### 생략 가능한 경우

- 변경 위치가 명확하다.
- 파일 1개만 수정한다.
- 테스트가 충분히 존재한다.
- 실패 비용이 낮다.

---

## 3. 사전 분석 입력

Phase 0에 들어가기 전 dev-lead는 사용자 요청을 다음 구조로 변환한다.

```markdown
## 작업 의도
- 무엇을 바꾸려는가?

## 알려진 제약
- 수정 금지 영역
- 커밋 금지
- 테스트 생략 여부

## 예상 도메인
- Backend/Frontend/Fullstack/Data/AI/DevOps

## 예상 복잡도
- S/M/L/XL
```

## 4. 코드베이스 탐색 순서

정확한 검색이 먼저다.

| 단계 | 도구 | 목적 |
|------|------|------|
| 1 | RAG 또는 프로젝트 검색 | 의미 기반 후보 찾기 |
| 2 | rg | 정확한 심볼/문자열 검색 |
| 3 | rg --files | 파일 구조 파악 |
| 4 | Read/sed | 후보 파일 확인 |
| 5 | git diff/log | 최근 변경과 의도 파악 |

### 검색 예시

```typescript
const discovery = Skill("code-search", {
  semantic: "주문 취소 권한 검증",
  exact: ["cancelOrder", "OrderCancel", "CANCELLED"],
  output: ["files", "call graph", "tests"],
});
```

```python
scan = Agent("explorer", task="""
주문 취소 플로우 관련 파일을 찾아라.
구현하지 말고 파일, 함수, 테스트, 위험만 보고하라.
""")
```

## 5. Gemini 1M 스캔 활용

대규모 코드베이스에서는 Gemini를 넓은 스캐너로 사용한다.

**적합한 요청:**
- "이 기능이 전체 코드에서 어디에 걸려 있는가?"
- "비슷한 구현 패턴을 모두 찾아라"
- "API 계약 변경 시 영향 범위를 찾아라"
- "테스트가 어디에 있는지 찾아라"

**부적합한 요청:**
- 작은 오타 수정
- 이미 위치를 아는 단일 함수 수정
- 민감한 시크릿 포함 원문 전달

### Gemini 스캔 프롬프트 구조

```markdown
목표: 로그인 토큰 갱신 플로우 영향 분석

찾을 것:
- 서버 엔드포인트
- 프론트 인터셉터
- 테스트
- 운영 설정
- 레거시 fallback

출력:
- 관련 파일 목록
- 호출 흐름
- 리스크
- 구현 전 질문
```

## 6. 영향도 분석

영향도는 변경 파일보다 호출 흐름으로 판단한다.

| 영향 범위 | 확인할 것 |
|-----------|-----------|
| 함수 내부 | 입력값, 반환값, 예외 |
| 모듈 내부 | 호출자, 테스트, DI 설정 |
| API 계약 | 클라이언트, 문서, mock |
| 데이터 | migration, backfill, rollback |
| 운영 | config, secret, deployment |

### 영향도 기록 예시

```markdown
## 영향 범위
- Backend: OrderService.cancel(), OrderController.cancel()
- Frontend: OrderDetailCancelButton
- Data: order_status enum 변경 없음
- Test: OrderServiceTest, order-cancel.spec.ts
- Risk: 중복 요청, 이미 배송된 주문, 권한 없는 사용자
```

## 7. 기존 패턴 확인

새로운 설계를 만들기 전에 기존 방식이 있는지 확인한다.

**확인 대상:**
- 유사한 API 엔드포인트
- 같은 도메인의 DTO/Response 패턴
- 에러 코드와 메시지 규칙
- 테스트 fixture 작성법
- 프론트 상태 관리 방식
- 배포 설정 네이밍

```typescript
Agent("explorer", {
  task: "기존 환불/취소 API의 DTO, 에러 처리, 테스트 패턴만 요약",
  outputFormat: "파일 경로 + 패턴 + 재사용 권고",
});
```

## 8. 위험 목록 만들기

Phase 0의 핵심 산출물은 위험 목록이다.

| 위험 | 질문 | 후속 조치 |
|------|------|-----------|
| 권한 | 누가 이 작업을 수행할 수 있는가? | reviewer 추가 |
| 데이터 | 기존 데이터가 깨지는가? | migration 검토 |
| 동시성 | 중복 요청이 가능한가? | lock/idempotency |
| UI | 사용자가 상태를 오해하는가? | designer/qa |
| 운영 | 배포 순서가 필요한가? | ops-lead |

## 10. 하위 에이전트에게 전달할 요약

긴 스캔 결과를 그대로 전달하지 않는다.

전달 요약은 다음 정도로 충분하다.

- 목표 1문장
- 수정 대상 파일
- 사용해야 할 기존 패턴
- 주의할 위험
- 기대 산출물

```python
summary = {
    "goal": "주문 취소 API에 권한 검증 추가",
    "files": ["OrderService.ts", "OrderController.ts"],
    "pattern": "RefundService의 guard 패턴 재사용",
    "risks": ["중복 요청", "배송 완료 상태"],
}
Agent("backend-developer", task=summary)
```
