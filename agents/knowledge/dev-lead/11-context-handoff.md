# 컨텍스트 핸드오프

> 에이전트 간 전달 정보는 충분해야 하지만, 길고 무질서해서는 안 된다.

---

## 1. 컨텍스트 핸드오프의 목적

멀티에이전트 작업에서 품질은 핸드오프 품질에 크게 좌우된다.

하위 에이전트는 전체 대화가 아니라 자신의 역할에 필요한 압축된 맥락을 받아야 한다.

**좋은 핸드오프의 조건:**
- 목표가 한 문장으로 명확하다.
- 수정 범위와 금지 범위가 있다.
- 기존 패턴이 요약되어 있다.
- 위험과 검증 방법이 포함된다.
- 기대 산출물이 구체적이다.

---

## 2. 나쁜 핸드오프

```markdown
이거 전체적으로 보고 잘 고쳐줘.
관련 파일은 알아서 찾고 테스트도 알아서 해.
```

문제:
- 목표가 모호하다.
- 범위가 없다.
- 성공 기준이 없다.
- 파일 충돌 가능성이 크다.
- 결과 비교가 어렵다.

---

## 3. 좋은 핸드오프

```markdown
목표: 주문 취소 API에서 배송 완료 주문을 취소하지 못하게 막는다.

관련 파일:
- server/order/order.service.ts
- server/order/order.controller.ts
- server/order/order.service.spec.ts

기존 패턴:
- 환불 불가 상태는 `ConflictError`로 처리한다.
- 에러 코드는 `ORDER_NOT_CANCELABLE`을 사용한다.

제약:
- web/** 파일 수정 금지
- DB schema 변경 금지

출력:
- 변경 파일 목록
- 추가/수정 테스트
- 남은 위험
```

---

## 4. 핸드오프 기본 구조

| 항목 | 설명 |
|------|------|
| 목표 | 무엇을 달성할지 |
| 배경 | 왜 필요한지 |
| 입력 | Phase 0/1 요약 |
| 범위 | 수정할 파일과 금지 파일 |
| 계약 | API, 데이터, UI 규칙 |
| 위험 | 주의할 실패 경로 |
| 검증 | 실행할 테스트 |
| 출력 | 보고 형식 |

### 템플릿

```markdown
## Goal

## Context

## Scope

## Constraints

## Contract

## Risks

## Verification

## Output
```

## 5. 구현 에이전트 핸드오프

구현 에이전트에게는 결정된 내용을 준다.

결정되지 않은 문제를 숨기지 말고 질문으로 남긴다.

```typescript
Agent("backend-developer", {
  goal: "쿠폰 적용 API의 금액 계산 버그 수정",
  context: {
    pattern: "Money는 number가 아니라 MoneyValue 객체 사용",
    relatedFiles: ["coupon.service.ts", "coupon.service.spec.ts"],
  },
  scope: {
    allow: ["server/coupon/**"],
    deny: ["web/**", "infra/**"],
  },
  verification: ["coupon.service.spec.ts"],
  output: ["changed files", "tests", "open questions"],
});
```

## 6. 리뷰 에이전트 핸드오프

리뷰 에이전트에게는 diff와 검토 관점을 준다.

```python
Agent("code-reviewer", task="""
현재 변경분을 리뷰한다.

관점:
- 권한 우회 가능성
- 트랜잭션 정합성
- 중복 요청 처리
- 테스트 공백

출력:
- P0/P1/P2로 구분
- 파일과 근거 포함
- 수정 제안은 간결하게
""")
```

### 리뷰 핸드오프 주의점

- 구현 설명만 주지 말고 실제 변경 범위를 주어야 한다.
- 원하는 관점을 명확히 한다.
- 심각도 기준을 요구한다.
- Nit와 블로커를 구분시킨다.

## 7. QA 핸드오프

QA는 사용자 시나리오와 실패 경로를 필요로 한다.

```markdown
목표: 주문 취소 기능 테스트 전략

사용자:
- 일반 구매자
- 관리자

시나리오:
- 취소 가능 주문
- 배송 완료 주문
- 권한 없는 주문
- 중복 클릭
- 네트워크 실패

출력:
- 테스트 케이스 목록
- 자동화 우선순위
- 최소 회귀 세트
```

## 8. Designer 핸드오프

디자인 에이전트에게는 추상적 미감보다 사용 맥락을 전달한다.

| 전달 항목 | 예시 |
|-----------|------|
| 화면 목적 | 주문 상태 확인 |
| 사용자 행동 | 취소 가능 여부 판단 |
| 제약 | 기존 디자인 시스템 유지 |
| 상태 | loading, disabled, error, success |
| 검증 | 모바일과 데스크톱 모두 확인 |

---

## 9. Ops 핸드오프

운영 에이전트에게는 환경 차이와 롤백 가능성을 전달한다.

```typescript
Agent("ops-lead", {
  task: "Redis timeout 설정 변경 검토",
  context: {
    environments: ["local", "staging", "production"],
    configFiles: ["deploy/helm/values.yaml"],
    risk: "timeout 증가 시 connection pool 점유 증가",
  },
  output: ["deployment steps", "rollback", "metrics to watch"],
});
```

## 10. 결과 요약 수렴

하위 에이전트 결과는 다음 형식으로 수렴한다.

```markdown
## 에이전트 결과 요약

### backend-developer
- 완료:
- 변경 파일:
- 이슈:

### frontend-developer
- 완료:
- 변경 파일:
- 이슈:

### qa
- 테스트:
- 누락:

### dev-lead 판단
- 통합:
- 재수정:
- 최종 검증:
```

## 11. 컨텍스트 압축 원칙

| 원칙 | 설명 |
|------|------|
| 요약 우선 | 긴 로그보다 핵심 원인 |
| 파일 경로 포함 | 근거 추적 가능 |
| 결정과 미결정 분리 | 하위 에이전트 혼란 방지 |
| 금지 범위 명시 | 충돌 예방 |
| 출력 형식 고정 | 수렴 비용 감소 |
