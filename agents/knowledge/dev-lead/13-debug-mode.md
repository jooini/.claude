# 디버깅 모드 오케스트레이션

> 재현 → 수집 → 범위 축소 → 가설 → 검증 → 수정 → 확인 순서로 에이전트를 배치한다.

---

## 1. 디버깅 모드의 목적

디버깅은 추측 싸움이 아니다.

dev-lead는 증거를 모으고, 가설을 줄이고, 수정 후 재발 방지 테스트를 남긴다.

**디버깅 모드 원칙:**
- 재현 없이 원인 단정 금지
- 로그 없이 추측 금지
- 한 번에 여러 가설 수정 금지
- 수정 후 회귀 테스트 필수
- 2회 실패 시 접근 재검토

---

## 2. 디버깅 단계

| 단계 | 목적 | 담당 |
|------|------|------|
| 1. 재현 | 같은 실패 만들기 | debug-master, qa |
| 2. 수집 | 로그/상태/입력 확보 | explorer, ops |
| 3. 범위 축소 | 관련 코드 좁히기 | explorer |
| 4. 가설 | 원인 후보 정리 | debug-master |
| 5. 검증 | 후보별 증거 확인 | developer |
| 6. 수정 | 최소 변경 적용 | domain agent |
| 7. 확인 | 재현 케이스 통과 | qa, tester |

---

## 3. 디버깅 모드 진입 조건

- "버그", "에러", "안 돼", "실패" 요청
- 테스트 실패 원인이 불명확함
- 운영 로그의 5xx 증가
- 화면은 실패하지만 API는 성공
- 데이터 불일치 발생
- 성능 저하 원인 불명

---

## 4. Explore 단계

먼저 읽기 전용으로 증거를 모은다.

```typescript
Agent("explorer", {
  task: "로그인 실패 원인 후보를 코드 근거로 찾는다. 수정 금지.",
  searchOrder: ["RAG", "rg", "rg --files", "Read"],
  output: ["related files", "call flow", "suspected boundaries"],
});
```

### Explore 산출물

```markdown
## 재현 정보
- 입력:
- 기대:
- 실제:

## 관련 파일
- ...

## 호출 흐름
- ...

## 원인 후보
- ...
```

---

## 5. debug-master 단계

debug-master는 가설을 구조화한다.

```python
Agent("debug-master", task="""
로그인 API가 401을 반환하는 문제를 분석한다.
재현 정보와 관련 파일 요약을 바탕으로 가설 3개를 만들고,
각 가설을 검증할 읽기 전용 명령을 제안하라.
""")
```

### 가설 형식

| 가설 | 근거 | 검증 |
|------|------|------|
| 토큰 만료 계산 오류 | exp 단위 변환 코드 변경 | unit test 또는 timestamp 로그 |
| 쿠키 domain 불일치 | staging만 실패 | response header 확인 |
| user lookup exact 옵션 누락 | 유사 계정 존재 | query 결과 비교 |

---

## 6. 도메인 구현 단계

원인이 충분히 좁혀지면 해당 도메인 에이전트가 수정한다.

| 원인 | 담당 |
|------|------|
| API 로직 | backend-developer |
| UI 상태 | frontend-developer |
| 쿼리/지표 | data-analyst |
| 모델 응답 | ai-engineer |
| 배포 설정 | ops-lead |

```typescript
Agent("backend-developer", {
  task: "토큰 만료 계산 단위 오류 수정",
  evidence: "exp는 seconds인데 Date.now()는 milliseconds",
  scope: ["auth/token.service.ts", "auth/token.service.spec.ts"],
});
```

---

## 7. QA 회귀 단계

버그 수정은 회귀 테스트 없이는 완료하지 않는다.

```python
Agent("qa", task="""
이번 로그인 401 버그가 재발하지 않도록 테스트 케이스를 설계하라.
정상 토큰, 만료 토큰, clock skew, staging cookie domain을 포함하라.
""")
```

### 회귀 테스트 체크리스트

- [ ] 실패를 재현한 입력이 테스트에 포함되었는가?
- [ ] 정상 경로도 여전히 통과하는가?
- [ ] 경계값이 포함되었는가?
- [ ] 환경 차이가 원인이면 환경별 검증이 있는가?
- [ ] 테스트 이름이 버그 의도를 설명하는가?

---

## 8. 2회 실패 시 접근 재검토

같은 방식으로 두 번 수정 실패하면 멈춘다.

**재검토 질문:**
- 재현 조건이 틀린가?
- 로그 해석이 잘못되었는가?
- 다른 계층이 원인인가?
- 테스트가 실제 실패를 표현하는가?
- 최근 변경과 관련이 있는가?

```markdown
## 접근 재검토
- 시도 1:
- 시도 2:
- 공통 실패:
- 새 접근:
```

---

## 9. 운영 장애 디버깅

운영 장애는 ops-lead를 조기에 배치한다.

| 확인 | 담당 |
|------|------|
| 로그 패턴 | ops-lead |
| 배포 시점 | ops-lead |
| 코드 변경 | explorer |
| 재현 케이스 | debug-master |
| rollback 판단 | dev-lead + 사용자 |

운영에서 데이터 삭제, 결제, 인증 장애가 의심되면 사용자 에스컬레이션을 준비한다.

## 12. 최종 기준

디버깅 완료는 "에러가 사라짐"이 아니라 "원인, 근거, 수정, 회귀 테스트가 연결됨"이다.
