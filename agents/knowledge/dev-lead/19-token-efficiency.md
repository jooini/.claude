# 토큰 효율 오케스트레이션

> 큰 맥락은 요약하고, 단순 질문은 가벼운 모델에 맡기고, 대량 코드는 적절한 도구로 나눈다.

---

## 1. 토큰 효율의 목적

멀티에이전트 작업은 컨텍스트를 잘못 다루면 느려지고 비싸지고 부정확해진다.

dev-lead는 모델과 에이전트의 강점을 나눠 사용한다.

**목표:**
- 필요한 정보만 전달한다.
- 긴 원문은 요약 후 전달한다.
- 단순 질의는 가벼운 모델을 사용한다.
- 대규모 스캔은 Gemini에 맡긴다.
- 코드 구현과 검증은 Codex/전문 에이전트에 나눈다.

---

## 2. 모델/도구 역할

| 도구 | 적합한 일 | 피할 일 |
|------|-----------|---------|
| Ollama | 단순 요약, 빠른 질의 | 대규모 코드 변경 |
| Gemini | 대용량 코드베이스 스캔 | 세밀한 patch 작성 |
| Codex | 코드 구현, 리뷰, 검증 | 제품 정책 임의 결정 |
| 전문 에이전트 | 도메인 판단 | 무범위 전체 탐색 |
| RAG | 과거 문서/의미 검색 | 정확한 심볼 검색 단독 |
| rg | 정확한 문자열 검색 | 의미 기반 탐색 단독 |

---

## 3. 컨텍스트 예산 세우기

작업 시작 시 컨텍스트 예산을 정한다.

| 작업 규모 | 컨텍스트 전략 |
|-----------|----------------|
| S | 관련 파일 직접 읽기 |
| M | 검색 결과 + 핵심 파일 |
| L | Phase 0 요약 + 파일별 발췌 |
| XL | 별도 스펙/요약 파일 + 단계별 로드 |

### 예산 원칙

- 로그 전체를 붙이지 않는다.
- 스캔 결과는 요약으로 전달한다.
- 하위 에이전트에게 역할 밖 정보는 주지 않는다.
- 같은 파일을 여러 에이전트에게 중복 전달하지 않는다.

---

## 4. Ollama 단순 질의

로컬 Ollama는 빠른 보조 판단에 적합하다.

**적합한 요청:**
- 긴 로그의 한국어 요약
- 커밋 메시지 초안
- PR 설명 초안
- 간단한 대안 비교
- 문서 문체 정리

```python
Agent("ollama", task="""
다음 테스트 실패 로그를 5줄로 요약하고,
원인 후보를 우선순위로 정리하라.
코드 수정 제안은 하지 말라.
""")
```

---

## 5. Gemini 대용량 스캔

Gemini는 넓게 읽고 구조를 잡는 데 사용한다.

**사용 시점:**
- 관련 파일이 불명확함
- 대형 레거시 코드베이스
- 비슷한 패턴 전체 수집
- 영향 범위 분석
- 테스트 위치 찾기

### Gemini 출력 제한

```markdown
출력은 다음만 포함:
- 관련 파일 20개 이하
- 핵심 호출 흐름
- 기존 패턴
- 위험
- 구현 에이전트에게 줄 요약
전문 코드 덤프 금지
```

---

## 6. Codex 대량 코드 작업

Codex는 실제 코드 변경과 검증에 강하다.

```typescript
Agent("codex", {
  task: "repository 계층의 nullable 처리 일관화",
  ownership: ["server/user/**"],
  constraints: [
    "public API 변경 금지",
    "테스트 실패 시 로그 요약",
    "변경 파일 목록 보고",
  ],
});
```

### Codex에 주기 좋은 정보

- 정확한 파일 경로
- 기존 패턴
- 실패 테스트
- 기대 동작
- 금지 범위
- 검증 명령

---

## 7. 요약의 품질

나쁜 요약은 토큰은 줄이지만 판단을 망친다.

**좋은 요약 요소:**
- 결정된 사실
- 열린 질문
- 관련 파일
- 위험
- 테스트
- 다음 행동

```markdown
## 구현 핸드오프 요약
- 사실: OrderService는 배송 완료 상태를 cancel 불가로 처리하지 않음
- 파일: server/order/order.service.ts
- 패턴: RefundService는 ConflictError 사용
- 위험: 중복 요청 시 환불 API 재호출
- 테스트: order.service.spec.ts에 회귀 추가 필요
```

## 8. 토큰 낭비 패턴

| 낭비 | 개선 |
|------|------|
| 전체 로그 붙여넣기 | 에러 주변 100줄 |
| 전체 파일 전달 | 관련 함수와 import |
| 모든 에이전트에게 같은 맥락 | 역할별 요약 |
| 중복 검색 | 검색 결과 캐시 |
| 긴 회의식 설명 | 목표/범위/출력 표준화 |

---

## 9. 컨텍스트 갱신

작업이 길어질수록 요약을 갱신한다.

```markdown
## 현재 상태
- 완료:
- 실패:
- 결정:
- 남은 작업:
- 다음 에이전트:
```

이 요약은 이후 에이전트 핸드오프와 최종 보고의 기반이 된다.

## 10. 토큰 효율 체크리스트

- [ ] 하위 에이전트에게 역할 밖 컨텍스트를 주지 않았는가?
- [ ] 긴 로그와 파일을 요약했는가?
- [ ] 대규모 탐색은 Gemini/RAG를 사용했는가?
- [ ] 정확한 검색은 rg로 확인했는가?
- [ ] 단순 요약은 가벼운 모델로 처리 가능한가?
- [ ] 결과 수렴을 위한 공통 형식이 있는가?
- [ ] 같은 질문을 여러 번 반복하지 않았는가?

## 11. 최종 기준

토큰 효율은 아끼는 것이 목적이 아니라, 각 에이전트가 가장 관련 있는 정보로 더 정확하게 판단하게 만드는 것이다.
