## 1. QA 리더십이란

버그 찾기를 넘어 팀 전체의 품질 문화를 이끄는 것.
"QA만 품질을 책임진다"에서 "모두가 품질에 기여한다"로.

---

## 2. 품질 문화 구축

```
Shift Left (왼쪽으로 당기기):
  전통: 개발 → QA → 배포
  Shift Left: 요구사항 단계부터 QA 참여

  QA가 참여하는 시점:
    [ ] 요구사항 검토 (Defect Prevention)
    [ ] 스프린트 플래닝 (테스트 가능성 검토)
    [ ] 개발 중 (코드 리뷰, 테스트 지원)
    [ ] PR 리뷰 (테스트 코드 품질)
    [ ] 배포 전 (회귀 테스트)
    [ ] 배포 후 (모니터링)
```

## 3. QA 전략 문서화

```markdown
# QA 전략 v2.0

## 비전
모든 사용자가 신뢰할 수 있는 제품 경험을 제공한다.

## 목표 (OKR)
O: 운영 버그 50% 감소
  KR1: 결함 탈출율 10% → 5% 이하
  KR2: E2E 자동화율 30% → 70%
  KR3: CI 피드백 시간 15분 → 5분 이내

## 전략
1. 자동화 우선: 반복 회귀 테스트는 100% 자동화
2. Shift Left: 기획 단계부터 QA 참여
3. 데이터 기반: 메트릭 측정 및 의사결정
4. 개발팀 지원: 테스트 작성 교육 및 지원

## 현황 및 로드맵
Q1: 자동화 프레임워크 구축
Q2: E2E 핵심 시나리오 자동화
Q3: 성능 테스트 파이프라인
Q4: 보안 테스트 통합
```

## 4. 팀 역량 개발

```
QA 엔지니어 역량 로드맵:

Junior QA:
  - 테스트 케이스 작성/실행
  - 버그 리포트 작성
  - 기본 자동화 스크립트

Mid-Level QA:
  - 테스트 전략 수립
  - 자동화 프레임워크 설계
  - 성능/보안 테스트 기초

Senior QA:
  - QA 프로세스 설계
  - 팀 교육 및 멘토링
  - 품질 지표 정의 및 관리
  - 개발팀과 품질 문화 구축

QA Manager/Lead:
  - 조직 QA 전략 수립
  - 팀 구성 및 리소스 관리
  - 경영진과 품질 커뮤니케이션
```

## 5. 개발팀과의 협업

```
효과적인 QA-개발 협업:

스프린트 플래닝:
  - "이 기능 어떻게 테스트할 수 있을까요?"
  - "테스트 환경 준비가 필요한 게 있나요?"
  - 테스트 가능성 기반 Task 분리

데일리 스탠드업:
  - 블로킹 이슈 공유
  - 테스트 진행 상황

버그 발견 시:
  - 판단보다 사실 전달: "이 동작이 의도된 건가요?"
  - 재현 방법 + 증거 준비
  - 비난 없이 해결 방향 논의

릴리스 기준 합의:
  - 완료 기준 (DoD)에 테스트 기준 포함
  - 알려진 이슈 (Known Issues) 문서화
```

## 6. QA 온보딩 프로세스

```markdown
# QA 온보딩 (4주 플랜)

1주차: 제품 이해
  - 제품 전반 사용해보기
  - 비즈니스 도메인 이해
  - 기존 버그 목록 검토

2주차: 환경 및 프로세스
  - 개발 환경 세팅
  - CI/CD 파이프라인 이해
  - 기존 테스트 케이스 실행

3주차: 자동화 프레임워크
  - 자동화 도구 학습
  - 기존 자동화 코드 리뷰
  - 간단한 TC 자동화 작성

4주차: 실전 참여
  - 현재 스프린트 TC 작성
  - 독립적 기능 테스트 수행
  - 버그 리포트 작성
```

## 7. 인시던트에서 학습

```markdown
# 인시던트 후 QA 액션

인시던트 발생 후:
1. 어떤 테스트가 이 버그를 잡았어야 했는가?
2. 왜 테스트가 없었는가?
3. 테스트가 있었는데 통과했다면 왜 그랬는가?

개선 액션:
  - 재발 방지 테스트 케이스 추가
  - 테스트 전략 업데이트
  - 프로세스 개선 (코드 리뷰, 회귀 범위 등)

Post-mortem 기여:
  - QA 관점의 원인 분석 제공
  - 비난 없는 (Blameless) 접근
  - 구체적 예방 액션 제안
```

## 8. 안티패턴

- **QA = 버그 찾는 사람**: 품질 파트너로 포지셔닝
- **릴리스 직전에만 QA 참여**: 전체 개발 사이클에 참여
- **QA 혼자 품질 책임**: 개발팀도 테스트 코드 작성 공동 책임
- **메트릭 없는 QA**: 데이터 기반 개선이 어려움
- **도구에만 의존**: 자동화가 전부가 아님 — 탐색적 테스트 병행
