## 1. 테스트 계획이란

특정 릴리스/스프린트에서 무엇을 어떻게 테스트할지 구체적 계획.
전략(What & Why) → 계획(How & When & Who)

---

## 2. 테스트 계획 프로세스

```
요구사항 분석
    ↓
테스트 범위 정의
    ↓
테스트 케이스 도출
    ↓
우선순위 설정
    ↓
일정 및 리소스 산정
    ↓
실행 & 결과 추적
    ↓
완료 기준 확인
```

---

## 3. 테스트 계획 문서

```markdown
# 스프린트 15 테스트 계획

## 테스트 대상
- 결제 플로우 v2 (주요 변경)
- 사용자 프로필 편집 (신규)
- 알림 설정 (마이너 변경)

## 테스트 범위

### IN (테스트함)
- 결제 수단 추가/삭제/변경
- 결제 성공/실패/취소 시나리오
- 프로필 이미지 업로드 (5MB 이하)
- 알림 ON/OFF 토글

### OUT (테스트 안 함)
- PG사 내부 결제 처리 (서드파티)
- 기존 주문 내역 조회 (변경 없음)

## 테스트 환경
- 스테이징: https://staging.example.com
- 테스트 계정: qa-test@example.com / Qa1234!
- 결제 테스트 카드: 4242-4242-4242-4242

## 일정
| 단계 | 기간 | 담당 |
|------|------|------|
| 테스트 케이스 작성 | D-3 ~ D-2 | QA팀 |
| 기능 테스트 | D-1 ~ D+1 | QA팀 |
| 회귀 테스트 | D+2 | QA팀 |
| 버그 수정 검증 | D+3 | QA팀 |

## 완료 기준
- [ ] 모든 P1 테스트 케이스 통과
- [ ] P2 테스트 케이스 90% 통과
- [ ] Critical/High 버그 0개
- [ ] Medium 버그 3개 이하 (Known Issue로 관리)
```

## 4. 테스트 케이스 도출 기법

### 동등 분할 (Equivalence Partitioning)

입력값을 동일한 결과를 내는 그룹으로 분류.

```
나이 입력 필드 (1~120 유효)
├── 유효 파티션: 1~120
├── 무효 파티션 1: 0 이하
├── 무효 파티션 2: 121 이상
└── 무효 파티션 3: 숫자가 아닌 값

→ 각 파티션에서 대표값 1개씩만 테스트
```

### 경계값 분석 (Boundary Value Analysis)

경계에서 버그가 많이 발생.

```
나이 1~120 유효
→ 테스트 값: 0, 1, 2, 119, 120, 121
(경계값과 경계 ±1)
```

### 결정 테이블 (Decision Table)

여러 조건 조합을 체계적으로.

```
할인 로직:
조건1: 회원등급 VIP여부
조건2: 쿠폰 보유여부
조건3: 구매금액 5만원 이상

┌──────┬──────┬──────┬──────────┐
│ VIP  │ 쿠폰 │ 5만↑ │ 결과     │
├──────┼──────┼──────┼──────────┤
│  Y   │  Y   │  Y   │ 30% 할인 │
│  Y   │  Y   │  N   │ 20% 할인 │
│  Y   │  N   │  Y   │ 20% 할인 │
│  Y   │  N   │  N   │ 10% 할인 │
│  N   │  Y   │  Y   │ 15% 할인 │
│  N   │  Y   │  N   │ 10% 할인 │
│  N   │  N   │  Y   │  5% 할인 │
│  N   │  N   │  N   │  0% 할인 │
└──────┴──────┴──────┴──────────┘
```

## 5. 테스트 케이스 작성

```markdown
## TC-PAY-001: 신규 카드 등록 성공

**우선순위**: P1 (Critical)
**사전 조건**: 로그인된 상태, 등록된 카드 없음

**테스트 단계**:
1. 마이페이지 > 결제수단 > 카드 추가 클릭
2. 유효한 카드 번호 입력: 4242-4242-4242-4242
3. 유효기간 입력: 12/26
4. CVC 입력: 123
5. 카드 별명 입력: "내 주카드"
6. 등록 버튼 클릭

**기대 결과**:
- "카드가 등록되었습니다" 토스트 표시
- 카드 목록에 새 카드 표시 (마스킹: **** **** **** 4242)
- 카드 별명 "내 주카드" 표시

**관련 요구사항**: REQ-PAY-003
```

## 6. 우선순위 설정

| 우선순위 | 기준 | 예시 |
|---------|------|------|
| P1 (Critical) | 서비스 불가 또는 데이터 손실 | 결제 불가, 로그인 불가 |
| P2 (High) | 주요 기능 동작 불가 | 장바구니 추가 불가 |
| P3 (Medium) | 기능은 되지만 UX 불량 | 에러 메시지 부정확 |
| P4 (Low) | 사소한 UI 이슈 | 여백 1px 틀림 |

---

## 7. 안티패턴

- **테스트 케이스 없는 테스트 실행**: 기억에 의존 → 누락 발생
- **Happy Path만 테스트**: 엣지 케이스, 에러 시나리오 필수
- **지나치게 상세한 계획**: 변경에 취약 → 핵심만
- **QA만 계획 작성**: 개발자/PM과 함께 범위 합의
- **완료 기준 없는 계획**: 언제 끝났는지 모름
