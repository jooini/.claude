# 디버깅 안티패턴

> 안티패턴은 시간을 쓰는 방식의 문제다. 바쁘게 움직이지만 원인에는 가까워지지 않는다.

---

## 1. 재시작 증후군

증상:

- 에러가 나면 서버부터 재시작한다.
- 재시작 후 잠시 정상이라 완료했다고 판단한다.
- 원인 기록이 없다.

왜 위험한가:

- 메모리 누수, connection leak, stale state의 증거가 사라진다.
- 운영 장애에서 재발 시간을 늦출 뿐이다.
- 재시작이 부작용을 숨긴다.

대안:

- [ ] 재시작 전 메모리, connection, thread, 로그를 수집한다.
- [ ] 재시작 후 정상화 시간을 기록한다.
- [ ] 재시작이 왜 효과가 있었는지 가설을 세운다.

---

## 2. 추측 수정

나쁜 예:

```typescript
// 에러가 timeout이라서 일단 5초에서 60초로 늘림
const timeoutMs = 60_000;
```

문제:

- timeout 원인이 slow query인지 외부 API인지 모른다.
- 사용자 대기 시간이 늘어난다.
- connection pool 점유 시간이 늘어 장애가 커질 수 있다.

대안:

```typescript
logger.info({
    event: 'dependency.call.done',
    dependency: 'payment',
    durationMs,
    status,
});
```

측정 후 병목 구간을 수정한다.

---

## 3. 여러 변경 동시 적용

한 번에 로그 추가, 캐시 비활성화, 쿼리 수정, timeout 변경을 같이 하면 무엇이 효과였는지 모른다.

규칙:

- [ ] 한 실험에 하나의 변경.
- [ ] 변경 전 예상 결과를 기록.
- [ ] 결과가 나오면 유지/되돌림 결정.
- [ ] 다음 변경으로 이동.

---

## 4. 최근 변경만 보는 오류

최근 배포는 강한 힌트지만 항상 원인은 아니다.
데이터 증가, 외부 API 변경, 인증서 만료, 배치 시간 변경도 같은 시점에 겹칠 수 있다.

확인할 것:

- [ ] 배포 시각과 증상 시작 시각
- [ ] 설정 변경
- [ ] 데이터 마이그레이션
- [ ] 외부 장애 공지
- [ ] 트래픽 변화

---

## 5. 로그 없는 완료 선언

나쁜 완료:

```text
로컬에서 한 번 해보니 됩니다.
```

좋은 완료:

```text
기존 재현 스크립트 100회 실행에서 실패 23회 → 수정 후 0회.
회귀 테스트 `inventory.concurrent.spec.ts` 추가.
```

완료는 증거로 말한다.

---

## 6. 에러 삼키기

```python
try:
    send_email(user)
except Exception:
    pass
```

이 코드는 장애를 없애지 않고 관찰 가능성만 없앤다.

대안:

```python
try:
    send_email(user)
except EmailProviderError as exc:
    logger.warning("email.send.failed", extra={"user_id": user.id, "reason": str(exc)})
    metrics.increment("email_send_failed")
```

복구 가능한 에러라도 기록과 지표가 필요하다.

---

## 7. 테스트 약화

버그 수정 중 테스트 기대값을 실제 결과에 맞춰 바꾸는 경우가 있다.
이것은 회귀 방지가 아니라 버그를 승인하는 것이다.

체크:

- [ ] 요구사항이 바뀐 것인가?
- [ ] 기존 테스트가 잘못된 것인가?
- [ ] 버그 때문에 테스트가 실패한 것인가?
- [ ] 새 기대값을 설명하는 문서나 이슈가 있는가?

---

## 8. 운영 데이터 직접 수정

```sql
UPDATE users SET status = 'ACTIVE' WHERE id = 'u-1';
```

문제:

- 원인 코드가 남아 있으면 다시 깨진다.
- audit trail이 없으면 추적이 어렵다.
- 관련 테이블과 불일치가 생긴다.

대안:

- [ ] dry-run SELECT
- [ ] 트랜잭션
- [ ] 영향 row 수 확인
- [ ] 복구 쿼리
- [ ] 코드 수정과 데이터 보정 분리

---

## 9. 캐시 탓 고정관념

캐시는 자주 의심받지만 항상 원인은 아니다.
cache bypass 실험 없이 캐시를 지우면 증거가 사라진다.

검증:

```bash
curl -sS "$BASE_URL/api/products/p-1" \
    -H "x-debug-bypass-cache: true" \
    | jq .
```

bypass에서도 실패하면 캐시는 제외한다.

---

## 10. 변경사항 무시

디버깅 중 워크트리에 이미 변경이 있으면 반드시 확인한다.
사용자 변경, 생성 파일, 임시 수정이 원인 분석에 영향을 줄 수 있다.

```bash
git status --short
git diff -- src/order/order.service.ts
```

타인의 변경을 되돌리지 않는다.
필요하면 현재 변경과 새 수정의 경계를 명확히 한다.

---

## 11. 도구 출력 과신

APM, 로그 검색, AI 요약은 힌트다.
실제 코드와 데이터로 검증해야 한다.

질문:

- [ ] 샘플링 때문에 누락된 요청은 없는가?
- [ ] clock skew가 있는가?
- [ ] 로그 파서가 stack trace를 잘랐는가?
- [ ] metric label cardinality가 누락을 만들었는가?

---

## 12. 안티패턴 체크리스트

- [ ] 재현 없이 수정하고 있지 않은가?
- [ ] 증거 수집 전에 재시작하지 않았는가?
- [ ] 여러 조건을 동시에 바꾸지 않았는가?
- [ ] 실패한 가설을 붙잡고 있지 않은가?
- [ ] 테스트를 약화하지 않았는가?
- [ ] 완료를 숫자와 로그로 말할 수 있는가?
