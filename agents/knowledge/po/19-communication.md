## 제품 비전 스토리텔링

데이터와 로직만으로는 사람을 움직이지 못한다. 비전은 **감정적 공명**을 일으켜야 팀이 몰입한다.

### 비전 스토리 구조

```
1. 현재의 문제 (Pain)
   → "지금 사용자들은 이런 고통을 겪고 있습니다"
   → 구체적 사례, 감정적 연결

2. 이상적 미래 (Promise)
   → "우리 제품이 있는 세계에서는 이렇게 됩니다"
   → 생생한 묘사, 구체적 시나리오

3. 어떻게 도달하는가 (Path)
   → "이 비전에 도달하기 위한 우리의 전략은..."
   → 단계별 계획, 첫 번째 마일스톤

4. 왜 우리인가 (Proof)
   → "우리가 이것을 해낼 수 있는 이유는..."
   → 팀의 강점, 초기 성과, 고유 인사이트
```

**실전 팁:**
- 숫자 + 이야기: "30%의 사용자가..." + "김 매니저는 매일 아침..."
- 적의 설정: 문제/비효율을 적으로 프레이밍 (경쟁사가 아닌)
- 반복: 같은 비전을 다른 형식으로 7번 이상 전달

## 이해관계자별 커뮤니케이션 맞춤

### 경영진 (C-Level)

- **관심사**: 비즈니스 임팩트, ROI, 전략 정합성, 리스크
- **형식**: 1-page 요약, 핵심 지표 차트, BLUF (결론 먼저)
- **빈도**: 격주-월간
- **언어**: "revenue", "market share", "competitive advantage"

### 개발팀 (Engineering)

- **관심사**: 기술 방향, 범위, 우선순위, Why
- **형식**: 상세 PRD, API 스펙, 기술 제약 논의
- **빈도**: 스프린트 단위
- **언어**: "요구사항", "제약 조건", "트레이드오프"

### 디자인팀 (Design)

- **관심사**: 사용자 니즈, 흐름, 제약 조건
- **형식**: User story, Journey map, 사용성 데이터
- **빈도**: 주간
- **언어**: "사용자", "경험", "Pain point"

### Sales / CS

- **관심사**: 언제 나오는가, 무엇을 약속할 수 있는가
- **형식**: 기능 개요, 출시 날짜, 고객 FAQ
- **빈도**: 월간 + 주요 출시 시
- **언어**: "고객 가치", "경쟁 우위", "타임라인"

## 효과적인 글쓰기 원칙

### BLUF (Bottom Line Up Front)

군사 커뮤니케이션에서 유래. **결론을 먼저, 배경은 그 다음.**

```
❌ 나쁜 예:
"지난 분기 데이터를 분석한 결과, 여러 요인을 고려하여...
(3단락 후) ...따라서 A를 하는 것이 좋겠습니다."

✅ 좋은 예:
"A를 권고합니다.
이유: 데이터가 X를 보여주고, Y 위험이 있기 때문입니다.
배경: [필요한 경우만]"
```

### 피라미드 구조

```
결론 (1문장)
  ├── 핵심 논거 1
  │     └── 증거
  ├── 핵심 논거 2
  │     └── 증거
  └── 핵심 논거 3
        └── 증거
```

### 이메일 / 슬랙 커뮤니케이션

**이메일:**
- 제목: 행동 요청이 명확하게 ("[승인 필요] Q2 로드맵 변경안")
- 첫 문장: 핵심 요청/결론
- 본문: 맥락과 세부사항
- 마지막: 명확한 다음 스텝 + 데드라인

**슬랙:**
- 긴 내용은 Thread로
- @멘션은 꼭 필요할 때만
- 채널 목적에 맞게 (#general vs #product-team vs #eng)

## 어려운 대화 (Difficult Conversations)

### 나쁜 소식 전달

```
1. 직접적으로, 빠르게
   → "이번 출시가 지연됩니다"를 먼저

2. 원인 설명 (변명 X)
   → 구체적 이유, 학습 포인트

3. 다음 스텝 제시
   → "어떻게 할 것인가"를 함께

4. Q&A
   → 질문을 막지 말고 충분히 받기
```

### 의견 불일치 처리

```
1. 상대방 관점 먼저 이해
   → "말씀하신 것은 X라고 이해했는데 맞나요?"

2. 공통점 찾기
   → "우리 모두 Y를 원한다는 점은 같네요"

3. 차이점 명확히
   → "제가 다르게 생각하는 부분은..."

4. 데이터/원칙으로 판단
   → 개인 의견이 아닌 객관적 기준으로

5. Disagree and Commit
   → 결정 후에는 전적으로 지지
```

## 미팅 퍼실리테이션

### 효과적인 미팅의 조건

- **명확한 목적**: "이 미팅의 성공은 X를 결정하는 것"
- **사전 준비**: 자료를 미팅 전에 공유, 미팅 시간은 논의에만
- **올바른 참석자**: Decision maker + 필요한 전문가만
- **명확한 시간 관리**: 어젠다 + 시간 배분
- **액션 아이템**: 담당자 + 데드라인 명시

### 결정 회의 vs 정보 공유 회의

```
결정 회의:
  - 명확한 선택지 제시
  - Decision maker 참석 필수
  - 미팅 내에서 결정

정보 공유 회의:
  - 비동기(문서)로 대체 가능한지 먼저 고려
  - 질문에 답하는 시간 확보
  - 녹화 or 요약 공유
```
