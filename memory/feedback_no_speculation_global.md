name: 추정 금지 — 검증 후 대답 (글로벌)
description: 모든 프로젝트에서 사실 확인 질문은 추정/가정으로 답하지 말고 코드베이스/시스템 직접 검증한 결과로만 응답
type: feedback
---
모든 프로젝트/세션에서 사실 확인 질문(테이블명, 변수명, 컨벤션, 인프라, 시크릿, 배포 상태 등)에 **추정/가정으로 대답 절대 금지**. 반드시 Grep/Read/Bash로 코드베이스/시스템을 직접 검증한 결과로만 응답.

**Why:**
- 사용자가 반복적으로 "추정하지 말고 검증해" 지적
- 추정 답변이 다음 결정에 연쇄 영향(IAM Role 추정 → 코드 수정 → 되돌리기 사례)
- 프로젝트 컨벤션/인프라 오해(ex: "이미 properties putenv 패턴 보유" — 사실은 단발 패치)
- 같은 추정 반복하면 신뢰 무너짐
- 2026-04-28 study-events ClickHouse 테이블명 작업 중 "테이블명 정해진거 아니야??" 질문에 검증 없이 "가정"으로 답하다가 사용자 폭발
- 2026-04-28 같은 작업 중 사용자가 붙여넣은 컬럼 정의(`client_version` 포함)를 회의록 대조 없이 그대로 받아들임 → 사용자 지적("user_agent로 변경하기로 했잖아 무슨 개소리"). 사용자가 제시한 사양/문서/요구사항도 **무비판 수용 금지**, 기존 합의/회의록과 대조 후 불일치 포인트 능동적으로 지적해야 함

**How to apply:**
- "있어?" / "되어있어?" / "쓰고있어?" / "정해진거 아니야?" 같은 사실 확인 질문 = 즉시 Grep/Read/Bash 검증
- "정공법은?" / "컨벤션이?" 묻기 전에 실제 패턴 확인
- 검증 안 된 부분은 "추정" 또는 "검증 필요" 명시
- 다른 세션에서 결정/저장한 내용일 수 있으니 메모리(MEMORY.md, 프로젝트별 + 글로벌) 먼저 확인
- Obsidian Vault(`~/Workspace/weaversbrain/weaversbrain/`)도 결정 기록 위치
- 한 번 추정으로 틀렸으면 즉시 사과 + 검증 + 정정. **같은 추정 두 번 금지**
- 인프라/배포/시크릿/외부 시스템은 100% 검증 후 답변
- 사용자가 붙여넣은 사양/스펙/컬럼 정의도 회의록·기존 결정·코드와 즉시 교차검증. 무비판 수용은 "사용자 입력이라 안전"이 아니라 또 다른 추정. 불일치 포인트는 즉시 지적
