# goals-tracking

Goal 양식 표준, 달성률 계산, Hit/Partial/Miss 분류, 재설정 로직.

## Goal 양식

```markdown
### Goal: [제목]
- **근거 focus-area**: (파일·섹션)
- **측정 지표**: (Collector metrics.md 중 하나, 또는 "정성")
- **기준값 (baseline)**: 이 사이클 시작 시점의 값
- **목표값 (target)**: 다음 평가 시점의 목표
- **타입**: 정량(자동 점검 가능) / 정성(자가 보고)
```

## 달성률 계산

### 정량

```
달성률 = (현재값 - 기준값) / (목표값 - 기준값) × 100%
```

- 목표값 < 기준값인 경우(감소 목표): `(기준값 - 현재값) / (기준값 - 목표값) × 100%`
- 둘 다 0이면 정의 불가 → 정성 처리

### 정성

사용자에게 4단계 프롬프트:
- `100%` — 달성
- `부분` — 30~99%
- `0%` — 미진행
- `취소` — 목표 자체 철회

## 3분류

| 분류 | 조건 | 아이콘 |
|---|---|---|
| Hit | 100% 이상 | ✅ |
| Partial | 30~99% | 🟡 |
| Miss | 30% 미만 또는 취소 | ❌ |

## 분류별 처리

### Hit
- 축하 문구 1줄
- **다음 Goal 제안 필수**: "이 goal은 sustained maintenance로 전환하고 새 영역 X 도입 제안"
- 기존 goal은 `archived: true` 로 표시

### Partial
- 현재 trajectory 언급 (선형 가정 시 N개 사이클 후 달성)
- **연장 여부 결정**: 한 사이클 더 연장 vs 목표값 하향 조정
- 연장 시 목표값 동일, 기준값은 현재값으로 갱신

### Miss
- 원인 질문 3개 제시:
  1. 우선순위가 바뀌었나?
  2. 측정 지표가 현실과 맞나?
  3. 일정 자체가 비현실적이었나?
- 답변 기반으로 다음 중 하나:
  - **재설정**: 목표값 하향, 기준값 갱신, 다음 사이클로 이월
  - **포기**: "이 영역은 더 이상 focus가 아니다" 선언, 다른 영역으로 교체

## 연속 Miss 정책

같은 goal 2회 연속 Miss → 재설정 금지, **포기 or 교체만 허용**. 문서에 근거 명시.

## Goal 수 권장

- 사이클당 **3~5개**
- 많으면 집중력 분산, 적으면 분기 변화 포착 어려움
- 최소 1개는 정성 goal (Collector가 못 잡는 소프트 스킬 영역)

## 예시 계산

### Hit

```
Goal: identity-hub-frontend 테스트 커버리지 5% 돌파
기준값: 1.2% / 목표값: 5.0% / 현재값: 5.3%
달성률: (5.3 - 1.2) / (5.0 - 1.2) × 100% = 107.9%
→ ✅ Hit
```

### Partial

```
Goal: SSO 운영 지식 문서화 30건
기준값: 12 / 목표값: 30 / 현재값: 20
달성률: (20 - 12) / (30 - 12) × 100% = 44.4%
→ 🟡 Partial
```

### Miss

```
Goal: Kotlin 신규 도메인 1개 deliver (정성)
사용자 보고: "0%. 우선순위가 SSO 안정화로 바뀜"
→ ❌ Miss (연속 1회)
원인: 우선순위 shift
처리: 재설정 (다음 사이클로 이월, 목표값 동일)
```

## Goals 소스 체인

1. **첫 사이클**: `Plans/focus-areas-*.md`에서 체크포인트 추출 → Goal 3~5개 제안
2. **이후 사이클**: 이전 사이클 performance-cycle.md §7에서 직전 Goals 자동 추출
3. Collector가 지난 값과 현재 값 둘 다 가지고 있으므로 자동 계산
4. 정성 goal은 수동 답변 필요 (analyze 실행 시 프롬프트)
