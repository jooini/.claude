# 이력서 작성 6단계 워크플로우

`SKILL.md`의 workflow 섹션을 상세화한 가이드. 각 단계의 액션·산출물·체크포인트 포함.

> 본 문서의 모든 시간순 매트릭스·단편 추출·검수 보고 예시는 **가상 시나리오**입니다. 실제 호출 시 사용자의 실제 자료에서 추출한 내용으로 대체합니다.

---

## Step 1. 자료 수집

### 입력
- `$ARGUMENTS` = 자료 디렉토리 / 단일 파일 / 빈 값

### 액션
- 디렉토리 받았으면 `Glob`으로 `**/*.{md,html,txt,pdf}` 패턴 탐색
- 단일 파일 받았으면 `Read`로 직접 읽기
- 빈 값이면 사용자에게 `AskUserQuestion`으로 자료 위치 요청

### 추가 자료 요청 (필요 시)
사용자가 자료 부족하면 다음을 묻기:
- 최근 회사·역할·주요 프로젝트 3~5개
- 블로그/GitHub/포트폴리오 URL
- 본인이 가장 자신 있는 강점 3개 (자기 어휘로)

### 산출물
- 자료 요약 메모 (회사별 기간/역할/대표 프로젝트 + 강점 후보 리스트)

### 체크포인트
- 자료가 최소 회사 3개 + 프로젝트 5개 이상 있는가?
- 사용자의 강점 키워드가 명확한가?
- 부족하면 다음 단계로 가지 말고 추가 요청

---

## Step 2. 포커스 + 타겟 결정

### 액션
사용자에게 `AskUserQuestion`으로 묻기. 각 질문은 옵션 형태로 제공:

#### 질문 1: 타겟 포커스 (단일 선택)
옵션 후보 (자료에서 도출 가능한 것 우선 노출):
- **Backend Platform** — 백엔드/플랫폼/인프라
- **Mobile · ML** — 모바일 + ML/Camera/온디바이스
- **Frontend** — 웹 프론트엔드 + UI/UX
- **SDK · Infrastructure** — 라이브러리/SDK/플랫폼 인프라
- **AI Builder** — AI 워크플로우/에이전트 운영/자동화
- **Data · ML Engineering** — 데이터 파이프라인/ML 인프라
- **Full-stack** — 풀스택 제너럴

#### 질문 2: 타겟 시장 (단일)
- **한국** (PDF 2장+2장, 한국형 라벨, 직책 이중 표기 적용)
- **글로벌** (1~2 페이지 영문, 빅테크 LP/STAR 적용)
- **둘 다** (별도 이력서 2개 생성)

#### 질문 3: 채용 공고 URL (선택)
있으면 `WebFetch`로 가져와서 키워드 추출. 없으면 일반론으로 진행.

#### 질문 4: 본인 시그너처 동사 3~5개
사용자가 못 떠올리면 다음 절차:
1. **자료 본문에서 반복 등장 동사 추출** (예: 분리/통합/자동화/정리)
2. **블로그·GitHub README 어휘 검토** 후 후보 5~7개 도출
3. **사용자에게 후보 제시** → 사용자가 3~5개 선택 또는 추가

참고 어휘 (시장별):
- 한국어: 분리한다 / 회수한다 / 통일한다 / 자산화한다 / 흐름으로 본다 / 경계를 짓는다 / 자동화한다
- 영어: separate / decouple / own / shipped / unblock / consolidate / automate

### 채용 공고 분석 (URL이 있을 때)
- `WebFetch`로 공고 가져오기
- 추출:
  - 핵심 기술 스택 (필수 vs 우대)
  - 인재상 키워드 (자율/주도/협업/오너십 등)
  - 회사 도메인·규모·단계
  - "절대 어기면 안 되는" 자격 요건
- 포커스와 일치 검증. 불일치하면 사용자에게 알림.

### 산출물
- 포커스 1개 확정
- 타겟 시장 확정 (한국이면 Part 4 적용)
- 시그너처 동사 3~5개 메모
- (있으면) 채용 공고 키워드 메모

### 체크포인트
- 포커스가 두 개 이상이면 → 별 이력서로 분리 제안
- 시그너처 동사가 안 나오면 → 사용자 블로그/GitHub README 읽고 도출 시도

## Step 3. 실(thread) 추출 — Thread Re-narration

**이 단계가 핵심.** 원칙 #13.

### 절차

#### 3-1. 시간순 매트릭스 작성
모든 회사·프로젝트를 시간순으로 나열. 각 프로젝트의 활동을 키워드로 메모.

**가상 예시 (백엔드 엔지니어 케이스)**:
```
[2019 회사 A — 스타트업] PHP 모놀리식 REST API, MySQL, 첫 백엔드
[2020 회사 B — 핀테크] Spring Boot 전환, 결제 모듈 Owner
[2021 회사 B — 핀테크] 결제 모듈 분리 (모듈화 첫 시도), Kafka 도입
[2022 회사 B — 핀테크] 정산 시스템 재설계, p99 250ms → 80ms
[2023 회사 C — 빅테크] MSA 전환 주도, 결제 도메인 4개 서비스 분리
[2024 회사 C — 빅테크] 99.95% SLO 운영, RCA 6건
[2025 회사 C — 빅테크] Claude Code 도입, 마이그레이션 가속화
```

#### 3-2. 타겟 포커스 필터링
각 프로젝트에서 **타겟 포커스와 관련된 단편**만 추출.

**가상 예시 — Backend Platform 포커스**:
```
[2019 회사 A] PHP REST API → 첫 백엔드 경험 (origin)
[2020 회사 B] Spring Boot 전환 → 프레임워크 전환 의사결정
[2021 회사 B] 결제 모듈 분리 + Kafka → 모듈화·이벤트 첫 도입
[2023 회사 C] MSA 전환 → 단독 도메인 운영 책임
[2024 회사 C] SLO 운영 → 안정성 운영 단계
```

**가상 예시 — AI Builder 포커스 (같은 사람, 다른 실)**:
```
[2020 회사 B] Spring Boot 전환 시 빌드 자동화 → 자동화 첫 시도
[2022 회사 B] 정산 시스템 검증 자동화 → 검증 파이프라인
[2024 회사 C] Observability 자동화 → AI Ops 기반
[2025 회사 C] Claude Code 도입 → AI 워크플로우 단독 도입
```

#### 3-3. 일관성 검증
단편들이 **하나의 강점이 점점 깊어지는 흐름**으로 읽히는지:
- ✅ "첫 백엔드 → 프레임워크 전환 → 모듈화 → MSA → SLO 운영" — 5단계 일관
- ❌ "백엔드 → 프론트엔드 → DB → 백엔드" — 끊어진 흐름

#### 3-4. Origin story 라벨 박기
초기 경력 단편에 **명시적 라벨** 추가:

**가상 예시**:
- ✅ "[2019 회사 A] PHP REST API — **(현재 결제 플랫폼 운영의 기반이 된 첫 백엔드 경험)**"
- ✅ "[2020 회사 B] Spring Boot 전환 — **첫 대규모 프레임워크 전환 의사결정**"

### 산출물
- 단편 4~6개의 시간순 리스트 (척추)
- 각 단편에 origin story 라벨 (초기 경력)

### 체크포인트
- 단편이 4개 미만이면 → 자료 부족 또는 포커스가 사용자 경력과 안 맞음. 포커스 재검토 요청.
- 단편이 7개 이상이면 → 너무 늘어짐. 핵심 5개로 압축.
- 끊어진 흐름이면 → 사용자에게 "이 시기에 이 강점이 어떻게 연결되는지" 물어보고 보완

## Step 4. 일관 서사 설계

### 골격 구조

```markdown
# {이름}
## {Headline — 타겟 포커스로 좁힌 직책}

## Summary (4~5줄)
약 N년간 {포커스 영역}에서 {시그너처 동사 1~2개}를 해온 엔지니어입니다.
{실 1단계}부터 {실 N단계}까지 {일관된 흐름 한 문장}.
{대표 성과 1~2개} 경험.

## Core Competencies (4개, 서로 다른 4축)
1. {기술적 강점 1}
2. {기술적 강점 2}
3. {기술적 강점 3}
4. {Development 시그널 1개 — 사람/문화/온보딩}

## Experience (최근 → 과거)
### {회사 1} — {역할}
{Scope 카드 1줄}
- {프로젝트 1: 추출된 단편 부각}
- {프로젝트 2}
...

### {회사 N} — {역할}
{Scope 카드 1줄}
- {origin story 라벨 포함 단편}

## Side Project / Blog (포커스 관련만)

## Education
```

### 단편 배치 시 원칙별 메모

각 단편 옆에 어느 원칙을 충족하는지 메모.

**가상 예시 (한 프로젝트 단편)**:
```
[회사 C — MSA 전환]
- "결제 도메인 모놀리식 → MSA 4개 서비스, 중복 코드 80% 공통 추출" #2 수치
- "DAU 8M, 결제 RPS 2K, 백엔드 12명 팀" #9 Scope
- "gRPC 대신 REST 유지 — 디버깅 도구 호환성을 위해 latency 15% 손해 수용" #10 Trade-off
- "**'결제 지연'을 'p99 250ms 임계 위반 주 5회'로 재정의 → 모놀리식 락 경합 가설 → 분산 트레이스로 확인**" #14 Problem Framing
- "MSA 전환 출하 → p99 250ms→80ms → growth팀 결제 실험 unblock" #11 3층
```

### 체크포인트
- Competency 4축이 서로 다른가? (#6)
- Development 시그널 1개 포함? (#6)
- 한국 시장이면 Headline에 직책 이중 표기? (#19)

## Step 5. 생성

### 액션
골격에 살을 붙여 이력서 작성. 작성 중 즉시 원칙 적용.

### 작성 중 체크리스트
- [ ] Competency 헤더 평문 (#1)
- [ ] bullet당 수치 ≤ 2개 (#2)
- [ ] 코드 백틱·내부 클래스명 X (#3)
- [ ] 프로젝트당 bullet ≤ 5개 (#4)
- [ ] Ownership 동사 화이트리스트 (#5) — owned/drove/defined/shipped 사용
- [ ] 시그너처 동사 자연스럽게 흐르는가 (#5)
- [ ] 첫 포지션 첫 줄 Scope 카드 (#9)
- [ ] 주요 결정마다 Trade-off 1줄 (#10)
- [ ] 핵심 프로젝트 1~2개에 3층 구조 (#11)
- [ ] 외부 링크 상단 (#12)
- [ ] **초기 경력에 origin story 라벨** (#13)
- [ ] **핵심 프로젝트에 Problem Framing pattern 1개 이상** (#14) — 재정의/본질 추궁/가설-검증/모호함 명세화 중 최소 1개
- [ ] AI 활용 사례 있으면 Speed × Quality + Judgment Layer (#15~18)
- [ ] 한국 시장이면 직책 이중 표기 (#20)

### 출력 형식

**기본**: Markdown (`.md`)

**HTML 요청 시**: A4 인쇄 가능한 HTML
- 폰트: Pretendard 9pt
- 헤더: 좌측(이름·직책) + 우측(연락처) 두 열
- Section: Summary / Core Competencies / Experience / Side Project / Blog / Education
- 인쇄 마진: 15mm 18mm

**한국 시장 타겟**: 분량 2장 + 경력기술서 2장 가정
- 첫 2장: Summary + Competency + 최근 경력 2개
- 뒤 2장: 상세 경력 + Side Project + Education

### 산출물
- 이력서 파일 (`.md` 또는 `.html`)
- 사용자에게 저장 경로 확인

## Step 6. 검수

### 액션
생성된 이력서를 20원칙 체크리스트로 자체 검수.

### 검수 보고 형식

**가상 예시**:
```markdown
## 21원칙 체크리스트 — {이력서 파일명}

### Part 1. 기본 형식
✅ #1 평문 헤더 — 모든 Competency 통과
✅ #2 수치 절제 — 한 bullet당 최대 2개
⚠️ #3 코드 백틱 — 내부 클래스명 1회 잔존 (line 287). 평문화 권장.
✅ #4 Bullet ≤ 5
✅ #5 Ownership 동사 — "owned", "shipped" 사용 일관
⚠️ #6 Competency 4축 — Development 시그널 누락. "주니어 온보딩" 추가 권장.
✅ #7 포지션별 갈아끼움
✅ #8 솔직 라벨

### Part 2. 시니어 시그널
✅ #9 Scope 카드
⚠️ #10 Trade-off — 핵심 의사결정 1건에서 누락. "X 대신 Y 선택, 이유는 Z 포기 수용" 패턴 추가 권장.
✅ #11 Delivery → Outcome → Impact (대표 프로젝트 1회)
✅ #12 외부 링크
⚠️ #13 Thread Re-narration — 초기 경력 origin story 라벨 누락. 추가 권장.
⚠️ #14 Problem Framing — 핵심 프로젝트에 가설-검증 흔적 부재. 4 pattern 중 최소 1개 추가 권장 (재정의/본질 추궁/가설-검증/모호함 명세화).

### Part 3. AI 시대 (해당 시)
✅ #15~18 적용됨

### Part 4. 한국 시장 (한국 타겟)
✅ #19 PDF 2장+2장
✅ #20 직책 이중 표기
✅ #21 공백 라벨

## 총 17/21 통과. 수정 제안 4건.
```

### 수정 제안 후 액션
- 사용자에게 수정 제안 보이고 확인 요청
- 동의하면 `Edit`으로 즉시 수정
- 검수 다시 한 번

### 종료 조건 (무한 루프 방지)
- **검수는 최대 2회까지** — 1차 검수 → 수정 → 2차 검수까지만.
- 2차 검수 후 잔여 issue가 있으면 **사용자 검토용 보고**로 종료 (자동 수정 중단).
- 잔여 issue 보고에는 다음 포함:
  - 각 issue가 어느 원칙(#번호) 위반인지
  - 자동 수정이 어려운 이유 (예: "Scope 카드의 실제 시스템 규모 숫자가 자료에 없음")
  - 권장 수정 방향 (사용자가 직접 적용)
- **같은 issue가 2차 검수에서도 잔존하면 즉시 종료** — 동일 수정을 반복하지 않음.

### 산출물
- 최종 이력서 파일
- 검수 리포트
- (2차 검수 후 잔여 issue 있으면) 사용자 검토 가이드

## 흔히 발생하는 실수

### 1. 자료 부족인데 진행
**증상**: 단편이 안 나옴, 같은 키워드 반복
**해결**: Step 1에서 멈추고 사용자에게 추가 자료 요청

### 2. 포커스 두 개 동시에
**증상**: "Backend + Frontend 둘 다 강조"
**해결**: 별 이력서로 분리 제안. 한 이력서로 욱여넣지 않음.

### 3. Origin story 라벨 누락
**증상**: 초기 경력이 "그냥 한 일들" 나열
**해결**: 명시적 라벨 추가. "(현재 X의 기반이 된 첫 Y 경험)" 패턴.

### 4. 같은 프로젝트, 같은 디테일
**증상**: 두 포커스 이력서에서 같은 프로젝트가 동일한 단편으로 부각
**해결**: 포커스별로 부각할 디테일을 다르게. 원칙 #13의 표 참조.

### 5. 한국·글로벌 한 이력서로 동시 대응
**증상**: 분량·포맷이 어느 한쪽도 못 맞춤
**해결**: 별 이력서로 분리.
