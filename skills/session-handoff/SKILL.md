# Session Handoff

현재 세션에서 수행한 작업을 다른 세션으로 인계하기 위한 문서를 생성하는 스킬.

## 사용 시점

- 다른 Claude 세션에서 이어서 작업해야 할 때
- 다른 프로젝트 리포지토리에서 후속 작업이 필요할 때
- 팀원에게 작업 컨텍스트를 전달해야 할 때
- "핸드오프", "인계", "전달", "다른 세션", "다른 프로젝트에서 작업" 키워드 감지 시

## 사용하지 않을 때

- 같은 세션 내에서 계속 작업할 때
- 단순 커밋/PR 생성일 때
- 일일/주간 보고서 작성일 때 (별도 문서 규칙 적용)

## 실행 순서

### 1단계: 컨텍스트 수집

현재 세션에서 아래 정보를 수집한다:

- **완료된 작업**: 이번 세션에서 실제로 수행한 코드 변경, 설정 변경 등
- **변경된 파일**: `git diff`, `git log` 등으로 실제 변경 목록 확인
- **브랜치 정보**: 현재 브랜치, 커밋 히스토리
- **미완료 작업**: 아직 안 한 것, 후속으로 해야 할 것
- **의사결정 내역**: 왜 이렇게 했는지, 사용자가 거부/수정한 것
- **관련 인프라**: 환경변수, 외부 서비스, 설정값 등

### 2단계: 대상 확인

사용자에게 확인할 것:
- 인계 대상 프로젝트/리포지토리
- 인계받을 사람이 누구인지 (다른 Claude 세션 / 팀원)
- 문서 저장 위치 (기본: Obsidian Vault의 프로젝트 디렉토리)

**사용자가 이미 대상을 명시한 경우 이 단계를 건너뛴다.**

### 3단계: 문서 생성

대상이 1개 프로젝트면 단일 문서, 여러 프로젝트면 **마스터 핸드오프 1개**로 생성한다.

#### 단일 프로젝트 대상

저장 경로:
```
~/Workspace/weaversbrain/weaversbrain/Projects/{대상 프로젝트명}/{YYYY-MM}/{YYYY-MM-DD-HHMM}-{제목}-handoff.md
```

#### 멀티 프로젝트 대상 (마스터 핸드오프)

저장 경로 (소스 프로젝트 디렉토리에 저장):
```
~/Workspace/weaversbrain/weaversbrain/Projects/{소스 프로젝트명}/{YYYY-MM}/{YYYY-MM-DD-HHMM}-{제목}-master-handoff.md
```

프로젝트별 분리 문서를 만들지 않는다. 마스터 1개에 프로젝트별 섹션을 넣는다.

#### 문서 구조

```markdown
---
date: {YYYY-MM-DD}
type: handoff
project: {소스 프로젝트명}
target: {대상 프로젝트 목록 (콤마 구분)}
---

# {제목}

## 현재 상황
{왜 이 인계가 필요한지 1-2줄 요약}

## 완료된 작업 (소스 프로젝트)
{이번 세션에서 완료한 작업 목록. 관련 파일, 설정값 포함}

## 인계 대상 작업

### {프로젝트A} — {작업 요약}

> 프로젝트 경로: `~/Workspace/{프로젝트A}`

#### 작업 1: {작업명}
{구체적인 작업 설명}
- 위치: {파일 경로 또는 디렉토리}
- 예시 코드/설정 (있다면)

#### 작업 2: ...

#### 브랜치: {브랜치 지시}
#### 완료 조건: {테스트/스모크 기준}

### {프로젝트B} — {작업 요약}

> 프로젝트 경로: `~/Workspace/{프로젝트B}`

#### 브랜치: ...
#### 완료 조건: ...

(프로젝트 수만큼 반복)

## 세션 히스토리

이번 세션에서 일어난 일의 기록. 향후 "왜 이렇게 되었는지" 추적할 때 사용.

### 시도 및 의사결정
{세션 중 내린 주요 결정과 그 이유. 기각된 대안 포함}

### 사용자 수정사항

### 4단계: @dev 실행 프롬프트 생성

문서 하단에 `## @dev 실행 프롬프트` 섹션을 추가한다.

**핵심: 프롬프트는 1개만 생성한다.** 모든 대상 프로젝트에서 동일한 프롬프트를 붙여넣으면 된다. `@dev`는 각 프로젝트의 `.claude/agents/dev.md`를 읽는 공통 호출명이므로 프로젝트별 에이전트 이름 차이를 신경 쓸 필요 없다.

`@dev`가 핸드오프 문서를 읽고, 현재 프로젝트(`pwd`)에 해당하는 섹션을 자동 식별하여 실행한다.

```markdown
## @dev 실행 프롬프트

> 대상 프로젝트에서 새 Claude 세션을 열고 아래를 붙여넣기 (모든 프로젝트 동일):

\```
@dev 핸드오프 문서 기반 작업 실행

핸드오프: {마스터 핸드오프 문서 절대 경로}
현재 프로젝트에 해당하는 섹션만 실행할 것.
\```
```

프롬프트 작성 규칙:
1. **핸드오프 문서 경로**만 명시 — 작업 상세는 문서의 프로젝트 섹션에 있으므로 프롬프트에 중복 서술 금지
2. **프로젝트 자동 식별** 지시 — `@dev`가 `pwd` 또는 프로젝트 경로로 해당 섹션 매칭
3. 프롬프트는 **1개만** — 프로젝트별 프롬프트를 따로 만들지 않는다

**문서 생성 후 대화에도 출력**: 문서에 프롬프트를 기록한 뒤, 대상 프로젝트 목록과 함께 복사 가능한 프롬프트를 대화에 직접 출력한다. 사용자가 문서를 열지 않고 바로 붙여넣을 수 있도록.

출력 형식:
```
핸드오프 생성 완료: {문서 경로}

대상 프로젝트에서 새 세션 열고 붙여넣기:

{코드블록으로 @dev 프롬프트}

대상: {프로젝트1} (`~/Workspace/{프로젝트1}`), {프로젝트2} (`~/Workspace/{프로젝트2}`)
```

### 5단계: 핸드오프 수신 및 완료 처리

핸드오프를 받는 쪽(인계받은 세션)의 시작과 끝 절차.

#### 수신 (세션 시작 시)

인계받은 세션이 핸드오프 문서를 읽고 작업을 시작할 때:

1. 핸드오프 문서의 frontmatter에 `status: in_progress`, `started_date`, `started_by` 추가
2. 자기 프로젝트 섹션의 작업 항목을 확인하고 순서대로 진행

```yaml
status: in_progress
started_date: 2026-04-16
started_by: {프로젝트명} 세션
```

#### 완료 (작업 끝난 후)

인계받은 세션이 자기 프로젝트 섹션의 작업을 모두 완료했을 때:

1. 해당 프로젝트 섹션 제목에 ✅ 표시 (예: `### maxai-b2c-backend — ... ✅ 완료`)
2. 완료 조건 충족 여부를 섹션 하단에 기록
3. **모든 프로젝트 섹션이 완료되면** frontmatter를 갱신:

```yaml
status: done
completed_date: 2026-04-16
```

4. 마스터 핸드오프의 경우, 마지막으로 완료한 세션이 `status: done`을 기록

**완료 시 대화에도 출력**: 문서 업데이트 후, 완료 상태를 대화에 직접 출력한다.

출력 형식:
```
핸드오프 완료: {문서 경로}
- {프로젝트명} 섹션 ✅ (커밋: {해시}, 검증: {결과})
- 남은 섹션: {미완료 프로젝트 목록 또는 "없음 — 전체 완료"}
```

#### 부분 완료 / 추가 핸드오프

작업 중 새로운 이슈가 발생하거나 일부만 완료된 경우:
- 해당 섹션에 `⚠️ 부분 완료` 표시 + 미완료 사유 기록
- 필요 시 새 핸드오프 문서를 생성하고 `superseded_by` 필드로 연결

```yaml
status: superseded
superseded_by: Projects/speakingmax-backend/2026-04/2026-04-16-1123-후속-handoff.md
```

### 7단계: 소스 프로젝트 문서 업데이트

코드 변경이 있었으면 소스 프로젝트의 docs 에이전트(`identity-hub-docs` 등)를 백그라운드로 실행하여 `docs/` 디렉토리를 동기화한다.

실행 조건:
- 이번 세션에서 코드 파일 변경이 있었을 때 (설정/문서만 수정한 경우 생략)
- 소스 프로젝트에 docs 에이전트가 설정되어 있을 때 (`.claude/agents/docs.md` 존재 여부 확인)

실행 방법:
```
Agent(subagent_type="identity-hub-docs", run_in_background=true, prompt="이번 세션 변경사항 기반 docs/ 업데이트: {변경 요약}")
```

프로젝트별 docs 에이전트가 없으면 이 단계를 건너뛴다.

### 8단계: 핵심 규칙

1. **구체적으로 쓴다**: "S3 설정 필요" ❌ → "`.env`에 `S3_AUDIO_ENABLED=true` 추가, AWS 키 발급 필요" ✅
2. **코드/설정 예시 포함**: 대상 세션이 바로 작업 가능하도록 docker-compose, SQL, 설정 파일 예시를 넣는다
3. **의사결정 이유 포함**: 왜 이 구조/이름을 선택했는지 (예: "maxai가 아닌 speakingmax로 통일")
4. **환경별 차이 명시**: local/dev/prod 간 다른 설정이 있으면 표로 정리
5. **대상 프로젝트의 기존 구조 확인**: 가능하면 대상 프로젝트의 디렉토리/파일 구조를 읽어서 기존 패턴에 맞춰 작성
6. **세션 히스토리는 판단 근거 위주**: 디버깅 삽질 과정은 생략. 의사결정, 리뷰 피드백, 사용자 수정사항만 기록. "왜 이렇게 되었나" 추적 가능하면 충분.
7. **멀티 프로젝트는 마스터 1개**: 프로젝트별 분리 문서 생성 금지. 마스터 문서 내 프로젝트 섹션으로 구분.
8. **프롬프트는 1개**: 프로젝트별 프롬프트 생성 금지. `@dev`가 현재 프로젝트 기준으로 자동 식별.
